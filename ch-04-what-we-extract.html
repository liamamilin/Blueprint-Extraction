6  What We Extract: Blueprint Kernel and Views

Blueprint extraction does not attempt to reconstruct every detail of a system implementation. Doing so would be computationally expensive, architecturally noisy, and analytically unnecessary. Extraction operates at a higher level of abstraction.

Instead, the objective is to recover the architectural structure of the system in a stable, canonical, and computable form. This structure captures what the system is—its essential organization and design—rather than how it is implemented.

The extracted structure is represented through two layers:

  1. The Blueprint Kernel: The minimal architectural representation of a system
  2. Blueprint Views: Different projections of the kernel for different analytical purposes

The Kernel defines the canonical, complete architectural model. Views provide specialized perspectives on that model. This separation ensures a single source of architectural truth while enabling diverse analytical applications.

Extraction therefore targets the Kernel as the primary artifact. Views are not extracted independently; they are generated from the kernel through deterministic projection rules.

6.1 The Blueprint Kernel as the Extraction Target

The Blueprint Kernel defines the minimal set of structural elements required to represent a system architecture. It is designed to be:

  • Minimal: Contains only essential architectural information
  • Canonical: Represents the authoritative architectural model
  • Stable: Changes only when actual architecture changes
  • Language-independent: Abstracts away implementation details
  • Computable: Can be processed by automated tools and algorithms

The Kernel consists of six mandatory slots:

Goal
Capabilities
Components
Interfaces
Data
Workflows

Each slot represents a distinct, non-overlapping dimension of system structure. Together, they form a structural skeleton that captures the essential architecture of any system.

The six kernel slots:

Goal describes the problem space the system addresses and the value it provides. It answers: What is this system for?

Capabilities define the functional abilities the system provides to users or other systems. They answer: What can this system do?

Components represent the structural units responsible for implementing capabilities. They answer: What are the building blocks of this system?

Interfaces describe how components or external actors interact with the system. They answer: How is this system accessed?

Data represents the persistent information owned or managed by the system. It answers: What information does this system maintain?

Workflows describe the control flow through which components coordinate behavior. They answer: How does execution flow through this system?

These six slots provide comprehensive coverage of system architecture while remaining abstract enough to apply across diverse systems—from simple CLI tools to complex distributed systems.

The kernel as a structural skeleton:

The six slots form a structural skeleton that supports multiple architectural analyses:

  • Component analysis: Examining components and their relationships
  • Interface analysis: Understanding interaction surfaces and contracts
  • Data analysis: Mapping data ownership and flow
  • Workflow analysis: Understanding execution coordination
  • Capability analysis: Mapping functional abilities to structure
  • Evolution analysis: Tracking how architecture changes over time

The kernel provides a stable representation that is independent of specific programming languages, frameworks, or deployment environments. This stability is essential for building blueprint datasets that remain valid as underlying technologies evolve.

6.2 Kernel Slots in Detail

Each kernel slot has a specific definition, extraction target, and evidence requirements. Understanding these details is essential for designing extraction systems and interpreting blueprints.

6.2.1 Goal

Definition: The high-level problem the system addresses and the value proposition it delivers.

Extraction targets: Repository artifacts that describe system purpose, including: - Repository README files and descriptions - Project documentation and design documents - Issue tracker descriptions and mission statements - Marketing materials or product descriptions - Source code comments expressing architectural intent

Evidence characteristics: Goal is often the most abstract kernel slot. Evidence may be: - Explicit: Direct statements of purpose in documentation - Inferred: Derived from aggregated capabilities and interfaces - Partial: Goal may be reconstructed indirectly from behavior

Confidence implications: Goal often requires interpretation and synthesis, so confidence is typically moderate to low. Unknown goals are common when documentation is absent.

6.2.2 Capabilities

Definition: The functional abilities the system provides to users or other systems.

Extraction targets: Artifacts that reveal what the system can do: - API endpoints and route definitions - CLI commands and subcommands - Event handlers and message consumers - Workflow steps and pipeline tasks - Test cases describing expected behavior

Evidence characteristics: Capabilities are often strongly evidenced by: - Explicit route or command definitions - Interface specifications (OpenAPI, protobuf, etc.) - Workflow definitions describing operations - Documentation of system functions

Confidence implications: Capabilities directly visible through interfaces often have high confidence. Capabilities inferred from indirect behavior have moderate confidence.

6.2.3 Components

Definition: The structural units of responsibility that implement capabilities.

Extraction targets: Artifacts that reveal structural decomposition: - Service boundaries and module organization - Worker processes and background tasks - Scheduler or orchestrator components - Plugin or extension systems - Microservice or container boundaries

Evidence characteristics: Component evidence varies in strength: - Strong: Explicit service boundaries, deployment manifests - Moderate: Directory structure suggesting decomposition - Weak: Naming conventions or module organization

Confidence implications: Explicit component boundaries (services, containers) have high confidence. Inferred components from directory structure have moderate confidence.

6.2.4 Interfaces

Definition: The interaction surfaces through which capabilities are accessed.

Extraction targets: Artifacts defining interaction mechanisms: - HTTP route definitions and API specifications - Event topics and message schemas - CLI command definitions and argument parsers - RPC service definitions and protocol buffers - WebSocket or streaming endpoints

Evidence characteristics: Interfaces typically have strong evidence: - Explicit: Route definitions, API specifications - Documented: Interface documentation or contract files - Tested: Integration tests verifying interface behavior

Confidence implications: Interfaces with explicit definitions and documentation have high confidence. Inferred interfaces from code patterns have moderate confidence.

6.2.5 Data

Definition: The persistent information owned, maintained, or transformed by the system.

Extraction targets: Artifacts revealing data structures: - Database migration files and schema definitions - ORM model definitions and data classes - Document collection schemas and validation rules - Cache structures and key-value stores - Data lake or object storage schemas

Evidence characteristics: Data evidence is typically strong: - Explicit: Migration files, schema definitions - Structural: ORM models reflecting database structure - Operational: Storage configuration and deployment specs

Confidence implications: Explicit schema definitions have high confidence. Inferred data entities from access patterns have moderate confidence.

6.2.6 Workflows

Definition: The execution sequences through which components coordinate to realize capabilities.

Extraction targets: Artifacts describing execution coordination: - DAG definitions and workflow specifications - Pipeline configurations and orchestration scripts - Scheduler configurations and cron jobs - Event flow definitions and processing pipelines - State machine definitions

Evidence characteristics: Workflow evidence varies: - Strong: Explicit DAG specifications, workflow engines - Moderate: Scheduler configurations, cron jobs - Weak: Event patterns inferred from code structure

Confidence implications: Explicit workflow definitions have high confidence. Inferred workflows from code organization have moderate confidence.

6.3 Kernel Slots as Extraction Objectives

During extraction, each kernel slot becomes an explicit target of inference. The extraction pipeline systematically attempts to populate each slot with evidence-backed elements.

The extraction process for each slot:

  1. Signal detection: Identify repository artifacts that may correspond to the slot
  2. Hypothesis formation: Formulate provisional interpretations in terms of the slot
  3. Evidence collection: Gather concrete artifacts supporting or refuting each hypothesis
  4. Ontology mapping: Map validated evidence to canonical entities for the slot
  5. Slot population: Assemble mapped entities into the kernel slot

Slot-specific extraction patterns:

Kernel Slot Primary Signal Sources Typical Evidence Common Ambiguities
Goal Documentation, descriptions, README Explicit statements, indirect inference Abstract, multi-faceted, often undocumented
Capabilities Routes, commands, workflows, tests Explicit definitions, test cases Implicit capabilities, undocumented features
Components Services, workers, plugins, modules Deployment manifests, directory structure Logical vs. deployable boundaries
Interfaces Routes, topics, commands, RPC Route definitions, API specs Internal vs. external interfaces
Data Migrations, schemas, ORM models Schema files, migration history Transient vs. persistent data
Workflows DAGs, pipelines, schedulers Workflow specs, cron configs Ad-hoc vs. formalized workflows

Handling incomplete extraction:

Not all slots can always be fully reconstructed from repository artifacts. The extraction process allows:

  • Partial completion: Some elements of a slot are populated, others remain unknown
  • Explicit unknowns: Slots or fields marked as “unknown” when evidence is insufficient
  • Confidence annotations: Low-confidence elements marked appropriately for review

This honest representation of uncertainty prevents blueprints from accumulating unsupported assumptions.

6.4 The Evidence Traceability Principle

A central, non-negotiable rule governs kernel extraction:

Every extracted field must be traceable to repository evidence.

This principle is the foundation of reliable extraction. It prevents architectural claims from becoming speculative interpretations and ensures that blueprints remain auditable, reproducible, and debuggable.

What the principle requires:

Each element that fills a kernel slot must reference one or more evidence objects that originate from concrete repository artifacts:

  • Source files: Code implementing component behavior or interface handlers
  • Configuration files: Files defining services, routing, or dependencies
  • Dependency manifests: Files declaring required libraries or frameworks
  • Schema definitions: Files defining data structures and constraints
  • Workflow specifications: Files describing execution coordination
  • Documentation fragments: Text describing system intent or design

Evidence object structure:

Each evidence object captures: - Source: The repository artifact (file path, URL, or location) - Location: The exact position within the artifact - Excerpt: The relevant content fragment - Type: The artifact classification (code, config, schema, deploy, documentation) - Strength: How directly the artifact supports the interpretation

The traceability payoff:

Evidence traceability ensures that blueprint elements remain:

  • Auditable: Reviewers can verify claims by inspecting source artifacts
  • Reproducible: The same extraction process yields consistent results
  • Debuggable: Errors can be traced to specific evidence or mapping decisions
  • Evolvable: Blueprints can be updated as repositories change
  • Verifiable: Independent systems can validate extraction results

Handling insufficient evidence:

When insufficient evidence exists to populate a kernel slot, the extraction process must:

  1. Mark the field as unknown rather than filling it with speculation
  2. Document why evidence is insufficient in the extraction report
  3. Flag the field for potential human review if it corresponds to a critical architectural element

This rule maintains the integrity of blueprint datasets and enables reliable architectural analysis. Blueprints with honest unknown representations are more valuable than blueprints filled with confident but incorrect assumptions.

6.5 Blueprint Views as Projections

While the Kernel represents the canonical blueprint structure, different stakeholders require different perspectives on the system. A single architectural model cannot simultaneously optimize for all analytical and operational needs.

Blueprint views provide these specialized perspectives. A view is a projection of the kernel structure that emphasizes specific architectural aspects while omitting or de-emphasizing others.

Key characteristics of views:

  • Derived: Views are generated from the kernel, not extracted independently
  • Deterministic: The same kernel always produces the same view
  • Perspective-specific: Each view serves a particular analytical or operational purpose
  • Lossy: Views may omit or summarize kernel information for clarity
  • Reproducible: Views can be regenerated deterministically from the kernel

The extraction process and views:

The extraction pipeline does not attempt to extract views independently. Instead:

  1. The pipeline extracts and populates the kernel slots
  2. Once the kernel is validated, view projections are generated through defined rules
  3. Views are stored alongside the kernel in the blueprint pack

This kernel-first strategy ensures that: - All views are consistent with a single architectural model - Changes to the kernel propagate correctly to all views - New views can be added without re-extraction - Views can be generated on-demand as needed

Common blueprint views:

The Spec View focuses on system behavior and externally visible contracts. It emphasizes: - System goal and purpose - Exposed capabilities and their descriptions - Interface contracts and specifications - Functional requirements and constraints

The Architecture View emphasizes structural relationships between architectural elements. It highlights: - Component composition and boundaries - Interface connections and dependencies - Data ownership and flow - Workflow coordination and control

The Ops View focuses on deployment topology and operational execution. It captures: - Deployment units and packaging - Runtime dependencies and services - Operational workflows (build, deploy, monitor) - Observability and monitoring instrumentation

The Security/Trust View focuses on security boundaries and trust relationships. It reveals: - Trust boundaries between components - Authentication and authorization mechanisms - Sensitive data access paths - Security-relevant interfaces and policies

Each view reorganizes the same underlying kernel information to serve different analytical contexts—design review, operational planning, security assessment, or architectural comparison.

6.6 Kernel-First Extraction

Because views are projections, the extraction pipeline adopts a kernel-first strategy. This strategy has important implications for the extraction process and system design.

The kernel-first extraction sequence:

  1. Reconstruct the kernel slots:
    • Goal
    • Capabilities
    • Components
    • Interfaces
    • Data
    • Workflows
  2. Validate the kernel:
    • Check structural completeness
    • Verify reference integrity
    • Assess confidence levels
    • Detect anomalies
  3. Generate view projections:
    • Apply projection rules to create each view
    • Validate view-specific constraints
    • Package views with the kernel
  4. Package the blueprint pack:
    • Assemble kernel, views, evidence, and reports
    • Generate diff-friendly serialization
    • Prepare for review and distribution

Benefits of kernel-first extraction:

Structural consistency: Since all views derive from a single kernel, they remain mutually consistent. Changes to the kernel propagate correctly to all views.

Evidence efficiency: Evidence is collected once for the kernel and reused across multiple views, avoiding redundant extraction effort.

Adaptability: New analytical views can be added by defining new projection rules without re-extracting repositories.

Maintainability: When extraction methodology improves, only kernel extraction needs updating—view projections regenerate automatically.

Traceability: Evidence traceability is maintained through the kernel; views inherit this traceability through their derivation from kernel elements.

The kernel as canonical truth:

The Kernel functions as the canonical representation of system architecture. Views act as interpretive lenses derived from that structure. This separation of concerns ensures:

  • A single source of architectural truth
  • Multiple analytical perspectives without inconsistency
  • Efficient update and evolution of the architectural model
  • Clear boundaries between extraction and analysis

The kernel-first strategy is therefore not merely a convenience but a fundamental architectural principle of blueprint extraction.

6.7 Chapter Summary

  • Blueprint extraction targets architectural structure, not implementation details
  • The extracted structure has two layers: the Blueprint Kernel (canonical) and Blueprint Views (projections)
  • The Kernel consists of six slots: Goal, Capabilities, Components, Interfaces, Data, and Workflows
  • Each slot represents a distinct dimension of system architecture with specific extraction targets
  • The evidence traceability principle requires every extracted field to reference concrete repository artifacts
  • Insufficient evidence results in explicit unknowns rather than speculative assumptions
  • Views are projections generated from the kernel, not extracted independently
  • Kernel-first extraction ensures structural consistency, evidence efficiency, and adaptability
  • The Kernel is the canonical source of architectural truth; views provide specialized analytical perspectives
