11  Confidence Semantics

Blueprint extraction reconstructs architectural structure from indirect repository signals. Because repositories rarely encode architecture explicitly, extracted elements inevitably contain varying degrees of certainty.

Confidence semantics provide a formal mechanism for representing this uncertainty. Rather than presenting extracted architecture as definitive truth, the extraction process attaches confidence annotations to every inferred element. These annotations communicate how strongly the available evidence supports the architectural interpretation.

Confidence semantics therefore serve two critical roles:

  • Representing uncertainty: Making uncertainty explicit rather than hidden
  • Guiding review: Directing human attention to uncertain, high-impact elements

Without confidence semantics, extraction would produce overconfident, unreliable architectural reconstructions. With confidence semantics, extraction produces honest, auditable representations that acknowledge the limits of available information.

11.1 The Problem of Uncertainty

Architectural reconstruction is inherently uncertain. Repositories provide incomplete, ambiguous, and sometimes contradictory signals about system structure. Extracting architecture from these signals inevitably involves interpretation, judgment, and inference—all of which introduce uncertainty.

Sources of uncertainty:

1. Missing information

Some architectural elements exist only outside repositories:

  • Operational constraints: Runtime policies, scaling requirements
  • Runtime environments: Container orchestration, service discovery
  • Organizational ownership: Team responsibilities, governance structures
  • External integrations: Third-party services, API dependencies

When information is missing, extraction must either: - Mark elements as unknown (honest representation) - Infer elements from partial signals (risky interpretation)

2. Implicit conventions

Architectural intent may be encoded indirectly:

  • Naming patterns: Directory names suggesting component boundaries
  • Framework conventions: Framework-specific structural patterns
  • Organizational standards: Team-specific conventions

Implicit conventions are fragile and may: - Vary across teams and organizations - Be applied inconsistently - Evolve over time without documentation

3. Ambiguous evidence

A single artifact may admit multiple interpretations:

  • Directory named services/ could mean microservices or logical modules
  • Queue dependency could indicate async processing or message transport
  • Scheduler configuration could indicate workflows or maintenance tasks

Ambiguity forces extraction to choose among plausible interpretations, introducing uncertainty.

4. Conflicting evidence

Different artifacts may suggest inconsistent architectures:

  • Outdated documentation: Describes components that no longer exist
  • Legacy configuration: References deprecated services
  • Evolutionary drift: Architecture evolved but artifacts not updated

Conflicting evidence forces extraction to choose which sources to trust.

The cost of ignoring uncertainty:

If extraction ignores uncertainty and presents all interpretations as equally certain:

  • Overconfident decisions: Users may trust incorrect architectures
  • Misguided actions: Architecture decisions based on incorrect information
  • Wasted resources: Time spent investigating non-existent structures
  • Erosion of trust: Users lose confidence in extraction accuracy

The value of representing uncertainty:

By making uncertainty explicit, extraction becomes:

  • Honest: Acknowledges what is known versus unknown
  • Useful: Guides users to focus attention appropriately
  • Improvable: Identifies areas where methodology needs improvement
  • Trustworthy: Builds confidence through transparency

Confidence semantics therefore transform uncertainty from a weakness into a feature—honest representation of knowledge limits.

11.2 Confidence Object Structure

Confidence annotations are attached to architectural entities and fields. The minimal confidence representation ensures that uncertainty is captured consistently across the extraction pipeline.

Minimal confidence representation:

confidence:
  state: <observed | inferred | unknown>
  score: <optional-numeric-value>  # 0.0 to 1.0
  evidence_refs:
    - <evidence-id>
    - <evidence-id>
  confidence_details:
    evidence_strength: <strong | moderate | weak>
    evidence_count: <number>
    evidence_consistency: <high | medium | low>
    pattern_alignment: <strong | moderate | weak | none>
  assigned_by:
    method: <manual | calculated>
    version: <confidence-model-version>
    timestamp: <iso8601>

Field explanations:

11.2.1 state

The qualitative confidence state. Three canonical states are defined:

  • observed: Directly supported by explicit repository artifacts
  • inferred: Derived from indirect signals and interpretation
  • unknown: Insufficient evidence exists to determine presence or structure

11.2.2 score

An optional numeric confidence value between 0.0 (no confidence) and 1.0 (complete confidence). Numeric scores enable:

  • Granular comparison: Comparing confidence across elements
  • Thresholding: Filtering elements by confidence level
  • Aggregation: Calculating aggregate confidence for groups
  • Machine learning: Feeding confidence into ML models

11.2.3 evidence_refs

List of evidence object IDs supporting this confidence assignment. This linkage enables:

  • Traceability: Connecting confidence to supporting evidence
  • Validation: Verifying confidence is justified
  • Debugging: Identifying which evidence contributed to confidence
  • Review: Focusing review on relevant evidence

11.2.4 confidence_details

Detailed breakdown of factors contributing to confidence:

  • evidence_strength: Strongest evidence type (strong/moderate/weak)
  • evidence_count: Number of supporting evidence objects
  • evidence_consistency: How well evidence aligns (high/medium/low)
  • pattern_alignment: Alignment with known patterns (strong/moderate/weak/none)

These details enable confidence calculation and explanation.

11.2.5 assigned_by

Metadata about how confidence was assigned:

  • method: Manual assignment by reviewer or calculated by pipeline
  • version: Confidence model version for reproducibility
  • timestamp: When confidence was assigned

11.3 Confidence States

Confidence Semantics v0.1 defines three canonical confidence states. These states represent increasing levels of uncertainty and guide review prioritization.

11.3.1 Observed

Definition: An element is classified as observed when its existence is directly supported by explicit repository artifacts.

Characteristics:

  • Explicit encoding: Architecture is directly declared in artifacts
  • Strong evidence: Evidence objects with strong strength ratings
  • Unambiguous interpretation: Only one plausible interpretation exists
  • Verifiable: Can be validated by inspecting source artifacts

Examples:

Observed component:

component:
  id: component.api_service
  type: Service
  confidence:
    state: observed
    score: 0.95
    evidence_refs:
      - evidence-001  # Server initialization code
      - evidence-002  # Deployment configuration
      - evidence-003  # API route definitions
    confidence_details:
      evidence_strength: strong
      evidence_count: 3
      evidence_consistency: high
      pattern_alignment: strong

Observed interface:

interface:
  id: interface.http_api
  type: API
  confidence:
    state: observed
    score: 0.92
    evidence_refs:
      - evidence-004  # Route definitions
      - evidence-005  # OpenAPI specification
      - evidence-006  # Controller implementations

Observed data entity:

data:
  id: data.payments
  confidence:
    state: observed
    score: 0.98
    evidence_refs:
      - evidence-007  # Schema migration
      - evidence-008  # ORM model
      - evidence-009  # Database DDL

Confidence implications:

Observed elements generally have: - High confidence scores (0.8-1.0) - Minimal review requirements (spot checks sufficient) - High reliability for architectural analysis

Observed elements represent the foundation of reliable extraction—the structures we can trust with high certainty.

11.3.2 Inferred

Definition: An element is classified as inferred when it is derived from indirect signals rather than explicit artifacts.

Characteristics:

  • Interpretive: Architecture is suggested rather than declared
  • Moderate evidence: Evidence objects with moderate strength ratings
  • Plausible but uncertain: Multiple interpretations may be possible
  • Validation required: Should be reviewed to confirm interpretation

Examples:

Inferred component:

component:
  id: component.payment_worker
  type: Worker
  confidence:
    state: inferred
    score: 0.65
    evidence_refs:
      - evidence-010  # Queue dependency
      - evidence-011  # Worker directory
      - evidence-012  # Scheduler configuration
    confidence_details:
      evidence_strength: moderate
      evidence_count: 3
      evidence_consistency: medium
      pattern_alignment: moderate

Inferred capability:

capability:
  id: capability.batch_processing
  confidence:
    state: inferred
    score: 0.58
    evidence_refs:
      - evidence-013  # Scheduler configuration
      - evidence-014  # Workflow definitions
    confidence_details:
      evidence_strength: moderate
      evidence_count: 2
      evidence_consistency: medium
      pattern_alignment: weak

Inferred workflow:

workflow:
  id: workflow.payment_processing
  confidence:
    state: inferred
    score: 0.72
    evidence_refs:
      - evidence-015  # DAG definition
      - evidence-016  # Task implementations
    confidence_details:
      evidence_strength: moderate
      evidence_count: 2
      evidence_consistency: high
      pattern_alignment: moderate

Confidence implications:

Inferred elements generally have: - Medium confidence scores (0.4-0.8) - Moderate review requirements (validate interpretation) - Questionable reliability without human confirmation

Inferred elements require human review to confirm that the interpretation is correct and to adjust confidence based on additional context.

11.3.3 Unknown

Definition: An element is classified as unknown when insufficient evidence exists to determine its presence or structure.

Characteristics:

  • Evidence absent: No or insufficient evidence available
  • Cannot be determined: Interpretation would be pure speculation
  • Honest representation: Explicitly acknowledges knowledge gap
  • Investigation opportunity: Indicates where additional information would be valuable

Examples:

Unknown constraint:

constraint:
  id: constraint.performance_slo
  confidence:
    state: unknown
    score: 0.0
    evidence_refs: []  # No supporting evidence
    confidence_details:
      evidence_strength: none
      evidence_count: 0
      evidence_consistency: none
      pattern_alignment: none
  notes: |
    No performance requirements found in repository.
    May exist in external documentation or organizational policies.

Unknown goal:

goal:
  id: goal.system_purpose
  confidence:
    state: unknown
    score: 0.15
    evidence_refs:
      - evidence-017  # Vague README mention
    confidence_details:
      evidence_strength: weak
      evidence_count: 1
      evidence_consistency: low
      pattern_alignment: none
  notes: |
    README mentions "payment processing" but lacks
    comprehensive system description. Goal is partially
    inferred from capabilities.

Unknown data ownership:

data:
  id: data.transaction_log
  confidence:
    state: unknown
    score: 0.0
    evidence_refs: []
    confidence_details:
      evidence_strength: none
      evidence_count: 0
      evidence_consistency: none
      pattern_alignment: none
  notes: |
    Transaction log table exists but ownership
    cannot be determined from available artifacts.
    Multiple services may write to this table.

Confidence implications:

Unknown elements generally have: - Low confidence scores (0.0-0.4) - High review requirements (investigation needed) - No reliability for architectural analysis without additional information

Unknown elements represent knowledge gaps that should be investigated rather than speculated. They highlight areas where additional documentation, architecture review, or instrumentation would improve extraction accuracy.

11.4 Confidence Derivation

Confidence classifications are not arbitrary. They are determined by analyzing the evidence associated with an architectural element. Several factors influence confidence.

11.4.1 Evidence Strength

Definition: How directly artifacts express architectural structure.

Strength levels:

  • Strong: Explicit encoding (schemas, routes, workflows, interfaces)
  • Moderate: Strong suggestions (dependencies, configuration, entrypoints)
  • Weak: Indirect hints (naming, structure, documentation)

Impact on confidence:

  • Strong evidence: Base confidence 0.7-0.9
  • Moderate evidence: Base confidence 0.4-0.6
  • Weak evidence: Base confidence 0.1-0.3

Example:

# Strong evidence (schema definition)
evidence:
  strength: strong
  excerpt: "CREATE TABLE payments (...)"
# → Base confidence: 0.85

# Moderate evidence (dependency)
evidence:
  strength: moderate
  excerpt: "celery==5.3.0"
# → Base confidence: 0.55

# Weak evidence (directory name)
evidence:
  strength: weak
  excerpt: "workers/"
# → Base confidence: 0.25

11.4.2 Evidence Quantity

Definition: Number of independent evidence objects supporting the same interpretation.

Impact on confidence:

More evidence increases confidence, but with diminishing returns:

confidence_boost = log(count(evidence))

Examples:

# Single evidence object
evidence_count: 1
confidence_boost: log(1) = 0.0

# Three evidence objects
evidence_count: 3
confidence_boost: log(3) = 0.48

# Ten evidence objects
evidence_count: 10
confidence_boost: log(10) = 1.0

11.4.3 Evidence Consistency

Definition: Whether different evidence sources agree with each other.

Consistency levels:

  • High: All evidence points to the same interpretation
  • Medium: Most evidence agrees, minor inconsistencies
  • Low: Significant disagreements or contradictions

Impact on confidence:

  • High consistency: No penalty
  • Medium consistency: Small penalty (-0.1)
  • Low consistency: Large penalty (-0.3)

Example:

# High consistency
evidence:
  - schema: "CREATE TABLE payments (...)"
  - orm: "class Payment(db.Model): ..."
  - test: "def test_payment(): ..."
consistency: high
penalty: 0.0

# Low consistency
evidence:
  - doc: "Payment service is standalone microservice"
  - code: "Payments processed inline with API"
  - config: "No payment service configuration"
consistency: low
penalty: -0.3

11.4.4 Pattern Alignment

Definition: Whether extracted structures match well-known architectural patterns.

Alignment levels:

  • Strong: Exact match to well-known pattern
  • Moderate: Similar to known pattern with variations
  • Weak: Some similarity to patterns
  • None: No recognizable pattern

Impact on confidence:

  • Strong alignment: Small boost (+0.1)
  • Moderate alignment: Tiny boost (+0.05)
  • Weak alignment: No boost
  • No alignment: No boost

Example:

# Strong pattern alignment (typical 3-tier web app)
structure:
  - API layer (HTTP routes)
  - Service layer (business logic)
  - Data layer (database)
alignment: strong
boost: +0.1

# No pattern alignment (unusual architecture)
structure:
  - Mixed responsibilities
  - Unclear boundaries
alignment: none
boost: +0.0

11.4.5 Confidence Calculation Formula

Combining all factors:

base_confidence = strength_to_confidence(strength)
evidence_boost = log(count(evidence))
consistency_penalty = consistency_to_penalty(consistency)
pattern_boost = alignment_to_boost(alignment)

final_confidence = base_confidence +
                   evidence_boost +
                   consistency_penalty +
                   pattern_boost

# Clamp to [0.0, 1.0]
final_confidence = max(0.0, min(1.0, final_confidence))

Confidence state mapping:

final_confidence >= 0.8  → state: observed
final_confidence >= 0.4  → state: inferred
final_confidence < 0.4   → state: unknown

Example calculation:

# Worker component inference
evidence:
  strength: moderate  # Base: 0.55
  count: 3           # Boost: log(3) = 0.48
  consistency: high   # Penalty: 0.0
  alignment: moderate # Boost: 0.05

final_confidence = 0.55 + 0.48 + 0.0 + 0.05 = 1.08
# Clamped: 1.0
# State: observed

11.5 Confidence Propagation

Confidence may propagate through relationships within the architecture graph. When one element has a certain confidence, related elements may inherit or derive confidence from that relationship.

Propagation scenarios:

1. Interface to capability propagation:

interface:
  id: interface.payment_api
  confidence:
    state: observed
    score: 0.92

capability:
  id: capability.create_payment
  confidence:
    state: inferred
    score: 0.75  # Inherited partially from interface

Rationale: If an interface is strongly observed, the capabilities it exposes can be inferred with moderate-high confidence.

2. Component to interface propagation:

component:
  id: component.api_service
  confidence:
    state: observed
    score: 0.95

interface:
  id: interface.http_api
  confidence:
    state: observed
    score: 0.90  # Derived from component

Rationale: If a component is strongly observed, its interfaces can be inferred with high confidence.

3. Workflow to component propagation:

workflow:
  id: workflow.payment_processing
  confidence:
    state: inferred
    score: 0.65

component:
  id: component.payment_worker
  confidence:
    state: inferred
    score: 0.58  # Derived from workflow

Rationale: If a workflow is inferred, the components implementing it can be inferred with similar confidence.

Propagation rules:

  1. Directional: Confidence propagates from stronger to weaker elements
  2. Attenuation: Propagated confidence is attenuated (reduced by 0.1-0.2)
  3. Non-destructive: Propagation never reduces confidence below original evidence-based level
  4. Explicit: Propagation should be documented in confidence metadata

Propagation formula:

propagated_confidence = source_confidence - propagation_attenuation
final_confidence = max(evidence_confidence, propagated_confidence)

Where propagation_attenuation is typically 0.1-0.2.

Propagation limitations:

  • No upward propagation: Weak elements cannot strengthen strong elements
  • Evidence priority: Explicit evidence always takes precedence over propagation
  • Documented assumptions: Propagation assumptions must be documented

11.6 Confidence and Review Prioritization

Confidence semantics guide the human review process by identifying where human attention should be focused. In large-scale extraction pipelines, manual review resources are limited, so prioritization is essential.

Review priority matrix:

Confidence Impact Review Priority Review Type
High Critical High Validate key assumptions
High Non-critical Low Spot check
Medium Critical Very High Careful validation
Medium Non-critical Medium Validate interpretation
Low Critical Very High Investigate, possibly add evidence
Low Non-critical High Review if time permits
Unknown Critical Mandatory Must resolve before use
Unknown Non-critical Medium Attempt to resolve

Impact categories:

Critical impact:

  • Core components (API services, data stores)
  • Key interfaces (public APIs, event schemas)
  • Essential workflows (payment processing, user authentication)
  • Critical data (PII, financial data)

Non-critical impact:

  • Peripheral components (utility services, helper modules)
  • Optional interfaces (debug endpoints, admin tools)
  • Secondary workflows (reporting, maintenance)
  • Non-critical data (logs, metrics)

Review workflows:

High-priority review (unknown + critical impact):

review_item:
  element: constraint.security_slo
  confidence: unknown
  impact: critical
  priority: mandatory
  action: "Must investigate security requirements before deployment"

Medium-priority review (inferred + moderate impact):

review_item:
  element: component.notification_worker
  confidence: inferred
  impact: moderate
  priority: high
  action: "Validate that notification service is correctly inferred"

Low-priority review (observed + non-critical):

review_item:
  element: component.logging_utility
  confidence: observed
  impact: low
  priority: low
  action: "Spot check to verify correctness"

Review effectiveness:

Confidence-based prioritization enables:

  • Efficient resource allocation: Human effort focused where most valuable
  • Risk reduction: High-risk, low-confidence elements receive attention
  • Quality improvement: Systematic review of uncertain elements
  • Scalability: Large-scale extraction remains practical

11.7 Dataset-Level Confidence Metrics

Confidence annotations also support dataset-level quality analysis. When analyzing entire blueprint datasets, confidence metrics provide insights into overall extraction quality.

Confidence distribution metrics:

dataset_metrics:
  total_elements: 15420
  confidence_distribution:
    observed:
      count: 8234
      percentage: 53.4
      average_score: 0.87
    inferred:
      count: 5123
      percentage: 33.2
      average_score: 0.62
    unknown:
      count: 2063
      percentage: 13.4
      average_score: 0.15

Quality indicators:

High-quality dataset:

  • Observed > 50%
  • Inferred ~30-40%
  • Unknown < 15%
  • Average confidence > 0.65

Medium-quality dataset:

  • Observed ~30-50%
  • Inferred ~30-40%
  • Unknown ~15-25%
  • Average confidence ~0.55-0.65

Low-quality dataset:

  • Observed < 30%
  • Inferred ~30-40%
  • Unknown > 25%
  • Average confidence < 0.55

Trend analysis:

Tracking confidence metrics over time enables:

  • Improvement measurement: Is extraction getting more accurate?
  • Methodology validation: Are new techniques improving confidence?
  • Dataset comparison: Which repositories produce better extractions?
  • Issue identification: Where are extraction methods failing?

Example trend analysis:

confidence_trends:
  monthly:
    2026-01:
      observed_percentage: 48.2
      average_confidence: 0.62
    2026-02:
      observed_percentage: 51.1
      average_confidence: 0.65
    2026-03:
      observed_percentage: 53.4
      average_confidence: 0.67

  interpretation: >
    Confidence has improved over the last three months.
    This suggests that extraction methodology improvements
    (better signal detection, improved evidence collection) are
    effectively increasing extraction accuracy.

11.8 Confidence as Architectural Honesty

Confidence semantics represent a philosophical commitment to architectural honesty. Rather than presenting extraction results as definitively correct, confidence semantics acknowledge the limits of available information.

The honesty principle:

Better to admit ignorance than to pretend knowledge.

Applied to extraction:

  • Mark fields as unknown rather than guessing
  • Assign low confidence to weak interpretations
  • Document assumptions and uncertainties
  • Invite human review of uncertain elements

Benefits of honesty:

For users:

  • Accurate understanding: Users know what is reliable versus uncertain
  • Informed decisions: Users can make appropriate decisions based on confidence
  • Risk awareness: Users understand where architectural understanding is weak
  • Trust building: Honesty builds trust in extraction accuracy

For extraction systems:

  • Targeted improvement: Unknown fields identify methodology gaps
  • Quality metrics: Confidence distribution measures overall quality
  • Iterative refinement: Systems can focus on improving low-confidence areas
  • Feedback loops: Review results inform confidence calibration

For architectural datasets:

  • Reliability: Datasets maintain quality by excluding speculation
  • Transparency: Users understand dataset limitations
  • Useability: Users can filter by confidence for appropriate use cases
  • Evolution: Datasets can be systematically improved over time

The alternative: false confidence

Without confidence semantics, extraction risks:

  • Misleading users: Presenting speculation as fact
  • Incorrect decisions: Actions based on incorrect information
  • Wasted effort: Time spent investigating non-existent structures
  • Erosion of trust: Users abandon extraction when it proves unreliable

Confidence semantics prevent these problems by making uncertainty explicit and actionable.

11.9 Chapter Summary

  • Architectural reconstruction is inherently uncertain due to missing information, implicit conventions, ambiguous evidence, and conflicts
  • Confidence semantics provide a formal mechanism for representing uncertainty in extracted architectures
  • Confidence objects include: state (observed/inferred/unknown), score, evidence references, and detailed metadata
  • Three canonical confidence states: observed (explicit evidence), inferred (indirect signals), unknown (insufficient evidence)
  • Confidence is derived from evidence strength, quantity, consistency, and pattern alignment
  • Confidence calculation combines factors: base confidence + evidence boost - consistency penalty + pattern boost
  • Confidence propagates through architectural relationships with attenuation
  • Confidence guides review prioritization: low-confidence, high-impact elements receive highest priority
  • Dataset-level confidence metrics measure overall extraction quality and trends
  • Confidence represents architectural honesty: better to admit ignorance than pretend knowledge
  • Honesty builds trust, enables targeted improvement, and prevents misleading conclusions
