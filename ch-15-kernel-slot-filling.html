17  Kernel Slot Filling Rules

Blueprint extraction ultimately produces a Blueprint Kernel, the minimal structural representation of a system architecture. Once entities and relationships have been extracted from repository evidence, the next step is to populate the kernel slots in a consistent and verifiable way.

Kernel slot filling is therefore governed by explicit rules that determine how extracted information maps into the six canonical slots. These rules ensure that blueprint drafts remain structurally coherent, evidence-backed, and comparable across repositories.

The kernel represents the canonical architecture representation from which different blueprint views can be generated. Kernel slot filling is therefore a critical discipline that transforms extracted entities into a coherent architectural skeleton.

17.1 The Six Core Kernel Slots

The kernel consists of six mandatory slots. Each slot captures a distinct dimension of system architecture. Understanding these slots is essential for effective kernel slot filling.

Kernel slot overview:

kernel_slots:
  Goal:
    description: "High-level problem the system addresses"
    extraction_source: "Repository descriptions, documentation, capabilities"
    complexity: "High - requires synthesis from multiple signals"
    typical_evidence: README, documentation, capability aggregation

  Capabilities:
    description: "Functional abilities provided by the system"
    extraction_source: "Interfaces, workflows, documented operations"
    complexity: "Medium - directly observable from interfaces"
    typical_evidence: API endpoints, CLI commands, workflow steps

  Components:
    description: "Structural units implementing capabilities"
    extraction_source: "Service boundaries, workers, schedulers"
    complexity: "High - requires boundary inference"
    typical_evidence: Entrypoints, deployment configs, directory structure

  Interfaces:
    description: "Interaction surfaces for capability access"
    extraction_source: "HTTP routes, events, CLI commands, RPC"
    complexity: "Medium - explicitly defined in code"
    typical_evidence: Route definitions, topic definitions, command registrations

  Data:
    description: "Persistent state owned or manipulated"
    extraction_source: "Schemas, migrations, ORM models"
    complexity: "Low - explicitly defined in DDL or models"
    typical_evidence: Schema files, ORM classes, migration scripts

  Workflows:
    description: "Execution sequences coordinating components"
    extraction_source: "DAGs, pipelines, schedulers"
    complexity: "High - requires dependency analysis"
    typical_evidence: DAG definitions, pipeline configs, cron schedules

Slot interdependencies:

The six slots are not independent; they form an interconnected structure:

slot_interdependencies:
  Goal → Capabilities:
    relationship: "Goal defines what capabilities should exist"
    validation: "Capabilities should align with goal"

  Capabilities → Interfaces:
    relationship: "Capabilities require interfaces for access"
    validation: "Each capability should have at least one interface"

  Capabilities → Components:
    relationship: "Capabilities are implemented by components"
    validation: "Each capability should be implemented by at least one component"

  Components → Interfaces:
    relationship: "Components expose interfaces"
    validation: "Components should expose at least one interface"

  Components → Data:
    relationship: "Components operate on data"
    validation: "Components that write should own data"

  Components → Workflows:
    relationship: "Components participate in workflows"
    validation: "Workflows should invoke component operations"

  Workflows → Capabilities:
    relationship: "Workflows realize capabilities"
    validation: "Workflows should implement at least one capability"

  triangular_consistency:
    capability ↔ interface ↔ workflow:
      validation: "Each capability should have interface and workflow support"
      strength: "Strong architectural indicator"

Evidence validation rule:

evidence_validation:
  principle: "No kernel slot can be filled without supporting evidence"

  validation_checklist:
    - Each entity in slot must have evidence_refs
    - Evidence must be traceable to repository artifacts
    - Confidence must be assigned based on evidence strength
    - Insufficient evidence must result in inferred or unknown status

  enforcement:
    - Automated validation during extraction
    - Evidence requirement checks before slot population
    - Confidence calculation from evidence strength
    - Unknown status when evidence insufficient

17.3 Optional Constraint Slot

In addition to the six mandatory slots, blueprints may include a Constraint slot. Constraints describe limitations or guarantees affecting the architecture.

Constraint examples:

constraint_slot:
  description: "Architectural limitations or guarantees"

  constraint_categories:
    Security:
      examples:
        - constraint: "All payment data must be encrypted at rest"
          evidence: encryption_configuration
          confidence: 0.92

        - constraint: "API requires authentication for all endpoints"
          evidence: middleware_configuration
          confidence: 0.95

    Performance:
      examples:
        - constraint: "Payment processing must complete within 30 seconds"
          evidence: service_level_agreement
          confidence: 0.78

        - constraint: "API must support 1000 concurrent requests"
          evidence: load_test_config
          confidence: 0.75

    Regulatory:
      examples:
        - constraint: "Payment data must comply with PCI DSS"
          evidence: compliance_documentation
          confidence: 0.85

        - constraint: "User data must comply with GDPR"
          evidence: privacy_policy
          confidence: 0.82

    Deployment:
      examples:
        - constraint: "System must be deployable across multiple regions"
          evidence: infrastructure_configuration
          confidence: 0.88

        - constraint: "Database must support read replicas"
          evidence: deployment_configuration
          confidence: 0.90

Constraint extraction challenges:

constraint_extraction_challenges:
  inconsistency:
    problem: "Constraints often inconsistent across repositories"
    signals:
      - inconsistent_documentation
      - outdated_requirements
      - implicit_assumptions
    strategy: "Use strongest available evidence, document conflicts"

  incompleteness:
    problem: "Constraints rarely documented comprehensively"
    signals:
      - partial_documentation
      - implicit_requirements
      - undocumented_assumptions
    strategy: "Extract only explicitly documented constraints"

  ambiguity:
    problem: "Constraints may be ambiguous or vague"
    signals:
      - vague_requirements
      - subjective_criteria
      - context_dependent_rules
    strategy: "Prefer specific, measurable constraints"

  optional_status:
    rationale: "Constraint signals are inconsistent across repositories"
    treatment: "Constraint slot is optional"
    confidence: "Lower confidence thresholds for constraints"

17.4 Triangular Consistency Checks

Kernel slot filling must maintain structural consistency across slots. A particularly important consistency relationship exists between capabilities, interfaces, and workflows.

The capability-interface-workflow triangle:

consistency_triangle:
  structure:
    capability: "What the system can do"
    interface: "How capabilities are accessed"
    workflow: "How capabilities are executed"

  relationships:
    capability → interface:
      rule: "Each capability should have at least one interface"
      validation: "Check interfaces reference capability"
      strength: strong

    capability → workflow:
      rule: "Each capability should be implemented by at least one workflow"
      validation: "Check workflows implement capability"
      strength: strong

    interface → capability:
      rule: "Each interface should expose at least one capability"
      validation: "Check capabilities expose interface"
      strength: strong

    interface → workflow:
      rule: "Interfaces should be invoked by workflows"
      validation: "Check workflows invoke interface"
      strength: moderate

    workflow → capability:
      rule: "Each workflow should implement at least one capability"
      validation: "Check capability implemented by workflow"
      strength: strong

    workflow → interface:
      rule: "Workflows should invoke interfaces"
      validation: "Check interfaces invoked by workflow"
      strength: moderate

Consistency validation examples:

consistency_validation:
  example_1_consistent:
    capability: create_payment
    interfaces:
      - interface: POST /payments
        exposes: create_payment
    workflows:
      - workflow: payment_processing
        implements: create_payment
        invokes: POST /payments
    validation: PASSED
    confidence: 0.95

  example_2_missing_interface:
    capability: process_payment
    interfaces: []
    workflows:
      - workflow: payment_processing
        implements: process_payment
    validation: FAILED
    issue: "Capability has no interface for access"
    severity: medium
    action: "Mark as inferred, flag for review"

  example_3_missing_workflow:
    capability: create_payment
    interfaces:
      - interface: POST /payments
        exposes: create_payment
    workflows: []
    validation: FAILED
    issue: "Capability has no workflow implementation"
    severity: medium
    action: "Mark as inferred, flag for review"

  example_4_orphaned_interface:
    interface: POST /payments
    exposes: []
    workflows:
      - workflow: payment_processing
        invokes: POST /payments
    validation: FAILED
    issue: "Interface exposes no capability"
    severity: low
    action: "Infer capability from interface"

  example_5_orphaned_workflow:
    workflow: payment_processing
    implements: []
    invokes: [POST /payments]
    validation: FAILED
    issue: "Workflow implements no capability"
    severity: low
    action: "Infer capability from workflow"

Consistency checking algorithm:

consistency_checking_algorithm:
  step_1_build_graph:
    action: "Build capability-interface-workflow graph"
    output: relationships between slots

  step_2_validate_capability_coverage:
    action: "Check each capability has interface and workflow"
    validation:
      - capability must have at least one interface
      - capability must have at least one workflow
    failures:
      - capability_without_interface
      - capability_without_workflow

  step_3_validate_interface_coverage:
    action: "Check each interface exposes capability"
    validation:
      - interface must expose at least one capability
    failures:
      - interface_without_capability

  step_4_validate_workflow_coverage:
    action: "Check each workflow implements capability"
    validation:
      - workflow must implement at least one capability
    failures:
      - workflow_without_capability

  step_5_validate_invocation_consistency:
    action: "Check workflows invoke interfaces"
    validation:
      - workflows should invoke interfaces they use
    failures:
      - workflow_invokes_missing_interface

  step_6_generate_report:
    action: "Generate consistency report"
    output:
      - validation_status
      - identified_issues
      - severity_assessment
      - recommended_actions

Consistency scoring:

consistency_scoring:
  metrics:
    capability_coverage:
      formula: capabilities_with_interface_and_workflow / total_capabilities
      target: "> 0.8"

    interface_coverage:
      formula: interfaces_exposing_capability / total_interfaces
      target: "> 0.9"

    workflow_coverage:
      formula: workflows_implementing_capability / total_workflows
      target: "> 0.8"

    invocation_consistency:
      formula: workflows_invoking_valid_interfaces / total_workflows
      target: "> 0.9"

  overall_consistency_score:
    formula: (capability_coverage + interface_coverage + workflow_coverage + invocation_consistency) / 4
    target: "> 0.85"

  quality_levels:
    excellent: "0.9 - 1.0"
    good: "0.8 - 0.9"
    acceptable: "0.7 - 0.8"
    poor: "< 0.7"

17.6 Chapter Summary

  • Blueprint Kernel is the minimal structural representation of system architecture
  • Kernel consists of six mandatory slots: Goal, Capabilities, Components, Interfaces, Data, Workflows
  • Each slot captures a distinct architectural dimension and requires specific evidence
  • Evidence requirements ensure kernel slots are populated with traceable evidence
  • Goal slot requires synthesis from documentation and capabilities (high complexity)
  • Capabilities slot is directly observable from interfaces and workflows (medium complexity)
  • Components slot requires boundary inference (high complexity)
  • Interfaces slot is explicitly defined in code (medium complexity)
  • Data slot is explicitly defined in schemas (low complexity)
  • Workflows slot requires dependency analysis (high complexity)
  • Optional Constraint slot captures architectural limitations and guarantees
  • Constraint extraction faces inconsistency, incompleteness, and ambiguity challenges
  • Triangular consistency checks validate relationships between capabilities, interfaces, and workflows
  • Consistency validation ensures each capability has interface and workflow support
  • Kernel completion produces a coherent architectural representation with all slots populated
  • Quality metrics measure slot completion, confidence levels, and consistency scores
  • Completed kernel serves as canonical representation for blueprint view generation
