2  Introduction

2.1 The Extraction Challenge

Every software repository contains a system blueprint — but extracting it is notoriously difficult. A repository is a collection of facts (code, configuration, documentation), while a blueprint is an interpretation of how those facts constitute a coherent system. The gap between repository reality and system understanding is the core challenge this volume addresses.

2.2 Why Evidence-Driven Extraction?

Traditional approaches to architecture understanding rely heavily on manual documentation or heuristic-based analysis. These methods suffer from several limitations:

  • Documentation decay: Architectural documentation quickly becomes outdated
  • Heuristic brittleness: Pattern-matching approaches fail at scale
  • Opacity violations: It’s unclear why specific conclusions were drawn
  • Auditability gaps: There’s no way to trace inferences back to source evidence

An evidence-driven approach addresses these challenges by making every inference explicit, traceable, and auditable.

2.3 The Extraction Pipeline

This volume presents a complete extraction pipeline:

Repository → Signal Detection → Evidence Collection → 
Hypothesis Generation → Ontology Mapping → Blueprint Draft → 
Validation → Quality Gate → Usable Blueprint

Each stage produces auditable artifacts and confidence scores, enabling human reviewers to understand, verify, and correct the extraction process.

2.4 Key Principles

2.4.1 1. Evidence Traceability

Every field in the extracted blueprint must be traceable to specific evidence in the repository. This ensures auditability and enables reviewers to validate inferences.

2.4.2 2. Confidence Semantics

Not all inferences are equally reliable. We explicitly model confidence levels (observed, inferred, unknown) to guide review prioritization and risk assessment.

2.4.3 3. Human-in-the-Loop

Automated extraction is powerful but imperfect. We design the pipeline with built-in review gates where human experts can correct, refine, and validate extraction results.

2.4.4 4. Scalability

The methodology scales from single repositories to large-scale mining operations, supporting incremental updates and cost control through cheap-pass/deep-pass strategies.

2.5 What You’ll Learn

After reading this volume, you will understand:

  • How to classify signals that leak system structure from repositories
  • How to build an evidence object model that supports aggregation and conflict resolution
  • How to map repository artifacts to system ontology entities
  • How to generate blueprint drafts through kernel slot filling
  • How to validate, review, and quality-gate extraction results
  • How to operate extraction pipelines at scale

2.6 Who Should Read This Book

This volume is for:

  • System architects who need to understand existing codebases
  • Platform engineers building developer tooling for architecture analysis
  • Researchers studying software architecture at scale
  • Technical leaders managing large software portfolios

2.7 Scope and Boundaries

This volume focuses on:

  • Evidence-driven methodology (not specific tools or implementations)
  • Architecture-level extraction (not detailed code analysis)
  • General principles (not language-specific patterns)

We explicitly exclude:

  • Specific model implementations (ML or symbolic)
  • Language-binding analysis rules
  • Detailed code comprehension beyond architectural level
  • Security vulnerability detection

2.8 The Evidence Mindset

The core shift this volume advocates is from “what is the architecture?” to “what evidence supports this architectural understanding?” This seemingly simple question transformation changes everything:

  • It makes architectural claims testable and refutable
  • It enables systematic improvement of extraction accuracy
  • It supports confidence-based risk assessment
  • It facilitates human-AI collaboration in architectural understanding

2.9 Structure of This Volume

The remaining chapters are organized as follows:

  • Part 0 establishes contracts and boundaries
  • Part I introduces the evidence-driven approach
  • Part II develops signal taxonomy and evidence model
  • Part III presents ontology mapping techniques
  • Part IV describes blueprint generation
  • Part V covers validation and quality gates
  • Part VI addresses scale operations

Each part builds upon the previous ones, culminating in a complete specification (Chapter 21) and practical appendices.

2.10 Getting Started

If you’re new to blueprint extraction, start with Part 0 to understand the scope and contracts, then proceed through Parts I-III to build foundational knowledge. Parts IV-VI can be read independently based on your specific interests.

For practitioners who want to dive straight into implementation, jump to Chapter 21 (the formal specification) and the appendices, then circle back to foundational chapters as needed.

2.11 Acknowledgments

This methodology draws from decades of research in software architecture, reverse engineering, and program analysis. We stand on the shoulders of giants who have explored these challenging problems before us.

Let the evidence guide us to understanding.
