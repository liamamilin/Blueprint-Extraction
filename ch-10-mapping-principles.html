12  Mapping Principles and Canonicalization

Repository artifacts do not directly correspond to system ontology elements. Instead, extraction must translate implementation structures into canonical architectural entities and relationships.

This translation process is called mapping. Mapping converts repository-derived evidence into the standardized vocabulary defined by the system ontology. The goal is to produce architectural representations that remain consistent across repositories, languages, and frameworks.

Without mapping, extraction would produce repository-specific representations that cannot be compared, analyzed, or aggregated. With mapping, extraction produces canonical representations that enable large-scale architecture datasets, pattern discovery, and architectural intelligence.

12.1 The Mapping Challenge

Repositories exhibit enormous diversity in how they represent the same architectural concepts. This diversity presents the core mapping challenge: how to translate diverse implementation representations into consistent architectural entities?

Diversity examples:

Same concept, different representations:

Architectural Concept Python Go Java JavaScript
Background Worker Celery Task goroutine ScheduledExecutor Worker Thread
API Interface FastAPI route HTTP handler (RestController?) Express route
Data Entity SQLAlchemy model struct JPA entity Mongoose schema
Configuration .env file struct tags (Configuration?) config.json
Workflow Airflow DAG Cron job (Scheduled?) Bull queue

Variation dimensions:

  1. Language-specific syntax: Different languages use different constructs
  2. Framework conventions: Different frameworks encourage different patterns
  3. Organizational standards: Different teams use different naming and organization
  4. Evolutionary drift: Systems evolve without updating conventions
  5. Implementation preferences: Developers choose different approaches for the same problem

The mapping objective:

Transform this diversity into canonical architectural representations:

Diverse Implementation Artifacts
              ↓
           Mapping
              ↓
Canonical Ontology Entities

Where “canonical” means: - Standardized: Using consistent terminology and structure - Technology-agnostic: Independent of language or framework - Architectural: Focused on structure rather than implementation - Comparable: Enabling cross-repository analysis

12.2 Interpretability First

Mapping rules must remain interpretable. This is a non-negotiable requirement for reliable extraction systems.

Why interpretability matters:

  1. Validation: Reviewers must verify that mappings are correct
  2. Debugging: Errors must be traced to specific mapping rules
  3. Evolution: Mappings must be improved over time
  4. Trust: Users must understand how extraction produces results

Opaque mappings are problematic:

# Bad: Opaque mapping rule
def map_component(evidence):
    result = complex_ml_model(evidence)
    return result  # Black box, unexplainable

Interpretable mappings are necessary:

# Good: Interpretable mapping rule
def map_component(evidence):
    if evidence.type == 'route_definition':
        return Component(type='Service', role='API')
    elif evidence.type == 'worker_implementation':
        return Component(type='Worker', role='Background')
    elif evidence.type == 'scheduler_config':
        return Component(type='Scheduler', role='Orchestration')
    else:
        return Component(type='Unknown', role='Unknown')

Interpretability characteristics:

1. Explicit rules:

mapping_rules:
  - rule: route_to_service_component
    condition:
      artifact_type: code
      pattern: "@app.route|@GetMapping|http.HandleFunc"
    mapping:
      entity_type: Component
      subtype: Service
      role: API

2. Documented rationale:

mapping_rules:
  - rule: route_to_service_component
    rationale: |
      HTTP route definitions indicate that the containing
      component is a service that exposes an API interface.
      This is a consistent pattern across most web frameworks.

3. Reviewable logic:

mapping_rules:
  - rule: queue_to_worker_component
    logic:
      - detect queue dependency
      - detect consumer implementation
      - confirm processing pattern
    decision: component_type == 'Worker'

4. Debuggable outputs:

mapping_result:
  input_evidence: evidence-001
  applied_rule: route_to_service_component
  intermediate_steps:
    - detected: "@app.route('/payments')"
    - matched: route_definition_pattern
    - classified: API service
  output_entity: component.api_service
  confidence: 0.92

When machine learning is used:

Even when machine learning or pattern recognition assists extraction, the final mapping must still be expressed through clear, explainable rules:

ml_assisted_mapping:
  model_prediction:
    entity_type: Component
    subtype: Worker
    confidence: 0.78
  post_processing_rules:
    - rule: validate_worker_indicators
      checks:
        - queue_dependency_present
        - consumer_implementation_exists
        - background_execution_pattern
    final_decision:
      entity_type: Component
      subtype: Worker
      confidence: 0.85
      rationale: |
        ML prediction validated by worker-specific
        indicators. All checks passed.

Interpretability ensures that:

  • Mappings can be reviewed and validated
  • Errors can be debugged and corrected
  • Mappings can be improved systematically
  • Extraction results can be trusted

12.3 Naming Canonicalization

Repositories frequently use inconsistent naming conventions. Different projects may refer to the same architectural concept using different terms. Naming canonicalization resolves these inconsistencies by translating repository-specific names into standardized ontology labels.

The naming problem:

Same concept, different names:

Concept Repository A Repository B Repository C
Background processing worker consumer processor
Payment processing payment billing transaction
User management user account customer
Data storage storage database persistence
API layer api interface endpoint

Canonicalization examples:

canonicalization_rules:
  # Component type normalization
  worker:
    variants: [worker, consumer, processor, job-runner, task]
    canonical: WorkerComponent

  service:
    variants: [service, server, application, app]
    canonical: ServiceComponent

  scheduler:
    variants: [scheduler, orchestrator, coordinator, cron]
    canonical: SchedulerComponent

  # Interface type normalization
  api:
    variants: [api, endpoint, route, handler]
    canonical: APIInterface

  event:
    variants: [event, message, topic, queue]
    canonical: EventInterface

  cli:
    variants: [cli, command, tool, utility]
    canonical: CLIInterface

  # Data type normalization
  table:
    variants: [table, entity, model, document]
    canonical: TableEntity

  collection:
    variants: [collection, bucket, container]
    canonical: CollectionEntity

Canonicalization process:

canonicalization_pipeline:
  input:
    repository_name: "PaymentProcessorService"
    detected_type: service

  step_1_normalize:
    lowercase: "paymentprocessorservice"
    remove_suffixes: "paymentprocessor"

  step_2_tokenization:
    tokens: ["payment", "processor"]

  step_3_mapping:
    payment -> canonical: "Payment"
    processor -> canonical: "Processor"

  step_4_assembly:
    canonical_name: "PaymentProcessorService"

  output:
    canonical_id: component.payment_processor_service
    canonical_name: PaymentProcessorService
    canonical_type: ServiceComponent

Benefits of canonical naming:

  1. Searchability: Users can find related components across repositories
  2. Comparability: Architectures can be compared despite different naming
  3. Analysis: Pattern mining works on consistent terminology
  4. Clarity: Canonical names have clear, unambiguous meaning

Handling ambiguous names:

ambiguous_name_handling:
  input: "service"

  context_analysis:
    - check: route_definitions_present
      if_true: canonical: ServiceComponent
    - check: worker_implementation_present
      if_true: canonical: WorkerComponent
    - check: scheduler_config_present
      if_true: canonical: SchedulerComponent

  default:
    if_no_context: canonical: UnknownComponent
    confidence: low
    notes: "Ambiguous term 'service' without context"

12.4 Type Canonicalization

Beyond naming differences, repositories may represent similar architectural elements using different technical constructs. Type canonicalization abstracts these variations into common architectural types defined in the system ontology.

The type problem:

Same architectural role, different implementation:

Architectural Role Python Go Java JavaScript
Async processing Celery channel ExecutorService Worker
HTTP interface FastAPI net/http Spring Boot Express
Data persistence SQLAlchemy sqlx Hibernate Mongoose
Configuration .env struct (Value?) config.json
Logging logging log SLF4J console.log

Type canonicalization examples:

type_canonicalization_rules:
  # Async processing canonicalization
  async_processing:
    implementations:
      - celery: Python
      - bull: Node.js
      - sidekiq: Ruby
      - channel: Go
      - ExecutorService: Java
    canonical: WorkflowExecutionNode

  # HTTP interface canonicalization
  http_interface:
    implementations:
      - fastapi: Python
      - express: Node.js
      - http.HandleFunc: Go
      - @RestController: Java
      - @app.route: Python
    canonical: APIInterface

  # Data persistence canonicalization
  data_persistence:
    implementations:
      - sqlalchemy: Python
      - orm: Python
      - gorm: Go
      - hibernate: Java
      - mongoose: Node.js
    canonical: DataEntity

Canonicalization example:

input_evidence:
  repository: python-payment-system
  implementation:
    framework: celery
    code: |
      @app.task
      def process_payment(payment_id):
          # Process payment
          pass

canonicalization:
  detected_pattern: celery_task
  mapped_to: WorkflowExecutionNode
  canonical_entity:
    type: WorkflowExecutionNode
    role: BackgroundProcessing
    implementation: celery
  confidence: 0.92

Canonicalization benefits:

  1. Abstraction: Focus on architectural role, not implementation
  2. Comparison: Compare architectures across technology stacks
  3. Pattern mining: Find patterns despite implementation differences
  4. Flexibility: Accommodate new implementations without changing ontology

Preserving implementation details:

While canonicalizing types, we preserve implementation details as metadata:

canonical_entity:
  id: workflow.payment_processing
  type: WorkflowExecutionNode
  role: BackgroundProcessing
  canonical_type: WorkflowExecutionNode

  implementation_metadata:
    original_implementation: celery
    language: python
    framework: celery
    version: "5.3.0"

This allows both canonical analysis (what it is architecturally) and implementation-aware analysis (how it’s implemented).

12.5 Relation Canonicalization

Architectural relationships between entities also require normalization. Repositories may express relationships in many forms, including function calls, configuration references, dependency declarations, and event subscriptions.

The relationship problem:

Same relationship, different representations:

Relationship Code Config Dependency Event
Depends on import import: requires: subscribes:
Exposes export routes: provides: publishes:
Reads from query db: uses: reads:
Writes to insert db: modifies: writes:

Canonical relationship types:

canonical_relationships:
  # Component relationships
  component_exposes_interface:
    variants: [exposes, provides, exports, routes]
    canonical: exposes

  component_depends_on_component:
    variants: [depends, imports, requires, uses]
    canonical: depends

  component_owns_data:
    variants: [owns, manages, controls, writes]
    canonical: owns

  component_reads_data:
    variants: [reads, queries, accesses, retrieves]
    canonical: reads

  # Workflow relationships
  workflow_triggers_component:
    variants: [triggers, invokes, executes, calls]
    canonical: triggers

  workflow_contains_task:
    variants: [contains, includes, has, steps]
    canonical: contains

  # Data relationships
  data_stored_in:
    variants: [stored_in, persisted_in, saved_to]
    canonical: stored_in

  data_accessed_by:
    variants: [accessed_by, read_by, queried_by]
    canonical: accessed_by

Relation canonicalization example:

input_relationship:
  repository: go-microservice
  source: "import 'github.com/user/payments'"
  relationship_type: import

canonicalization:
  detected_pattern: go_import
  mapped_to: component_depends_on_component
  canonical_relationship:
    source: component.order_service
    relation: depends
    target: component.payment_service
    type: ComponentDependency
  confidence: 0.95

Canonicalization process:

relationship_canonicalization_pipeline:
  input:
    raw_relationship: "import payments"
    context: source=order_service, target=payments

  step_1_classification:
    pattern_match: import_statement
    category: dependency

  step_2_semantic_analysis:
    source_type: Component
    target_type: Component
    semantic_role: dependency

  step_3_canonical_mapping:
    canonical_relation: depends

  step_4_confidence_assessment:
    evidence: explicit_import
    confidence: 0.95

  output:
    canonical_relationship:
      source: component.order_service
      relation: depends
      target: component.payment_service
      confidence: 0.95

Benefits of relation canonicalization:

  1. Consistency: Same relationship expressed consistently across systems
  2. Analysis: Relationship patterns can be mined across repositories
  3. Validation: Architectural constraints can be checked
  4. Visualization: Consistent graphs can be generated

12.6 Entity and Relation Graphs

The result of mapping and canonicalization is a pair of structured graphs: the entity graph and the relation graph. Together, these graphs form the structural foundation of the blueprint.

12.6.1 Entity Graph

The entity graph contains the architectural entities identified during extraction.

Entity graph structure:

entity_graph:
  nodes:
    - id: component.api_service
      type: Component
      subtype: Service
      role: API
      capabilities:
        - capability.create_payment
        - capability.get_payment
      interfaces:
        - interface.http_api
      data:
        - data.payments
      confidence: 0.95

    - id: component.payment_worker
      type: Component
      subtype: Worker
      role: Background
      capabilities:
        - capability.process_payment
      interfaces:
        - interface.queue
      data:
        - data.payments
      confidence: 0.88

    - id: component.notification_service
      type: Component
      subtype: Service
      role: Notification
      capabilities:
        - capability.send_notification
      interfaces:
        - interface.http_api
        - interface.email
      confidence: 0.82

Entity types:

entity_types:
  Component:
    subtypes: [Service, Worker, Scheduler, Plugin]
    fields: [id, type, subtype, role, capabilities, interfaces, data]

  Interface:
    subtypes: [API, Event, CLI, RPC]
    fields: [id, type, subtype, contract, protocol]

  Data:
    subtypes: [Table, Collection, Document]
    fields: [id, type, subtype, schema, ownership]

  Workflow:
    subtypes: [Pipeline, DAG, Schedule]
    fields: [id, type, subtype, tasks, triggers]

  Capability:
    subtypes: [Functional, Operational]
    fields: [id, type, description, interfaces]

12.6.2 Relation Graph

The relation graph represents the relationships between entities.

Relation graph structure:

relation_graph:
  edges:
    - source: component.api_service
      relation: exposes
      target: interface.http_api
      type: ComponentExposesInterface
      confidence: 0.95

    - source: component.api_service
      relation: owns
      target: data.payments

    - source: component.api_service
      relation: depends

      confidence: 0.88

    - source: workflow.payment_processing
      relation: triggers
      target: component.payment_worker
      type: WorkflowTriggersComponent
      confidence: 0.85

    - source: component.payment_worker
      relation: reads
      target: data.payments
      type: ComponentReadsData
      confidence: 0.90

Relation types:

relation_types:
  ComponentExposesInterface:
    direction: component → interface
    semantics: component provides access through interface

  ComponentOwnsData:
    direction: component → data
    semantics: component is primary authority for data

  ComponentReadsData:
    direction: component → data
    semantics: component reads but does not own data

  ComponentDependency:
    direction: component → component
    semantics: source depends on target

  WorkflowTriggersComponent:
    direction: workflow → component
    semantics: workflow invokes component

  WorkflowContainsTask:
    direction: workflow → task
    semantics: workflow includes task in execution

The entity and relation graphs together form a complete architectural representation:

blueprint_graph:
  entities:
    component.api_service:
      type: Component
      subtype: Service
      role: API
      relations:
        - exposes: interface.http_api
        - owns: data.payments
        - depends: component.notification_service

    component.payment_worker:
      type: Component
      subtype: Worker

      relations:
        - triggered_by: workflow.payment_processing

    component.api_service → component.notification_service: depends

Graph properties:

  • Directed: Relationships have direction (source → target)
  • Typed: Each relationship has a canonical type
  • Weighted: Relationships have confidence scores
  • Attributed: Relationships include metadata and evidence references

12.7 Mapping Rules Interface

To ensure consistency and interoperability, mapping rules follow a standardized interface. This interface defines how mapping rules are specified, applied, and validated.

Mapping rule structure:

mapping_rule:
  id: rule_id
  version: "1.0"
  name: "Descriptive rule name"

  description: |
    Clear explanation of what this rule does and why.

  detection:
    artifact_types: [code, config, schema, deploy]
    patterns:
      - pattern: "@app.route|@GetMapping|http.HandleFunc"
        language: regex
      - pattern: fastapi|flask|django
        language: string_match

  mapping:
    entity_type: Component
    entity_subtype: Service
    entity_role: API
    canonical_type: ServiceComponent

  confidence:
    base_confidence: 0.8
    boost_conditions:
      - condition: openapi_spec_present
        boost: 0.1
      - condition: test_coverage_present
        boost: 0.05

  evidence_requirements:
    min_evidence: 1
    preferred_evidence: 2
    required_types: [code, config]

  validation:
    required_context:
      - route_definitions
      - server_initialization
    forbidden_context:
      - static_file_serving

  examples:
    - input:
        code: "@app.route('/payments')"
      output:
        entity_type: Component
        entity_subtype: Service
    - input:
        code: "@GetMapping('/payments')"
      output:
        entity_type: Component
        entity_subtype: Service

  metadata:
    author: "Blueprint Extraction Team"
    created: "2026-03-08"
    last_modified: "2026-03-08"

Rule application process:

rule_application:
  input_evidence:
    id: evidence-001
    artifact_type: code
    excerpt: "@app.route('/payments')"

  step_1_rule_matching:
    matched_rules:
      - rule: route_to_service_component
        match_score: 0.95

  step_2_context_validation:
    context_checks:
      - check: server_initialization_present
        result: true
      - check: static_file_serving
        result: false

  step_3_mapping:
    mapping_result:
      entity_type: Component
      entity_subtype: Service
      entity_role: API

  step_4_confidence_calculation:
    base_confidence: 0.8
    context_boost: 0.0
    final_confidence: 0.8

  step_5_output:
    entity:
      id: component.api_service
      type: Component
      subtype: Service
      role: API
      confidence: 0.8
      evidence_refs: [evidence-001]

Rule validation:

Mapping rules must be validated before use:

rule_validation:
  required_fields:
    - id ✓
    - name ✓
    - detection ✓
    - mapping ✓

  semantic_validation:
    - detection_patterns_valid: true
    - mapping_entity_type_valid: true
    - confidence_range_valid: true

  operational_validation:
    - rule_executable: true
    - produces_valid_output: true
    - evidence_traceable: true

  validation_result: PASSED

12.8 Mapping Best Practices

Effective mapping follows several best practices that ensure quality, maintainability, and reliability.

12.8.1 1. Start Simple, Add Complexity Gradually

# Start with simple rule
mapping_rule_v1:
  detection:
    pattern: "@app.route"
  mapping:
    entity_type: Component

# Add complexity over time
mapping_rule_v2:
  detection:
    pattern: "@app.route|@GetMapping|http.HandleFunc"
    language: "web framework"
  mapping:
    entity_type: Component
    entity_subtype: Service
  context_validation:
    - server_initialization_required
  confidence_calculation:
    base: 0.8
    boost: test_coverage * 0.1

12.8.2 2. Document Everything

mapping_rule:
  documentation:
    purpose: "Identify API service components"
    rationale: |
      HTTP route definitions indicate API services because
      they expose HTTP interfaces. This pattern is consistent
      across web frameworks.
    assumptions:
      - Route definitions indicate external interfaces
      - Server initialization confirms service boundary
    known_limitations:
      - May misidentify internal-only routes
      - Requires context to distinguish services from modules
    validation_approach: "Check for server initialization"

12.8.3 3. Handle Edge Cases

mapping_rule:
  edge_case_handling:
    - case: "static file serving"
      detection: "@app.route('/static')"
      handling: |
        Static file routes do not indicate API services.
        Exclude this pattern or downgrade confidence.
      action: exclude

    - case: "internal routes"
      detection: "@app.route('/_internal')"
      handling: |
        Internal routes may indicate internal services or
        just internal endpoints. Require additional context.
      action: lower_confidence_by_0.2

    - case: "framework defaults"
      detection: "@app.route('/')"
      handling: |
        Root route alone insufficient. Look for additional
        routes or service-specific configuration.
      action: require_additional_evidence

12.8.4 4. Use Confidence Appropriately

mapping_rule:
  confidence_strategy:
    high_confidence:
      conditions:
        - explicit_route_definition
        - server_initialization
        - openapi_specification
      confidence: 0.9+

    medium_confidence:
      conditions:
        - explicit_route_definition
        - server_initialization
      confidence: 0.7-0.9

    low_confidence:
      conditions:
        - route_definition_only
        - no_server_context
      confidence: 0.4-0.7

    no_confidence:
      conditions:
        - conflicting_evidence
      confidence: 0.0-0.4

12.8.5 5. Preserve Evidence Traceability

mapping_rule:
  evidence_preservation:
    required_evidence_types:
      - code (route definitions)
      - config (server configuration)

    optional_evidence_types:
      - schema (OpenAPI spec)
      - test (API tests)

    evidence_linking:
      - entity.evidence_refs: all_supporting_evidence
      - entity.confidence_details: evidence_analysis
      - entity.mapping_details: rule_application_log

12.9 Mapping as Structural Translation

Mapping and canonicalization transform repository evidence into standardized architectural structures. This translation is essential for producing useful, comparable blueprints.

Translation process summary:

Repository Evidence
        ↓
    Mapping Rules
        ↓
  Interpretability
        ↓
 Naming Canonicalization
        ↓
  Type Canonicalization
        ↓
Relation Canonicalization
        ↓
   Entity Graph
        ↓
  Relation Graph
        ↓
Canonical Blueprint

Key transformations:

  • Interpretability: Ensure mappings are explainable and reviewable
  • Naming normalization: Standardize names across repositories
  • Type normalization: Abstract implementation differences
  • Relation normalization: Standardize relationship representations
  • Graph construction: Assemble entities and relations into structured graphs

The transformation payoff:

Canonical entity and relation graphs provide a canonical architectural representation that can be used to:

  • Populate blueprint kernels with standardized entities
  • Generate blueprint views through projection rules
  • Support large-scale architecture analysis and comparison
  • Enable architecture pattern mining and discovery
  • Build architecture intelligence systems

Mapping is therefore the bridge that transforms implementation-specific observations into architecture-independent understanding.

12.10 Chapter Summary

  • Mapping translates repository artifacts into canonical architectural entities and relationships
  • The mapping challenge arises from diversity in languages, frameworks, and conventions
  • Interpretability is a non-negotiable requirement for mapping rules
  • Naming canonicalization standardizes entity names across repositories
  • Type canonicalization abstracts implementation differences into architectural types
  • Relation canonicalization standardizes relationship representations
  • Mapping produces entity graphs (architectural entities) and relation graphs (relationships)
  • Mapping rules follow a standardized interface with detection, mapping, confidence, and validation
  • Best practices include starting simple, documenting everything, handling edge cases, using confidence appropriately, and preserving evidence traceability
  • Mapping transforms implementation-specific observations into architecture-independent understanding
  • Canonical graphs enable blueprint population, view generation, and large-scale architecture analysis
