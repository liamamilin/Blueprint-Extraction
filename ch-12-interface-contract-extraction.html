14  Interface and Contract Extraction

Interfaces define how a system interacts with external actors and internal components. They represent the points of interaction through which capabilities are accessed and coordinated. During blueprint extraction, interfaces are reconstructed from repository artifacts that expose interaction surfaces.

Interface extraction therefore focuses on identifying these interaction mechanisms and reconstructing the contracts that govern their behavior. The output of this stage is an interface graph together with a set of contract objects that formally describe interaction semantics.

Interfaces are critical architectural elements because they define:

  • How external actors (users, other systems) interact with the system
  • How internal components coordinate and communicate
  • What capabilities the system provides
  • What constraints govern these interactions

14.1 What Is an Interface?

An interface is a well-defined interaction surface that specifies how capabilities are invoked or accessed. Interfaces define the contract between providers and consumers of functionality.

Interface characteristics:

1. Interaction surface:

Interfaces provide access points through which functionality is invoked.

interface:
  id: interface.payment_api
  type: API
  role: "External interaction surface"
  exposed_by: component.payment_service
  description: "HTTP API for payment processing"

2. Contract specification:

Interfaces define the rules governing interactions, including request formats, response structures, and error handling.

interface:
  id: interface.payment_api
  contract:
    request_format: JSON
    response_format: JSON
    authentication: Bearer token
    rate_limit: 100 requests/minute
    error_codes: [400, 401, 404, 500]

3. Capability access:

Interfaces provide access to specific capabilities or operations.

interface:
  id: interface.payment_api
  capabilities:
    - capability.create_payment
    - capability.get_payment
    - capability.refund_payment

4. Protocol definition:

Interfaces specify the communication protocol used for interaction.

interface:
  id: interface.payment_api
  protocol:
    name: HTTP
    version: 1.1
    methods: [POST, GET, PUT, DELETE]
    content_type: application/json

Interface types:

The interface ontology defines several common interface archetypes:

interface_types:
  API:
    role: "Synchronous request-response interaction"
    protocols: [HTTP, HTTPS, gRPC, GraphQL]
    typical_consumers: [web applications, mobile apps, external systems]
    examples: [REST APIs, GraphQL endpoints, gRPC services]

  Event:
    role: "Asynchronous message-based interaction"
    protocols: [message queues, event streams, pub/sub]
    typical_consumers: [workers, microservices, event handlers]
    examples: [Kafka topics, RabbitMQ queues, Redis streams]

  CLI:
    role: "Command-line interaction"
    protocols: [stdin/stdout, command-line arguments]
    typical_consumers: [operators, automation scripts, cron jobs]
    examples: [CLI tools, administrative commands, batch jobs]

  RPC:
    role: "Remote procedure call interaction"
    protocols: [gRPC, Thrift, Protocol Buffers]
    typical_consumers: [microservices, internal services]
    examples: [gRPC services, Thrift interfaces]

API interfaces are the most common form of system interface. They are typically implemented through web frameworks that define route handlers and request processing logic.

API interface signals:

        return ResponseEntity.ok(payment);

    @GetMapping("/{paymentId}")
    public ResponseEntity<Payment> getPayment(@PathVariable String paymentId) {
        Payment payment = paymentService.get(paymentId);

    }
}

14.3 Event Interface Extraction

Many modern systems rely on asynchronous communication through event streams or messaging infrastructure. Event interfaces expose the system through topics, queues, or event schemas that allow components or external systems to publish and consume events.

Event interface signals:

1. Message topic definitions:

# event_topics.py
from kafka import KafkaProducer, KafkaConsumer
import json

KAFKA_BROKER = 'localhost:9092'

TOPICS = {
    'payment_created': 'payments.created',
    'payment_completed': 'payments.completed',
    'payment_failed': 'payments.failed',
    'refund_requested': 'payments.refund.requested',
}

class PaymentEventProducer:
    def __init__(self):
        self.producer = KafkaProducer(
            bootstrap_servers=KAFKA_BROKER,
            value_serializer=lambda v: json.dumps(v).encode('utf-8')
        )

    def publish_payment_created(self, payment):
        self.producer.send(
            TOPICS['payment_created'],
            value={
                'payment_id': payment.id,
                'amount': float(payment.amount),
                'status': payment.status,
                'created_at': payment.created_at.isoformat()
            }
        )

class PaymentEventConsumer:
    def __init__(self, topic):
        self.consumer = KafkaConsumer(
            topic,
            bootstrap_servers=KAFKA_BROKER,
            value_deserializer=lambda m: json.loads(m.decode('utf-8'))
        )

    def consume_events(self):
        for message in self.consumer:
            yield message.value

package main

import (

func main() {
    createCmd := flag.NewFlagSet("create", flag.ExitOnError)
    amount := createCmd.Float("amount", 0, "Payment amount")
    currency := createCmd.String("currency", "USD", "Payment currency")
    method := createCmd.String("method", "credit_card", "Payment method")

    getCmd := flag.NewFlagSet("get", flag.ExitOnError)
    paymentID := getCmd.String("id", "", "Payment ID")

    refundCmd := flag.NewFlagSet("refund", flag.ExitOnError)
    refundID := refundCmd.String("id", "", "Payment ID to refund")
    refundAmount := refundCmd.Float("amount", 0, "Refund amount")

    if len(os.Args) < 2 {
        fmt.Println("Expected 'create', 'get', or 'refund' subcommands")
        os.Exit(1)
    }

    switch os.Args[1] {
    case "create":
        createCmd.Parse(os.Args[2:])
        paymentService.Create(*amount, *currency, *method)
    case "get":
        getCmd.Parse(os.Args[2:])
        paymentService.Get(*paymentID)
    case "refund":
        refundCmd.Parse(os.Args[2:])
        paymentService.Refund(*refundID, *refundAmount)
    default:
        fmt.Println("Unknown subcommand")
    }
}

14.5 Contract Reconstruction

Interfaces alone describe interaction entrypoints. Contracts define the rules governing those interactions. Contract extraction attempts to reconstruct information such as request structures, response formats, validation rules, and error handling.

Contract extraction elements:

1. Request and response structures:

14.7 Chapter Summary

  • Interfaces are well-defined interaction surfaces that specify how capabilities are accessed
  • Interfaces define contracts between providers and consumers, including request formats, response structures, and protocols
  • Four main interface types: API (synchronous request-response), Event (asynchronous messages), CLI (command-line), RPC (remote procedure call)
  • API interfaces are extracted from route declarations, controllers, OpenAPI specifications, and RPC definitions
  • Event interfaces are extracted from topic definitions, event schemas, consumer implementations, and broker configurations
  • CLI interfaces are extracted from command registration, argument parsing, command modules, and embedded documentation
  • Contract reconstruction extracts request/response structures, parameter schemas, event payloads, command arguments, and validation constraints
  • Contracts may be partial or implicit when not explicitly defined
  • The interface graph represents relationships between components and their interaction surfaces
  • Interface graph analysis reveals external vs. internal interfaces, type distribution, component relationships, and critical interfaces
  • Interfaces and contracts together represent the system’s communication layer
