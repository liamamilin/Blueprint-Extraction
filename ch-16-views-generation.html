18  Views Generation (Projections)

The Blueprint Kernel represents the canonical structural model of a system. However, different stakeholders require different perspectives on that structure. Developers, operators, and reviewers often focus on distinct aspects of the system architecture. A single representation cannot simultaneously optimize for all of these viewpoints.

Blueprint views solve this problem by providing projections of the kernel. A view is a derived representation that reorganizes kernel information to emphasize a particular architectural dimension. Views do not introduce new architectural facts; they rearrange existing kernel entities and relationships into different analytical perspectives.

Blueprint extraction therefore follows a kernel-first, views-later approach. Once the kernel has been populated and validated, views are generated through deterministic projection rules. This approach ensures that all views remain consistent with the underlying architectural model.

18.1 Kernel-First, Views-Later Approach

The extraction process prioritizes kernel construction over view generation. This ordering ensures architectural integrity and enables flexible view generation.

Why kernel-first matters:

kernel_first_principles:
  architectural_integrity:
    principle: "Kernel is the single source of truth"
    benefit: "Prevents inconsistencies between views"
    mechanism: "Views are derived from kernel, not independently"

  evidence_traceability:
    principle: "Evidence traces to kernel, not to views"
    benefit: "Simplifies audit and validation"
    mechanism: "Kernel entities reference evidence directly"

  update_consistency:
    principle: "Kernel updates propagate to all views"
    benefit: "Views automatically stay synchronized"
    mechanism: "Deterministic projection rules"

  flexibility:
    principle: "New views can be added without changing kernel"
    benefit: "Enables future extensibility"
    mechanism: "View generation is independent of kernel structure"

View generation pipeline:

view_generation_pipeline:
  step_1_kernel_construction:
    action: "Construct canonical kernel from evidence"
    output: "BlueprintKernel with six slots"

  step_2_kernel_validation:
    action: "Validate kernel completeness and consistency"
    output: "Validated kernel with quality metrics"

  step_3_view_projection:
    action: "Generate views through projection rules"
    output: "Multiple view representations"

  step_4_view_validation:
    action: "Validate views against kernel"
    output: "Consistent, validated views"

  dependency: "Each step depends on previous step completion"

Projection vs. extraction:

projection_vs_extraction:
  extraction:
    process: "Infer architecture from repository artifacts"
    direction: "Artifacts → Kernel"
    complexity: "High - requires interpretation and evidence"
    uncertainty: "Significant - involves inference"

  projection:
    process: "Reorganize kernel into perspectives"
    direction: "Kernel → Views"
    complexity: "Low - deterministic transformation"
    uncertainty: "Minimal - no new inference"

18.2 Product View

The product view describes the system from the perspective of user-visible functionality. It focuses on the capabilities the system provides and the interfaces through which those capabilities are accessed. The purpose of this view is to explain what the system does rather than how it is implemented.

Product view characteristics:

product_view:
  focus: "User-visible functionality"
  audience: [product_managers, users, stakeholders]
  questions_answered:
    - "What can this system do?"
    - "How do users interact with it?"
    - "What problems does it solve?"
  excludes:
    - "Implementation details"
    - "Component structure"
    - "Data models"
    - "Operational aspects"

Product view projection rules:

product_view_projection:
  source_slots:
    - Goal (primary)
    - Capabilities (primary)
    - Interfaces (primary)

  projection_rules:
    - rule: "Include all capabilities"
      source: kernel.Capabilities
      output: view.capabilities

    - rule: "Include external interfaces"
      source: kernel.Interfaces
      filter: "interfaces accessible to external users"
      output: view.interfaces

    - rule: "Include high-level goal"
      source: kernel.Goal
      output: view.system_purpose

    - rule: "Exclude internal interfaces"
      source: kernel.Interfaces
      filter: "exclude interfaces for internal use"
      output: none

    - rule: "Exclude implementation details"
      source: kernel.Components
      output: none

    - rule: "Exclude data models"
      source: kernel.Data
      output: none

    - rule: "Exclude workflows"
      source: kernel.Workflows
      output: none

Product view example:

product_view:
  view_name: Product View
  view_type: user_facing
  generated_from: kernel.payment_system_v1

  system_purpose:
    value: "Process payment transactions and manage refunds for e-commerce platforms"

  capabilities:

        - interface: POST /payments
          protocol: HTTP
          access: public

    - id: capability.process_payment
      name: "Process Payment"
      description: "Process payment through payment gateway"
      interfaces:
        - interface: payment_events (payment.created)
          protocol: Event
          access: internal
      confidence: 0.90

    - id: capability.refund_payment
      name: "Refund Payment"
      description: "Process payment refunds"
      interfaces:
        - interface: POST /payments/{id}/refund
          protocol: HTTP
          access: public
      confidence: 0.92

    - id: capability.send_notification
      name: "Send Notifications"
      description: "Send payment status notifications"
      interfaces:
        - interface: payment_events (payment.completed)
          protocol: Event
          access: internal
      confidence: 0.88

  external_interfaces:
    - id: interface.payment_api
      type: API
      protocol: HTTP
      access: public
      capabilities:
        - create_payment
        - refund_payment
      confidence: 0.98

  user_stories:
    - story: "As a merchant, I want to create payments"
      capability: create_payment

    - story: "As a merchant, I want to refund payments"

  limitations:
    - note: "Product view is partial, inferred from capabilities"
    - confidence: "Product intent may not be explicitly documented"
    - recommendation: "Review with product team for validation"

Product view generation challenges:

product_view_challenges:
  challenge_1_implicit_product_intent:
    problem: "Repositories rarely encode product intent explicitly"
    signals:
      - README files may be incomplete
      - Product documentation may be outdated
      - User stories may be missing
    mitigation: "Infer product view from capabilities and interfaces"
    confidence: "Lower confidence for product view"

  challenge_2_capability_to_feature_mapping:
    problem: "Technical capabilities may not map directly to user features"
    signals:
      - Multiple capabilities may form one feature
      - One capability may support multiple features
      - Feature boundaries may be ambiguous
    mitigation: "Group capabilities into logical features"
    confidence: "Requires domain knowledge"

  challenge_3_user_interface_abstraction:
    problem: "API interfaces may not match user interaction model"
    signals:
      - APIs expose technical operations
      - Users interact through different abstractions
      - Command-line vs. GUI differences
    mitigation: "Map interfaces to user actions"
    confidence: "Moderate"

  recommendation: "Product view should be constructed cautiously and validated with product stakeholders"

18.3 Architecture View

The architecture view is the primary artifact produced by blueprint extraction. This view focuses on the structural composition of the system and the relationships between its core elements.

Architecture view characteristics:

architecture_view:
  focus: "Structural composition and relationships"
  audience: [architects, developers, system_designers]
  questions_answered:
    - "How is the system structured?"
    - "What components exist and how do they relate?"
    - "How do components interact?"
    - "What data do components operate on?"
  includes:
    - "Component structure"
    - "Interface relationships"
    - "Data ownership"
    - "Workflow coordination"
  excludes:
    - "Implementation details"
    - "Runtime deployment"
    - "Operational monitoring"

Architecture view projection rules:

architecture_view_projection:
  source_slots:
    - Components (primary)
    - Interfaces (primary)
    - Data (primary)
    - Workflows (primary)
    - Capabilities (secondary)

  projection_rules:
    - rule: "Include all components"
      source: kernel.Components
      output: view.components

    - rule: "Include all interfaces"
      source: kernel.Interfaces
      output: view.interfaces

    - rule: "Include component-interface relationships"
      source: kernel.Components, kernel.Interfaces
      transform: "map components to interfaces they expose"
      output: view.component_interfaces

    - rule: "Include data entities"
      source: kernel.Data
      output: view.data_entities

    - rule: "Include data ownership"
      source: kernel.Components, kernel.Data
      transform: "map components to data they own"
      output: view.data_ownership

    - rule: "Include workflows"
      source: kernel.Workflows
      output: view.workflows

    - rule: "Include workflow-component relationships"
      source: kernel.Workflows, kernel.Components
      transform: "map workflows to components they invoke"
      output: view.workflow_components

    - rule: "Include capability-component relationships"
      source: kernel.Capabilities, kernel.Components
      transform: "map capabilities to components that implement them"
      output: view.capability_components

Architecture view example:

architecture_view:
  view_name: Architecture View
  view_type: structural
  generated_from: kernel.payment_system_v1

  components:
    - id: component.payment_service
      type: Service
      responsibilities:
        - "Handle payment API requests"
        - "Manage payment data"
        - "Coordinate payment processing"
      interfaces:
        - exposes: interface.payment_api
          type: API
      data:
        - owns: data.payments
        - owns: data.transactions
        - owns: data.refunds
      workflows:
        - implements: workflow.payment_processing
      confidence: 0.95

    - id: component.payment_worker
      type: Worker
      responsibilities:
        - "Process payments asynchronously"
        - "Handle gateway interactions"
        - "Record transaction results"
      interfaces:
        - exposes: interface.payment_events
          type: Event
      data:
        - reads: data.payments
        - writes: data.transactions
      workflows:
        - participates_in: workflow.payment_processing
      confidence: 0.92

    - id: component.notification_service
      type: Service
      responsibilities:
        - "Send payment notifications"
        - "Manage notification templates"
      interfaces:
        - exposes: interface.notification_api
          type: API
        - consumes: interface.payment_events
          type: Event
      data:
        - owns: data.notifications
      workflows:
        - participates_in: workflow.payment_processing
      confidence: 0.88

    - id: component.orchestration_engine
      type: ControlPlane
      responsibilities:
        - "Coordinate workflow execution"
        - "Manage task dependencies"
        - "Track workflow state"
      interfaces:
        - exposes: interface.workflow_management_api
          type: API
      workflows:
        - controls: workflow.payment_processing
        - controls: workflow.daily_settlement
      confidence: 0.90

  interfaces:
    - id: interface.payment_api
      type: API
      protocol: HTTP
      exposed_by: component.payment_service
      operations:
        - POST /payments (create_payment)
        - GET /payments/{id} (get_payment)
        - POST /payments/{id}/refund (refund_payment)
      confidence: 0.98

    - id: interface.payment_events
      type: Event
      protocol: Kafka
      exposed_by: component.payment_worker
      consumed_by: component.notification_service
      events:
        - payment.created
        - payment.completed
        - payment.failed
      confidence: 0.92

  data_entities:
    - id: data.payments
      type: Table
      storage: PostgreSQL
      owned_by: component.payment_service
      read_by: component.payment_worker
      schema:
        - id (primary_key)
        - amount (decimal)
        - currency (string)
        - status (string)
        - created_at (timestamp)
      confidence: 0.98

    - id: data.transactions
      type: Table
      storage: PostgreSQL
      owned_by: component.payment_service
      schema:
        - id (primary_key)
        - payment_id (foreign_key)
        - gateway (string)
        - amount (decimal)
        - status (string)
      confidence: 0.98

  workflows:
    - id: workflow.payment_processing
      type: DAG
      controlled_by: component.orchestration_engine
      implemented_by: component.payment_service
      participates:
        - component.payment_worker
        - component.notification_service
      tasks:
        - validate_payment (component.payment_service)
        - charge_gateway (component.payment_worker)
        - record_transaction (component.payment_service)
        - send_notification (component.notification_service)
      confidence: 0.98

  component_graph:
    nodes: [payment_service, payment_worker, notification_service, orchestration_engine]
    edges:
      - payment_service → payment_api (exposes)
      - payment_service → payment_worker (coordinates)
      - payment_worker → payment_events (exposes)
      - payment_service → payments (owns)
      - payment_worker → transactions (writes)
      - notification_service → payment_events (consumes)
      - orchestration_engine → payment_processing (controls)

  architectural_patterns:
    - pattern: Service-Worker separation
      description: "API service and background worker separation"
      components: [payment_service, payment_worker]

    - pattern: Event-driven communication
      description: "Async communication through events"
      interfaces: [payment_events]

    - pattern: Centralized orchestration
      description: "Workflow engine coordinates execution"
      component: orchestration_engine

Architecture view significance:

architecture_view_significance:
  primary_artifact:
    reason: "Most accurate representation of system structure"
    confidence: "High - directly maps to kernel"
    completeness: "Comprehensive - includes all structural elements"

  use_cases:
    - system_understanding: "Understand system architecture at a glance"
    - design_review: "Evaluate architectural decisions"
    - impact_analysis: "Assess impact of changes"
    - comparison: "Compare architectures across systems"
    - documentation: "Document system structure"

  advantages:
    - completeness: "All structural elements represented"
    - consistency: "Consistent with kernel structure"
    - accuracy: "Based on extracted evidence"
    - traceability: "Elements reference evidence"

18.4 Operations View

The operations view highlights the runtime and operational structure of the system. While the architecture view focuses on structural relationships, the operations view emphasizes how the system is deployed, executed, and monitored in practice.

Operations view characteristics:

operations_view:
  focus: "Runtime and operational structure"
  audience: [devops_engineers, sre, operations_teams]
  questions_answered:
    - "How is the system deployed?"
    - "What are the runtime dependencies?"
    - "How is the system monitored?"
    - "What are the deployment units?"
  includes:
    - "Deployment topology"
    - "Runtime dependencies"
    - "Monitoring and logging"
    - "Scaling configuration"
  excludes:
    - "Internal component structure"
    - "Business logic"
    - "Data model details"

Operations view projection rules:

operations_view_projection:
  source_slots:
    - Components (filtered by deployment signals)
    - Constraints (operational)
    - Evidence (deployment and observability)

  projection_rules:
    - rule: "Include deployment units"
      source: kernel.Components
      filter: "components with deployment configuration"
      transform: "extract deployment information"
      output: view.deployment_units

    - rule: "Include runtime dependencies"
      source: kernel.Components
      filter: "components with dependency signals"
      transform: "extract external service dependencies"
      output: view.runtime_dependencies

    - rule: "Include monitoring configuration"
      source: deployment and observability signals
      filter: "monitoring, logging, metrics configuration"
      transform: "extract monitoring setup"
      output: view.monitoring

    - rule: "Include operational constraints"
      source: kernel.Constraints
      filter: "performance, deployment, regulatory constraints"
      output: view.operational_constraints

    - rule: "Include infrastructure requirements"
      source: deployment signals
      filter: "infrastructure definitions, resource requirements"
      output: view.infrastructure_requirements

18.5 Security and Trust View

The security and trust view focuses on system boundaries and trust relationships. Security concerns often appear indirectly in repositories through authentication mechanisms, authorization rules, encryption libraries, or access control configuration.

Security view characteristics:

security_view:
  focus: "System boundaries and trust relationships"
  audience: [security_engineers, architects, compliance_officers]
  questions_answered:
    - "Where are the trust boundaries?"
    - "How is authentication handled?"
    - "What are the authorization rules?"
    - "What data is sensitive?"
  includes:
    - "Trust boundaries between components"
    - "Authentication and authorization mechanisms"
    - "Sensitive data access paths"
    - "Security-relevant interfaces"
  excludes:
    - "Non-security functionality"
    - "Business logic details"

Security view projection rules:

security_view_projection:
  source_slots:
    - Components (security-relevant)
    - Interfaces (security-relevant)
    - Constraints (security)
    - Evidence (security hooks)

  projection_rules:
    - rule: "Include trust boundaries"
      source: kernel.Components
      filter: "components with security boundaries"
      transform: "identify component isolation"
      output: view.trust_boundaries

    - rule: "Include authentication mechanisms"
      source: security_evidence
      filter: "authentication middleware, JWT tokens"
      transform: "extract authentication setup"
      output: view.authentication

    - rule: "Include authorization rules"
      source: security_evidence
      filter: "authorization policies, role-based access"
      transform: "extract authorization rules"
      output: view.authorization

    - rule: "Include sensitive data access"
      source: kernel.Data
      filter: "data with security classification"
      transform: "map data to access controls"
      output: view.sensitive_data_access

    - rule: "Include security constraints"
      source: kernel.Constraints
      filter: "security constraints"
      output: view.security_constraints

18.6 Projection as Deterministic Transformation

Blueprint views are not extracted independently. Instead, they are generated from the kernel through deterministic projection rules. This approach ensures that all views remain consistent with the underlying architectural model.

Deterministic projection principles:

deterministic_projection:
  principle_1_single_source:
    name: "Single source of truth"
    rule: "Views derived only from kernel"
    benefit: "Consistency across views"

  principle_2_no_inference:
    name: "No new inference in views"
    rule: "Views only reorganize existing kernel information"
    benefit: "No additional uncertainty introduced"

  principle_3_reproducible:
    name: "Reproducible generation"
    rule: "Same kernel produces same views"
    benefit: "Predictable and testable"

  principle_4_validatable:
    name: "Validatable against kernel"
    rule: "Views can be validated against kernel"
    benefit: "Quality assurance possible"

18.7 Chapter Summary

  • Views are projections of the canonical kernel that emphasize different architectural perspectives
  • Kernel-first, views-later approach ensures architectural integrity and enables flexible view generation
  • Product view focuses on user-visible functionality and interfaces accessible to external users
  • Product view is derived from Goal, Capabilities, and external Interfaces
  • Product view faces challenges: implicit product intent, capability-to-feature mapping, user interface abstraction
  • Architecture view is the primary artifact, focusing on structural composition and relationships
  • Architecture view includes components, interfaces, data, workflows, and their relationships
  • Architecture view is most accurate and comprehensive representation of system structure
  • Operations view highlights runtime and operational structure including deployment, dependencies, monitoring
  • Operations view is derived from deployment configuration and observability signals
  • Security and trust view focuses on system boundaries, authentication, authorization, and sensitive data
  • Security view identifies trust boundaries, access controls, and compliance requirements
  • Views are generated through deterministic projection rules from the kernel
  • Projection ensures consistency, reproducibility, and validity across all views
  • Views can be validated against the kernel for quality assurance
  • Different views serve different audiences and answer different questions about the system
