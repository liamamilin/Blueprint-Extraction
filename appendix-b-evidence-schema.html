28  Appendix B — Evidence Object Schema v0.1

The Evidence Object Schema defines the canonical structure used to represent traceable architectural evidence during blueprint extraction.

Evidence objects are the foundation of the extraction discipline.
Every architectural element included in a blueprint must be supported by one or more evidence objects that reference concrete repository artifacts.

The purpose of the evidence schema is to ensure that architectural interpretations remain:

  • traceable

  • auditable

  • reviewable

  • reproducible

Evidence objects form the bridge between raw repository artifacts and structured architectural entities.

28.1 Evidence Object Overview

An evidence object represents a specific fragment of a repository artifact that supports a hypothesis about system structure.

Evidence objects are attached to architectural entities such as:

  • components

  • interfaces

  • workflows

  • data entities

  • capabilities

  • constraints

Multiple evidence objects may support a single architectural element.
Evidence objects may also contradict one another, in which case both are preserved for review.

28.2 Core Evidence Schema

The minimal evidence object representation is defined as follows:

evidence:
  id: <evidence-id>
  source:
    repo: <repository-identifier>
    file: <file-path>
    revision: <commit-or-tag>

  location:
    type: <line-range | json-path | object-path>
    value: <location-reference>

  excerpt: |
    <artifact-fragment>

  artifact_type: <code | config | schema | deploy | doc | test>

  signal_category: <structure | dependency | entrypoint | config | deploy | docs | test | observability | security>

  strength: <strong | moderate | weak>

  created_by:
    extractor: <tool-or-pipeline-id>
    version: <extractor-version>
    timestamp: <iso8601>

This schema ensures that evidence references remain stable even as repositories evolve.

28.3 Field Semantics

28.3.1 id

A unique identifier for the evidence object.

Evidence identifiers should remain stable within a blueprint pack to support traceability and cross-referencing.

28.3.2 source

The source field identifies the repository artifact from which the evidence originates.

It includes:

  • repository identifier

  • file path

  • revision reference

This information ensures that reviewers can locate the exact artifact used during extraction.

28.3.3 location

The location field specifies the exact position within the artifact.

Depending on the artifact type, this may include:

  • line ranges in source code

  • JSON paths in configuration files

  • object paths in structured documents

  • table or schema identifiers in migration files

Precise location references enable automated tools to retrieve the same evidence fragment during audits.

28.3.4 excerpt

The excerpt field contains the relevant fragment of the artifact.

Providing the excerpt improves review efficiency because reviewers can immediately inspect the relevant content without navigating the entire file.

Excerpts should remain minimal and focused on the architectural implication.

28.3.5 artifact_type

This field classifies the artifact that produced the evidence.

Typical artifact types include:

code
config
schema
deploy
doc
test

Artifact classification helps extraction systems evaluate the reliability of the evidence.

For example, schema definitions often provide stronger architectural signals than documentation.

28.3.6 signal_category

This field records the signal taxonomy category that produced the evidence.

Signal categories allow the extraction pipeline to trace architectural interpretations back to their originating signal class.

Examples include:

structure
dependency
entrypoint
config
deploy
docs
test
observability
security

28.3.7 strength

Evidence strength estimates how directly the artifact supports an architectural interpretation.

Strength levels are defined as:

strong

Artifacts that explicitly encode architectural structure.

Examples:

  • schema migrations

  • route definitions

  • workflow DAG specifications

moderate

Artifacts that strongly suggest architecture but do not explicitly define it.

Examples:

  • dependency declarations

  • scheduler configuration

  • deployment manifests

weak

Artifacts that provide indirect hints about system structure.

Examples:

  • directory names

  • naming conventions

  • documentation references

Strength classification contributes to confidence evaluation during extraction.

28.3.8 created_by

The created_by section records metadata about the extraction process that generated the evidence.

It typically includes:

  • the extraction tool identifier

  • extractor version

  • timestamp

This metadata ensures that evidence objects remain traceable to the pipeline that produced them.

28.4 Evidence Aggregation

Multiple evidence objects may support a single architectural entity.

For example, a worker component may be supported by:

  • a queue dependency declaration

  • a worker directory structure

  • a background task initialization script

  • a deployment configuration referencing worker processes

These evidence objects form an evidence set associated with the entity.

Evidence aggregation increases confidence and improves auditability.

28.5 Evidence Conflict Preservation

Extraction pipelines must preserve conflicting evidence.

Conflicts may arise from:

  • outdated documentation

  • legacy configuration files

  • partially removed components

  • inconsistent naming conventions

Rather than resolving conflicts automatically, the system records all evidence and defers resolution to later validation or review stages.

This preserves transparency and prevents silent misinterpretation.

28.6 Evidence Bundle

Evidence objects are packaged into an evidence bundle within the blueprint pack.

The bundle contains:

  • all evidence objects produced during extraction

  • indexes linking evidence to architectural entities

  • references to original repository artifacts

This bundle forms the audit trail for the blueprint.

Reviewers can trace every architectural element in the blueprint back to the evidence objects that justify it.

28.7 Schema Summary

The Evidence Object Schema v0.1 establishes a standardized representation for architectural evidence.

It ensures that:

  • every architectural claim remains traceable

  • extraction outputs remain auditable

  • evidence can be reviewed independently of the blueprint structure

By formalizing evidence representation, the schema transforms blueprint extraction from heuristic inference into a verifiable architecture reconstruction process.
