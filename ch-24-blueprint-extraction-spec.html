26  Blueprint Extraction Specification v0.1

This chapter serves as the formal specification for the Blueprint Extraction framework. It consolidates the methodology, data structures, and operational procedures defined throughout this volume into a single authoritative reference. The specification provides the foundation for implementing extraction pipelines, building extraction tools, and ensuring consistency across extraction operations.

The specification is organized into sections covering: terminology, data structures, extraction methodology, quality standards, and operational procedures. Each section provides precise definitions and constraints to guide implementation.

26.1 Terminology

Terms and definitions:

terminology:
  blueprint:
    definition: "Structured representation of system architecture derived from repository artifacts"
    scope: "Includes kernel, views, evidence, and metadata"
    synonym: "Architectural reconstruction"

  kernel:
    definition: "Minimal architectural skeleton with six slots: Goal, Capabilities, Components, Interfaces, Data, Workflows"
    scope: "Represents essential architectural structure"
    synonym: "Blueprint Kernel"

  view:
    definition: "Projection of kernel tailored to specific stakeholders or concerns"
    scope: "Includes Product, Architecture, Operations, Security views"
    synonym: "Architectural view"

  signal:
    definition: "Repository artifact that may indicate architectural structure"
    scope: "Includes files, configurations, dependencies"
    synonym: "Architectural indicator"

  evidence:
    definition: "Structured observation linking signal to architectural claim"
    scope: "Includes signal reference, claim, confidence"
    synonym: "Extraction evidence"

  confidence:
    definition: "Three-state semantic indicating evidence strength: observed, inferred, unknown"
    scope: "Applies to architectural claims"
    synonym: "Confidence state"

  ontology_mapping:
    definition: "Translation of implementation artifacts to canonical entities"
    scope: "Components, interfaces, data, workflows"
    synonym: "Entity mapping"

  blueprint_pack:
    definition: "Standard output unit containing blueprint, views, evidence, and report"
    scope: "Version-controlled, reviewable representation"
    synonym: "Extraction output"

  extraction_maturity_level:
    definition: "Classification of extraction reliability (EML 0-4)"
    scope: "No extraction → reviewed and validated"
    synonym: "EML"

26.2 Data Structures

26.2.1 Blueprint Kernel

Kernel schema:

blueprint_kernel:
  $schema: "https://spec.blueprintextraction.org/v0.1/kernel.schema.json"

  system:
    id: string (required)
    name: string (required)
    description: string (optional)
    repository_url: string (required)
    extraction_version: string (required)
    extraction_date: timestamp (required)

  kernel:
    Goal:
      value: string (required)
      confidence: number (0.0-1.0) (required)
      status: enum(observed, inferred, unknown) (required)
      evidence: array(evidence_reference) (optional)

    Capabilities:
      - id: string (required)
        name: string (required)
        description: string (optional)
        confidence: number (0.0-1.0) (required)
        status: enum(observed, inferred, unknown) (required)
        interfaces: array(interface_id) (optional)
        workflows: array(workflow_id) (optional)
        evidence: array(evidence_reference) (optional)

    Components:
      - id: string (required)
        name: string (required)
        type: enum(Service, Worker, Scheduler, Plugin, ControlPlane) (required)
        confidence: number (0.0-1.0) (required)
        status: enum(observed, inferred, unknown) (required)
        responsibilities: array(string) (optional)
        exposes: array(interface_id) (optional)
        owns: array(data_id) (optional)
        reads: array(data_id) (optional)
        participates_in: array(workflow_id) (optional)
        evidence: array(evidence_reference) (optional)

    Interfaces:
      - id: string (required)
        name: string (required)
        type: enum(API, Event, CLI, RPC) (required)
        protocol: string (optional)
        confidence: number (0.0-1.0) (required)
        status: enum(observed, inferred, unknown) (required)
        exposed_by: component_id (optional)
        exposes_capability: array(capability_id) (optional)
        contract: object (optional)
        evidence: array(evidence_reference) (optional)

    Data:
      - id: string (required)
        name: string (required)
        type: enum(Table, Collection, Document, Key-Value) (required)
        storage_type: enum(relational, document, key-value, object, analytical) (required)
        confidence: number (0.0-1.0) (required)
        status: enum(observed, inferred, unknown) (required)
        schema: object (optional)
        owned_by: component_id (optional)
        read_by: array(component_id) (optional)
        foreign_keys: array(object) (optional)
        evidence: array(evidence_reference) (optional)

    Workflows:
      - id: string (required)
        name: string (required)
        type: enum(DAG, Pipeline, Schedule, Event, StateMachine) (required)
        confidence: number (0.0-1.0) (required)
        status: enum(observed, inferred, unknown) (required)
        implemented_by: component_id (optional)
        participates: array(component_id) (optional)
        implements_capability: array(capability_id) (optional)
        triggers: array(object) (optional)
        tasks: array(object) (optional)
        evidence: array(evidence_reference) (optional)

    Constraint: (optional)
      non_functional_requirements: array(object) (optional)
      performance_requirements: array(object) (optional)
      security_requirements: array(object) (optional)

26.3 Extraction Methodology

26.3.1 Extraction Pipeline

Pipeline stages:

extraction_pipeline:
  stage_1_signal_detection:
    objective: "Identify repository signals"
    input: "Repository files"
    output: "Signal inventory"
    process:
      - "Scan repository structure"
      - "Identify signal types"
      - "Extract signal metadata"
    quality_requirement: "Signal detection completeness > 0.8"

  stage_2_hypothesis_formation:
    objective: "Form architectural hypotheses"
    input: "Signal inventory"
    output: "Hypothesis set"
    process:
      - "Analyze signal patterns"
      - "Form component hypotheses"
      - "Form interface hypotheses"
      - "Form data hypotheses"
      - "Form workflow hypotheses"
    quality_requirement: "Hypothesis coverage > 0.7"

  stage_3_evidence_collection:
    objective: "Collect evidence for hypotheses"
    input: "Hypothesis set, signals"
    output: "Evidence bundle"
    process:
      - "Map hypotheses to signals"
      - "Extract evidence objects"
      - "Calculate confidence"
      - "Preserve conflicts"
    quality_requirement: "Evidence ratio > 0.8"

  stage_4_ontology_mapping:
    objective: "Map to canonical entities"
    input: "Evidence bundle"
    output: "Mapped entities"
    process:
      - "Normalize names"
      - "Canonicalize types"
      - "Establish relationships"
    quality_requirement: "Mapping consistency > 0.85"

  stage_5_kernel_construction:
    objective: "Construct blueprint kernel"
    input: "Mapped entities"
    output: "Blueprint kernel"
    process:
      - "Fill kernel slots"
      - "Apply triangular consistency"
      - "Calculate slot confidence"
    quality_requirement: "Kernel completeness > 0.8"

  stage_6_view_generation:
    objective: "Generate stakeholder views"
    input: "Blueprint kernel"
    output: "Architectural views"
    process:
      - "Project product view"
      - "Project architecture view"
      - "Project operations view"
      - "Project security view"
    quality_requirement: "View consistency = 1.0"

  stage_7_validation:
    objective: "Validate blueprint structure"
    input: "Blueprint kernel, views"
    output: "Validation report"
    process:
      - "DSL validation"
      - "Reference integrity check"
      - "Kernel completeness check"
      - "Anomaly detection"
    quality_requirement: "All critical checks must pass"

  stage_8_review:
    objective: "Human review and approval"
    input: "Validated blueprint"
    output: "Approved blueprint"
    process:
      - "Slot-oriented review"
      - "Priority-based review"
      - "Decision logging (ADR)"
      - "Maturity assessment"
    quality_requirement: "EML >= 3 for production use"

Gate definitions:

quality_gates:
  gate_1_signal_detection:
    metric: "Signal detection completeness"

  gate_2_evidence_collection:

  gate_3_kernel_completeness:

    calculation: "populated_slots / required_slots"
    action: "Retry or fail if below threshold"

  gate_4_validation:
    metric: "Validation status"
    threshold: "PASSED"
    checks:
      - DSL validation: must PASS

  gate_5_review:
    metric: "EML level"

      - EML_0: No extraction
      - EML_1: Partial signals
      - EML_2: Partial kernel
      - EML_3: Evidence-backed draft
      - EML_4: Reviewed and validated
    action: "EML < 3 requires additional review"

26.3.3 Confidence Calculation

Confidence derivation:

confidence_calculation:
  observed:
    definition: "Claim directly supported by strong signal"
    confidence_range: "0.8 - 1.0"
    derivation:
      - "Signal strength: strong"
      - "Signal count: >= 3"
      - "Consistency: no conflicts"
    example: "API route definition → interface exists"

  inferred:
    definition: "Claim derived from multiple weak signals or patterns"
    confidence_range: "0.4 - 0.8"
    derivation:
      - "Signal strength: moderate or weak"
      - "Signal count: 1-2"
      - "Consistency: some conflicts"
      - "Pattern inference: high probability"
    example: "Directory structure + queue usage → worker component"

  unknown:
    definition: "Claim cannot be supported by evidence"
    confidence_range: "0.0 - 0.4"
    derivation:
      - "Signal strength: none"
      - "Signal count: 0"
      - "Pattern inference: low probability"
    example: "Missing documentation → unknown goal"

  confidence_formula:
    confidence = (
      (signal_strength * 0.4) +
      (signal_count_normalized * 0.3) +
      (consistency_score * 0.2) +
      (pattern_probability * 0.1)
    )

  where:
    signal_strength:
      strong: 1.0
      moderate: 0.6
      weak: 0.3

    signal_count_normalized:
      count >= 3: 1.0
      count = 2: 0.7
      count = 1: 0.4
      count = 0: 0.0

    consistency_score:
      no_conflicts: 1.0
      minor_conflicts: 0.7
      major_conflicts: 0.3

    pattern_probability:
      high: 1.0
      medium: 0.6
      low: 0.3

26.4 Quality Standards

26.4.1 Blueprint Quality Metrics

Metric definitions:

quality_metrics:
  coverage:
    definition: "Proportion of kernel slots populated"
    calculation: "populated_slots / required_slots"
    threshold: "> 0.8"
    weight: 0.25

  correctness:
    definition: "Proportion of correctly extracted elements"
    calculation: "correct_elements / audited_elements"
    threshold: "> 0.8"
    weight: 0.30
    method: "Sampling-based audit"

  consistency:
    definition: "Internal coherence of blueprint"
    calculation: "triangular_consistency_score"
    threshold: "> 0.85"
    weight: 0.20
    checks:
      - capability_interface_consistency
      - capability_workflow_consistency
      - component_interface_consistency

  evidence_ratio:
    definition: "Proportion of elements with traceable evidence"
    calculation: "elements_with_evidence / total_elements"
    threshold: "> 0.8"
    weight: 0.25

  overall_quality:
    calculation: "weighted_sum of metrics"
    threshold: "> 0.8"
    formula:
      quality_score = (
        (coverage * 0.25) +
        (correctness * 0.30) +
        (consistency * 0.20) +
        (evidence_ratio * 0.25)
      )

26.7 Chapter Summary

  • Blueprint Extraction Specification v0.1 provides formal definition for extraction framework
  • Terminology defines core concepts: blueprint, kernel, view, signal, evidence, confidence, ontology mapping, blueprint pack, EML
  • Data structures include Kernel schema (6 slots with specific fields), Evidence object schema, Blueprint Pack structure
  • Extraction methodology defines 8-stage pipeline: signal detection → hypothesis formation → evidence collection → ontology mapping → kernel construction → view generation → validation → review
  • Quality gates enforce thresholds: signal detection completeness >0.8, evidence ratio >0.8, kernel completeness >0.8, validation PASSED, EML >=3
  • Confidence calculation derives from signal strength (40%), signal count (30%), consistency (20%), pattern probability (10%)
  • Quality metrics include coverage (>0.8), correctness (>0.8), consistency (>0.85), evidence ratio (>0.8)
  • Extraction Maturity Levels (EML 0-4): No extraction → Partial signals → Partial kernel → Evidence-backed draft → Reviewed and validated
  • Dataset quality targets define production, high-quality, and acceptable dataset standards
  • Operational procedures include repository filtering (quality, type, size, language, license, dependency), two-pass extraction (cheap pass + deep pass), incremental extraction (change detection → impact analysis → selective extraction)
  • Normalization and deduplication include family identification, normalization, duplicate detection, canonical generation
  • Specification uses semantic versioning (v0.1 initial), with roadmap for v0.2 and v1.0
  • This specification serves as authoritative reference for implementing extraction pipelines and tools
