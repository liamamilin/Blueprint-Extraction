15  Data Model and Ownership Extraction

Data is one of the most stable structural elements of a system. While components and interfaces may change frequently, data models often persist across system evolution. For this reason, blueprint extraction treats the data layer as a critical architectural dimension.

The goal of this stage is to reconstruct data entities and their ownership relationships from repository artifacts. This involves identifying persistent data structures, determining which components control them, and detecting potential consistency risks within the system.

The output consists of data entities and an ownership map describing how components interact with persistent data. Together, these artifacts represent the data layer of the blueprint.

15.1 Why Data Matters in Architecture

Data is the foundation upon which system capabilities are built. Understanding data structure and ownership is essential for architectural understanding because data:

1. Represents system state:

Data entities capture the persistent state that the system maintains over time. Understanding what data exists and how it’s structured reveals what the system is designed to track and manage.

data_insights:
  payment_data:
    structure: transactions, refunds, settlements
    meaning: system tracks financial transactions
    implications: requires ACID properties, audit trails

  user_data:
    structure: profiles, preferences, permissions
    meaning: system manages user identities
    implications: requires privacy controls, authentication

2. Defines component boundaries:

Data ownership often defines component boundaries. Components that own distinct data entities typically have distinct responsibilities and can be deployed independently.

boundary_insights:
  payment_service:
    owns: [payments, transactions]
    boundary: "Financial transaction management"

  user_service:
    owns: [users, profiles, permissions]
    boundary: "User identity and access management"

  notification_service:
    owns: [notifications, templates]
    boundary: "Notification delivery"

3. Indicates architectural constraints:

Data access patterns reveal architectural constraints such as consistency requirements, performance characteristics, and scalability considerations.

constraint_insights:
  shared_data_access:
    pattern: multiple services access same data
    constraint: requires coordination mechanisms
    implications: eventual consistency, distributed transactions

  data_replication:
    pattern: data replicated across regions
    constraint: requires synchronization
    implications: conflict resolution, consistency trade-offs

4. Exposes integration points:

Data entities often represent integration points between components or systems. Understanding these relationships reveals how components coordinate.

integration_insights:
  payment_processing:
    data_flow: payment_service → payment_worker → database
    integration: queue-based data passing
    coordination: asynchronous processing

  user_authentication:
    data_flow: api_service → user_service → database
    integration: service-to-service calls
    coordination: synchronous queries

15.2 Data Entity Extraction

Data entity extraction identifies persistent data structures from repository artifacts. These entities represent tables, collections, documents, or other storage structures that hold system state.

Data entity signals:

15.5 Ownership Inference

Once data entities are identified, extraction must determine which components own or control those entities. Ownership refers to the component responsible for maintaining the authoritative state of a data entity.

Ownership signal types:

1. Data access layers associated with specific services:

# payment_service/repositories/payment_repository.py
class PaymentRepository:
    """Payment data access layer"""

    def create(self, payment_data):
        payment = Payment(**payment_data)
        db.session.add(payment)
        db.session.commit()
        return payment

    def get(self, payment_id):
        return Payment.query.get(payment_id)

    def update(self, payment_id, updates):
        payment = Payment.query.get(payment_id)
        for key, value in updates.items():
            setattr(payment, key, value)
        db.session.commit()
        return payment

    def delete(self, payment_id):
        payment = Payment.query.get(payment_id)
        db.session.delete(payment)
        db.session.commit()

15.6 Detecting Consistency Risks

Repository artifacts can also reveal potential data consistency risks. One common risk is the presence of multiple writers to the same data entity. When multiple components write to the same storage resource, coordination mechanisms are required to maintain consistency.

15.8 Chapter Summary

  • Data is a stable structural element that often persists across system evolution
  • Data extraction identifies persistent data structures and ownership relationships
  • Data entities are extracted from ORM models, migration files, DDL statements, and schema definitions
  • Storage types include relational databases, document stores, key-value stores, object storage, and analytical stores
  • Each storage type has different operational roles and typical use cases
  • Data lifecycle signals reveal how data evolves through creation, transformation, archival, and deletion
  • Ownership inference identifies components that control data entities through data access layers, migration locations, configuration, and write operations
  • Consistency risks are detected through multiple writers, duplicated access layers, replication scripts, and cross-database transactions
  • The output consists of data entities (with schema, relationships, lifecycle) and an ownership map (with component access and consistency risks)
  • Data entities and ownership relationships form the data layer of the blueprint
  • Understanding data ownership reveals component boundaries, architectural constraints, and integration points
