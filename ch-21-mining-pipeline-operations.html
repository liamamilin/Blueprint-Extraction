23  Mining Pipeline Operations

The preceding chapters describe how to extract a blueprint from a single repository. In practice, blueprint extraction often operates at scale—across hundreds or thousands of repositories. Scaling extraction requires operational infrastructure: repository discovery and filtering, cost control strategies, incremental extraction, and workflow orchestration.

This chapter defines the mining pipeline operations framework for extracting blueprints at scale. The framework balances thoroughness with efficiency, ensuring that extraction can be applied to large repository collections without excessive cost or time.

23.1 Repository Discovery and Filtering

Blueprint extraction operates on repositories, but not all repositories are suitable targets. The first operational step is therefore repository discovery and filtering to identify which repositories should be processed.

Repository discovery sources:

repository_discovery:
  github_search:

23.2 Cheap Pass and Deep Pass

Extraction cost varies dramatically based on repository characteristics. A small, well-structured API service may be extracted in seconds, while a complex monorepo may require hours. To control costs, the mining pipeline uses a two-pass strategy: cheap pass (lightweight analysis) followed by deep pass (full extraction).

Cheap pass definition:

cheap_pass:
  objective: "Lightweight, fast analysis to estimate extraction value"
  duration: "Seconds to minutes"
  resource_usage: "Low"

  analysis_scope:
    signal_detection:
      - directory_structure_analysis
      - entrypoint_identification
      - dependency_inference_from_build_files
      - basic_config_parsing

    quality_assessment:
      - architecture_complexity_estimate
      - extraction_feasibility_score
      - expected_coverage_estimate

    resource_estimation:
      - expected_extraction_time
      - expected_compute_resources
      - expected_storage_requirements

  cheap_pass_outputs:
    repository_profile:
      repo_id: "owner/repo"
      language: "python"
      size_estimate: "medium"
      complexity_estimate: "moderate"

    extraction_feasibility:
      feasibility_score: 0.8
      factors:
        - has_clear_structure: true
        - has_build_files: true
        - has_entrypoints: true
        - has_observability: false

    resource_estimate:
      expected_time: "5-10 minutes"
      expected_cost: "low"
      confidence: 0.7

  pass_or_fail:
    pass:
      condition: "feasibility_score >= 0.6"
      action: "Proceed to deep pass"
      criteria:
        - "Sufficient architectural structure"
        - "Reasonable extraction cost"
        - "Expected good coverage"

    fail:
      condition: "feasibility_score < 0.6"
      action: "Skip repository"
      criteria:
        - "Poor architectural structure"
        - "High extraction cost"
        - "Expected poor coverage"
        - "Template or test repository"

  cheap_pass_examples:
    high_feasibility:
      repo: "payment-service"
      structure:
        - clear separation: services/workers
        - entrypoints: API routes, CLI
        - dependencies: requirements.txt
      assessment:
        feasibility_score: 0.9
        action: "PASS - proceed to deep pass"

    medium_feasibility:
      repo: "data-pipeline"
      structure:
        - some separation: unclear
        - entrypoints: only one
        - dependencies: requirements.txt
      assessment:
        feasibility_score: 0.7
        action: "PASS - proceed to deep pass with caution"

    low_feasibility:
      repo: "example-project"
      structure:
        - flat structure
        - no clear entrypoints
        - no build files
      assessment:
        feasibility_score: 0.3
        action: "FAIL - skip repository"

23.3 Incremental Extraction

Repositories evolve over time through commits, merges, and refactoring. To keep blueprints current without repeatedly running full extraction, the mining pipeline supports incremental extraction—updating existing blueprints based on repository changes.

Incremental extraction strategy:

incremental_extraction:
  principle: "Update only what has changed"
  benefit: "Reduce extraction cost and time"
  challenge: "Determine which architectural elements are affected by code changes"

  change_detection:
    commit_analysis:
      action: "Analyze repository changes since last extraction"
      metadata:
        - commit_hash
        - commit_date
        - files_changed
        - lines_added
        - lines_deleted

    change_categorization:
      structural_changes:
        patterns:
          - "new service directory created"
          - "service directory deleted"
          - "component merged or split"
        impact: "Component boundaries may change"
        action: "Re-run component discovery"

      interface_changes:
        patterns:
          - "API route added or modified"
          - "event topic added or modified"
          - "CLI command added or modified"
        impact: "Interfaces may change"
        action: "Re-run interface extraction"

      data_changes:
        patterns:
          - "schema migration added"
          - "ORM model modified"
          - "data directory restructured"
        impact: "Data model may change"
        action: "Re-run data model extraction"

      workflow_changes:
        patterns:
          - "DAG definition modified"
          - "pipeline configuration changed"
          - "scheduler configuration updated"
        impact: "Workflows may change"
        action: "Re-run workflow extraction"

      minor_changes:
        patterns:
          - "business logic modified"
          - "test files added"
          - "documentation updated"
        impact: "Minimal architectural impact"
        action: "No re-extraction needed"

  change_impact_analysis:
    example_1:
      commit: "Add refund service"
      files_changed:
        - "services/refund_service/ (new directory)"
        - "services/refund_service/api.py (new file)"
        - "requirements.txt (modified)"
      impact_analysis:
        - structural_change: "New component added"
        - interface_change: "New API interface"
        - workflow_change: "May affect payment workflow"
      extraction_updates:
        - re_run: "component_discovery"
        - re_run: "interface_extraction"
        - re_run: "workflow_extraction"
        - re_run: "kernel_construction"

    example_2:
      commit: "Fix payment calculation bug"
      files_changed:
        - "services/payment_service/calc.py (modified)"
        - "tests/test_calc.py (modified)"
      impact_analysis:
        - structural_change: "None"
        - interface_change: "None"
        - data_change: "None"
        - workflow_change: "None"
      extraction_updates:
        - re_run: "none"
        - action: "Skip extraction, no architectural change"

    example_3:
      commit: "Refactor data access layer"
      files_changed:
        - "services/payment_service/db.py (modified)"
        - "migrations/002_refactor_payments.sql (new file)"
        - "models/payment.py (deleted)"
        - "models/refactored_payment.py (new file)"
      impact_analysis:
        - structural_change: "None"
        - interface_change: "None"
        - data_change: "Schema and model changed"
        - workflow_change: "None"
      extraction_updates:
        - re_run: "data_model_extraction"
        - re_run: "kernel_construction"

Incremental extraction workflow:

incremental_workflow:
  step_1_version_tracking:
    action: "Track extraction version"
    storage: "Blueprint pack metadata"
    metadata:
      - last_extraction_commit
      - extraction_date
      - extraction_version

  step_2_change_detection:
    action: "Detect repository changes"
    input: "Last extraction commit"
    output: "Changed files and patterns"
    comparison:
      - "git diff last_extraction_commit HEAD"
      - "Pattern matching on changed files"

  step_3_impact_analysis:
    action: "Analyze change impact"
    input: "Changed files and patterns"
    output: "Affected architectural elements"
    analysis:
      - "Map changes to architectural layers"
      - "Determine re-extraction scope"

  step_4_selective_extraction:
    action: "Re-run affected extraction stages"
    input: "Affected architectural elements"
    output: "Updated blueprint elements"
    strategy:
      - "Only re-run affected stages"
      - "Reuse unchanged elements"

  step_5_blueprint_update:
    action: "Update blueprint pack"
    input: "Updated blueprint elements"
    output: "Incrementally updated blueprint"
    update:
      - "Replace changed elements"
      - "Preserve unchanged elements"
      - "Update metadata"

  step_6_validation:
    action: "Validate updated blueprint"
    input: "Updated blueprint"
    output: "Validation results"
    checks:
      - "DSL validation"
      - "Reference integrity"
      - "Kernel completeness"

  efficiency_gains:
    time_reduction: "70-90% faster than full extraction"
    cost_reduction: "80-95% less compute"
    accuracy: "Maintains blueprint currency"

Incremental extraction challenges:

incremental_challenges:
  cross_slot_dependencies:
    problem: "Changes to one slot may affect other slots"
    example: "Component change may affect interfaces and workflows"
    solution:
      - "Analyze dependency graph"
      - "Cascade re-extraction to dependent slots"
      - "Run triangular consistency checks"

  boundary_inference:
    problem: "Structural changes may invalidate boundary inferences"
    example: "Component merge may affect multiple components"
    solution:
      - "Re-run boundary inference on structural changes"
      - "Verify ownership relationships"
      - "Check for orphaned entities"

  confidence_recalculation:
    problem: "Changes may affect confidence calculations"
    example: "New evidence may increase confidence"
    solution:
      - "Recalculate confidence for affected elements"
      - "Propagate confidence changes through dependencies"
      - "Update EML classification"

  version_conflicts:
    problem: "Concurrent changes may cause conflicts"
    example: "Multiple commits affecting same component"
    solution:
      - "Use latest repository state"
      - "Merge conflicting changes"
      - "Re-run full extraction if conflicts severe"

23.4 Pipeline Orchestration

Operating extraction at scale requires orchestration infrastructure to manage queue processing, resource allocation, failure handling, and output management. The pipeline must be robust, scalable, and observable.

Pipeline architecture:

pipeline_architecture:
  components:

      purpose: "Manage repository processing queue"

      features:
        - "Priority-based processing"
        - "Retry on failure"
        - "Dead-letter queue for failed repos"

    coordinator:
      purpose: "Orchestrate extraction workflow"

        - "Schedule cheap passes"

23.5 Chapter Summary

  • Repository discovery identifies potential extraction targets from GitHub search, organization inventories, code hosting platforms, and package registries
  • Repository filtering excludes unsuitable targets based on quality (activity, adoption), type (templates, examples), size, language, and license criteria
  • Filtering pipeline processes repositories through multiple stages with 5-20% retention rate from discovery to final list
