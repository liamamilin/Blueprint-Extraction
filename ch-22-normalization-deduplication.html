24  Normalization and Deduplication

As the mining pipeline extracts blueprints from thousands of repositories, it inevitably encounters similar architectures. Different organizations may implement payment services, user authentication, or data pipelines in similar ways. These similarities provide valuable insights into architectural patterns, but they also create redundancy in blueprint datasets.

This chapter defines the normalization and deduplication framework for managing blueprint similarity. The framework identifies blueprint families, normalizes variations, detects duplicates, and creates canonical representations that support pattern discovery.

24.1 Blueprint Families

Architectural similarity emerges from common problems, constraints, and conventions. A blueprint family represents a group of blueprints that share fundamental architectural structure while differing in specific implementation details.

Family characteristics:

blueprint_families:
  definition: "Group of blueprints with shared fundamental architectural structure"
  criteria:
    structural_similarity:
      - similar_kernel_slot_population
      - similar_component_types
      - similar_workflow_structure
      - similar_interface_types

    functional_similarity:
      - similar_capabilities
      - similar data models
      - similar integration patterns

    domain_similarity:
      - solve similar problems
      - operate in similar domains
      - address similar requirements

  family_identification:
    kernel_structure:
      components:
        - similar_component_count
        - similar_component_types
        - similar_component_boundaries

      interfaces:
        - similar_interface_count
        - similar_interface_types
        - similar_interface_patterns

      data:
        - similar_data_model_structure
        - similar_storage_types
        - similar_data_relationships

      workflows:
        - similar_workflow_types
        - similar_workflow_structure
        - similar_trigger_types

    capability_overlap:
      metric: "Jaccard similarity of capabilities"
      calculation: "intersection / union"
      threshold: "> 0.6"

    structural_similarity:
      metric: "Graph similarity of component relationships"
      calculation: "Graph edit distance"
      threshold: "Similar within 20% edit distance"

24.2 Variant Normalization

Blueprints within a family exhibit variation due to technology choices, team conventions, and specific requirements. Variant normalization reduces these variations to canonical forms that highlight the shared architectural structure while preserving meaningful differences.

Normalization strategies:

normalization_strategies:
  technology_abstraction:
    objective: "Abstract specific technologies to generic categories"
    transformations:
      web_frameworks:
        - FastAPI → web_framework
        - Flask → web_framework
        - Express.js → web_framework
        - Spring Boot → web_framework

      databases:
        - PostgreSQL → relational_database
        - MySQL → relational_database
        - MongoDB → document_database
        - Redis → key_value_store

      message_queues:
        - RabbitMQ → message_queue
        - Kafka → message_queue
        - SQS → message_queue

    rationale: "Technology choices are implementation details, not architectural differences"

  naming_normalization:
    objective: "Normalize entity names to canonical forms"
    transformations:
      component_names:
        - payment_service → api_service
        - order_service → api_service
        - user_service → api_service

      interface_names:
        - payment_api → api_interface
        - order_api → api_interface
        - user_api → api_interface

      data_names:
        - payments → domain_data
        - orders → domain_data
        - users → domain_data

    rationale: "Names are domain-specific, not architectural"

  capability_normalization:
    objective: "Map capabilities to canonical operations"
    transformations:
      crud_operations:
        - create_payment → create_entity
        - create_order → create_entity
        - create_user → create_entity
        - read_payment → read_entity
        - read_order → read_entity
        - read_user → read_entity

      async_operations:
        - process_payment → process_entity_async
        - process_order → process_entity_async
        - process_user → process_entity_async

    rationale: "Specific capabilities are instances of general patterns"

  structural_normalization:
    objective: "Normalize structural variations"
    transformations:
      optional_components:
        - "If optional component present, preserve as optional"
        - "If optional component absent, represent as optional with null"

      interface_variations:
        - "Normalize interface protocols to generic types"
        - "Preserve interface semantics"

      data_variations:
        - "Normalize schema variations to generic structure"
        - "Preserve data relationships"

    rationale: "Optional variations should be represented consistently"

Normalization example:

normalization_example:
  input_blueprint:
    blueprint_id: "payment-service-v1"
    kernel:
      Goal: "Process payment transactions"
      Capabilities:
        - create_payment
        - process_payment
        - refund_payment
      Components:
        - payment_service (Service, FastAPI)
        - payment_worker (Worker)
      Interfaces:
        - payment_api (API, HTTP/REST)
        - payment_events (Event, RabbitMQ)
      Data:
        - payments (Table, PostgreSQL)
        - transactions (Table, PostgreSQL)
      Workflows:
        - payment_processing (Pipeline)

  normalized_blueprint:
    blueprint_id: "rest-api-service-template"
    kernel:
      Goal: "Provide CRUD and async operations"
      Capabilities:
        - create_entity
        - read_entity
        - update_entity
        - delete_entity
        - process_entity_async
      Components:
        - api_service (Service, web_framework)
        - worker_service (Worker, optional: true)
      Interfaces:
        - api_interface (API, generic_protocol)
        - event_interface (Event, message_queue, optional: true)
      Data:
        - domain_data (Table, relational_database)
        - transaction_log (Table, relational_database)
      Workflows:
        - processing_pipeline (Pipeline)

  transformations_applied:
    - "Technology abstraction: FastAPI → web_framework, PostgreSQL → relational_database, RabbitMQ → message_queue"
    - "Naming normalization: payment_* → generic names"
    - "Capability normalization: specific CRUD → generic CRUD"
    - "Structural normalization: worker marked as optional"

  family_representation:
    family: "REST API Services"
    canonical_template: "normalized_blueprint"
    variation_count: 3
    variation_types:
      - "Worker present/absent"
      - "Caching layer present/absent"
      - "Async processing present/absent"

Normalization challenges:

normalization_challenges:
  over_normalization:
    problem: "Losing meaningful differences"
    example:
      - "Normalizing microservices to single service"
      - "Losing architectural complexity"
    mitigation:
      - "Preserve structural differences"
      - "Mark optional variations explicitly"
      - "Maintain family sub-clusters"

  context_loss:
    problem: "Losing domain-specific information"
    example:
      - "Payment service vs. Order service"
      - "Different business semantics"
    mitigation:
      - "Retain original blueprint"
      - "Keep family as abstraction layer"
      - "Link normalized to original"

  semantic_drift:
    problem: "Normalization may introduce semantic errors"
    example:
      - "CRUD operations don't capture domain logic"
      - "Generic names lose business meaning"
    mitigation:
      - "Careful capability mapping"
      - "Review normalization decisions"
      - "Human validation of templates"

24.3 Duplicate Detection

Similar repositories or repeated extractions can produce duplicate blueprints within the dataset. Duplicate detection identifies these duplicates to maintain dataset quality and efficiency.

Duplicate sources:

duplicate_sources:
  repository_duplicates:
    cause: "Same repository extracted multiple times"
    scenarios:
      - "Extraction repeated after code changes"
      - "Different branches of same repository"
      - "Repository cloned under different names"
    detection:
      - "Compare repository URLs"
      - "Compare commit hashes"
      - "Compare file structures"

  architectural_duplicates:
    cause: "Different repositories with identical architecture"
    scenarios:
      - "Tutorial examples copied"
      - "Boilerplate projects cloned"
      - "Standard templates used"
    detection:
      - "Kernel structure comparison"
      - "Capability comparison"
      - "Component comparison"

  variant_duplicates:
    cause: "Minor variations within same family"
    scenarios:
      - "Configuration differences only"
      - "Technology stack changes only"
      - "Naming differences only"
    detection:
      - "Normalized form comparison"
      - "Semantic similarity analysis"
      - "Edit distance calculation"

Duplicate detection metrics:

duplicate_detection_metrics:
  exact_duplicates:
    criterion: "100% match on kernel structure"
    comparison:
      - component_identical: true
      - interface_identical: true
      - data_identical: true
      - workflow_identical: true
      - capabilities_identical: true
    action: "Remove duplicates, keep one instance"

  near_duplicates:
    criterion: "> 95% similarity on normalized forms"
    comparison:
      - kernel_similarity: > 0.95
      - capability_similarity: > 0.95
      - component_similarity: > 0.95
    action: "Merge or flag for review"

  architectural_clones:
    criterion: "> 90% similarity with same normalized template"
    comparison:
      - normalized_form_identical: true
      - capability_semantic_similarity: > 0.90
    action: "Keep as variants, link to template"

  detection_process:
    step_1_exact_match:
      - "Compare repository URLs and commit hashes"
      - "Identify exact repository duplicates"
      - "Remove duplicate extractions"

    step_2_kernel_comparison:
      - "Compare kernel structures"
      - "Identify near duplicates"
      - "Calculate similarity scores"

    step_3_normalized_comparison:
      - "Compare normalized forms"
      - "Identify architectural clones"
      - "Link to family templates"

    step_4_review:
      - "Review ambiguous cases"
      - "Human validation of duplicates"
      - "Finalize duplicate removal"

Duplicate detection example:

duplicate_detection_example:
  blueprints:
    - blueprint_a:
        id: "payment-service-main"
        repo: "https://github.com/org/payment-service"
        commit: "abc123"
        kernel:
          components: [payment_service, payment_worker]
          interfaces: [payment_api, payment_events]

    - blueprint_b:
        id: "payment-service-feature-branch"
        repo: "https://github.com/org/payment-service"
        commit: "def456"
        kernel:
          components: [payment_service, payment_worker, refund_worker]
          interfaces: [payment_api, payment_events, refund_api]

    - blueprint_c:
        id: "payment-service-tutorial"
        repo: "https://github.com/other/payment-tutorial"
        commit: "ghi789"
        kernel:
          components: [payment_service, payment_worker]
          interfaces: [payment_api, payment_events]

  detection_results:
    exact_duplicates: []
    rationale: "Different commit hashes and architectures"

    near_duplicates: []
    rationale: "Component differences (b has refund_worker)"

    architectural_clones:
      - pair: [blueprint_a, blueprint_c]
        similarity: 0.95
        rationale: "Identical kernel structure, different repositories"
        action: "Link as variants"

    repository_duplicates: []
    rationale: "Different repository URLs"

  actions:
    - "Keep blueprint_a as canonical"
    - "Keep blueprint_b as separate (architectural change)"
    - "Link blueprint_c as clone of blueprint_a"
    - "Mark clone relationship in metadata"

24.5 Normalization and Deduplication Pipeline

The normalization and deduplication pipeline operates after extraction to organize blueprints into families, normalize variants, detect duplicates, and generate canonical representations.

Pipeline workflow:

normalization_deduplication_pipeline:
  stage_1_family_identification:
    action: "Identify blueprint families"
    input: "Extracted blueprints"
    process:
      - "Calculate pairwise similarity"
      - "Cluster blueprints"
      - "Label families"
    output: "Blueprint families"

  stage_2_normalization:
    action: "Normalize blueprints within families"
    input: "Blueprint families"
    process:
      - "Abstract technologies"
      - "Normalize names"
      - "Normalize capabilities"
      - "Normalize structure"
    output: "Normalized blueprints"

  stage_3_duplicate_detection:
    action: "Detect duplicates"
    input: "Normalized blueprints"
    process:
      - "Find exact duplicates"
      - "Find near duplicates"
      - "Find architectural clones"
    output: "Duplicate relationships"

  stage_4_canonical_generation:
    action: "Generate canonical templates"
    input: "Normalized blueprints"
    process:
      - "Identify common patterns"
      - "Create template structure"
      - "Document variations"
    output: "Canonical templates"

  stage_5_dataset_organization:
    action: "Organize dataset"
    input: "Canonical templates and blueprints"
    process:
      - "Link blueprints to families"
      - "Link blueprints to templates"
      - "Index for search"
    output: "Organized dataset"

  stage_6_quality_metrics:
    action: "Calculate quality metrics"
    input: "Organized dataset"
    process:
      - "Family coverage"
      - "Template confidence"
      - "Duplicate rate"
    output: "Quality metrics"

Quality metrics:

quality_metrics:
  family_coverage:
    metric: "Proportion of blueprints assigned to families"
    calculation: "blueprints_in_families / total_blueprints"
    target: "> 0.7"
    interpretation: "Higher is better - more structure in dataset"

  template_confidence:
    metric: "Confidence in canonical templates"
    calculation: "average_similarity_to_template"

  duplicate_rate:
    metric: "Proportion of duplicate blueprints"

  variation_coverage:
    metric: "Proportion of variations captured in templates"
    calculation: "variations_in_templates / total_variations"
    target: "> 0.8"
    interpretation: "Higher is better - templates are comprehensive"

  example_metrics:

24.6 Chapter Summary

  • Blueprint families represent groups of blueprints with shared fundamental architectural structure
  • Family identification uses structural similarity (kernel structure), functional similarity (capability overlap), and domain similarity criteria
  • Similarity metrics include kernel structure similarity, Jaccard capability similarity, component/interface type similarity, and graph edit distance
  • Variant normalization reduces variations to canonical forms through technology abstraction, naming normalization, capability normalization, and structural normalization
  • Technology abstraction maps specific technologies to generic categories (FastAPI → web_framework, PostgreSQL → relational_database)
  • Naming normalization maps domain-specific names to canonical forms (payment_service → api_service)
  • Capability normalization maps specific operations to generic patterns (create_payment → create_entity)
  • Duplicate detection identifies exact duplicates (100% match), near duplicates (>95% similarity), and architectural clones (>90% similarity)
  • Duplicate sources include repository duplicates (same repo extracted multiple times), architectural duplicates (identical architecture), and variant duplicates (minor variations)
  • Canonical representation generates normalized templates capturing shared architectural structure for each family
  • Canonical templates include kernel templates with required/optional elements, variation patterns, technology patterns, and statistics
  • Normalization benefits include pattern discovery, dataset organization, quality metrics, learning, and code generation
  • Pipeline workflow: family identification → normalization → duplicate detection → canonical generation → dataset organization → quality metrics
  • Quality metrics include family coverage (>0.7), template confidence (>0.85), duplicate rate (<0.1), and variation coverage (>0.8)
