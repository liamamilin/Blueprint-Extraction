20  Validation: Schema and Semantics

Blueprint extraction produces structured architectural representations from repository artifacts. Before these representations can be accepted as usable blueprints, they must pass a validation stage.

Validation ensures that the extracted blueprint is not only syntactically correct, but also structurally coherent and architecturally plausible. Two categories of validation are required: schema validation, which verifies structural correctness, and semantic validation, which verifies architectural consistency.

Together, these checks prevent malformed or misleading blueprints from entering blueprint libraries or architecture datasets. Validation acts as the first quality gate between automated extraction and human review.

20.1 Why Validation Matters

Validation serves as the critical bridge between automated extraction and architectural understanding. Without validation, extraction pipelines could produce misleading, incomplete, or structurally unsound representations that would degrade the quality of blueprint datasets and mislead architectural analysis.

20.2 DSL Validation

Blueprints are expressed using a formal Blueprint DSL. The first validation step ensures that the generated blueprint conforms to the DSL specification. This verification ensures that the blueprint is syntactically well-formed and compatible with downstream processing tools.

DSL validation requirements:

dsl_validation:
  specification_version: "Blueprint Extraction Spec v0.1"

  validation_rules:
    field_presence:
      rule: "All required fields must be present"
      required_fields:
        - blueprint.system.id
        - blueprint.system.name
        - blueprint.kernel.Goal
        - blueprint.kernel.Capabilities
        - blueprint.kernel.Components
        - blueprint.kernel.Interfaces
        - blueprint.kernel.Data
        - blueprint.kernel.Workflows
      consequence: "Blueprint rejected if missing required field"

    field_type_correctness:
      rule: "Each field must have correct type"
      type_requirements:
        - blueprint.system.id: string
        - blueprint.kernel.Capabilities: array
        - blueprint.kernel.Components: array
        - blueprint.kernel.Interfaces: array
        - blueprint.kernel.Data: array
        - blueprint.kernel.Workflows: array
      consequence: "Blueprint rejected if type mismatch"

    structural_nesting:
      rule: "Structure must follow DSL schema"
      nesting_rules:
        - blueprint:
          - system: required
          - kernel: required
          - kernel.Goal: required
          - kernel.Capabilities: array
          - kernel.Components: array
      consequence: "Blueprint rejected if structure invalid"

    valid_enumeration_values:
      rule: "Enumeration fields must use valid values"
      enumerations:
        - component.type: [Service, Worker, Scheduler, Plugin, ControlPlane]
        - interface.type: [API, Event, CLI, RPC]
        - data.type: [Table, Collection, Document]
        - workflow.type: [DAG, Pipeline, Schedule, Event, StateMachine]
        - confidence.state: [observed, inferred, unknown]
        - evidence.strength: [strong, moderate, weak]
      consequence: "Blueprint rejected if invalid enum value"

    identifier_formatting:
      rule: "Identifiers must follow naming conventions"
      identifier_rules:
        - component.id: "component.{name}"
        - interface.id: "interface.{name}"
        - data.id: "data.{name}"
        - workflow.id: "workflow.{name}"
        - capability.id: "capability.{name}"
      consequence: "Blueprint rejected if identifier format invalid"

    schema_compatibility:
      rule: "Blueprint must match current specification version"
      specification_version: "0.1"
      consequence: "Blueprint rejected if incompatible"

20.3 Reference Integrity

Architectural entities within a blueprint reference each other through relationships. For example, interfaces reference components that expose them, workflows reference tasks implemented by components, and components reference data entities they access. Validation must ensure that these references are complete and consistent.

Reference integrity requirements:

reference_integrity:
  validation_rules:
    all_references_must_resolve:
      rule: "Every referenced entity must exist"
      check_types:
        - interface.exposed_by: must reference existing component
        - interface.exposes_capability: must reference existing capability
        - workflow.implemented_by: must reference existing component
        - workflow.participates: must reference existing component
        - component.exposes: must reference existing interface
        - component.owns: must reference existing data
        - component.reads: must reference existing data
        - workflow.invokes: must reference existing interface
      consequence: "Blueprint flagged if reference does not resolve"

    identifiers_must_be_unique:
      rule: "All entity IDs must be unique"
      check_types:
        - component IDs: unique within Components
        - interface IDs: unique within Interfaces
        - data IDs: unique within Data
        - workflow IDs: unique within Workflows
        - capability IDs: unique within Capabilities
      consequence: "Blueprint rejected if duplicate IDs"

    relationship_consistency:
      rule: "Relationships must be mutually consistent"
      check_types:
        - component.exposes ↔ interface.exposed_by: must be bidirectional
        - component.owns ↔ data.owned_by: should be bidirectional
        - workflow.implements ↔ capability.implemented_by: should be bidirectional
      consequence: "Blueprint flagged if relationships inconsistent"

    foreign_key_constraints:
      rule: "Foreign key references must be valid"
      check_types:
        - data.foreign_keys: must reference primary keys
        - task.dependencies: must reference existing tasks
        - event.producer/consumer: must reference existing interfaces
      consequence: "Blueprint flagged if foreign keys invalid"

Reference integrity validation example:

reference_integrity_validation:
  entities:
    component.payment_service:
      id: "comp.payment_service"
      exposes: ["int.payment_api"]
      owns: ["data.payments", "data.transactions"]

    interface.payment_api:
      id: "int.payment_api"
      exposed_by: "comp.payment_service"

    data.payments:
      id: "data.payments"
      owned_by: "comp.payment_service"

  validation_checks:
    - check: interface.references_exist
      entity: interface.payment_api
      reference: exposed_by → comp.payment_service
      status: EXISTS
      severity: error

    - check: component.references_exist
      entity: component.payment_service
      reference: exposes → int.payment_api
      status: EXISTS
      severity: error

    - check: data.references_exist
      entity: component.payment_service
      reference: owns → data.payments
      status: EXISTS
      severity: error

    - check: bidirectional_consistency
      entities: [component.payment_service, interface.payment_api]
      consistency:
        - component.exposes: [interface.payment_api]
        - interface.exposed_by: component.payment_service
      status: CONSISTENT
      severity: warning

    - check: identifier_uniqueness
      all_ids: [comp.payment_service, int.payment_api, data.payments]
      duplicates: []
      status: UNIQUE
      severity: error

  overall_result: PASSED
  validation_summary: "All reference integrity checks passed"

    severity: error
    resolution: "Correct foreign key reference"

20.4 Kernel Completeness

The kernel represents the minimal architectural skeleton of the system. Validation therefore verifies that the kernel slots have been populated to an acceptable degree. Completeness checks evaluate whether sufficient structural information exists to represent the system meaningfully.

Completeness requirements:

kernel_completeness:
  validation_rules:
    minimal_slot_population:
      rule: "Each slot must have at least minimal content"
      slot_requirements:
        Goal:
          minimal: "One value or marked as unknown"
          acceptable: "Can be inferred if evidence insufficient"
        Capabilities:
          minimal: "At least one capability"
          acceptable: "Must be observed, not unknown"
        Components:
          minimal: "At least one component"
          acceptable: "Must be observed, not unknown"
        Interfaces:
          minimal: "At least one interface"
          acceptable: "Must be observed, not unknown"
        Data:
          minimal: "Optional if no persistence signals"
          acceptable: "Can be empty if no evidence"
        Workflows:
          minimal: "Optional if no orchestration signals"
          acceptable: "Can be empty if no evidence"
      consequence: "Blueprint flagged if critical slot underpopulated"

    capability_interface_coverage:
      rule: "Each capability should have at least one interface"
      requirement: "All observed capabilities must have interface access"
      exception: "Capabilities with only workflow implementation"
      consequence: "Blueprint flagged if capability lacks interface"

    component_interface_coverage:
      rule: "Each component should expose at least one interface"
      requirement: "All components must have at least one interface"
      exception: "Pure data processing components"
      consequence: "Blueprint flagged if component has no interfaces"

    data_ownership_coverage:
      rule: "Data entities should have identifiable ownership"
      requirement: "All data entities should have owner or reader"
      exception: "Shared data with multiple readers"
      consequence: "Blueprint flagged if data has no ownership"

    workflow_task_coverage:
      rule: "Workflows should have at least one task"
      requirement: "Workflows must define execution steps"
      exception: "State machines (event-driven)"
      consequence: "Blueprint flagged if workflow has no tasks"

20.5 Anomaly Detection

Even when a blueprint passes schema and reference checks, the extraction process may still produce structurally suspicious results. Anomaly detection identifies patterns that suggest potential extraction errors or architectural peculiarities that require investigation.

Anomaly types:

anomaly_types:
  structural_anomalies:
    - type: orphaned_component
      description: "Component with no interfaces, workflows, or responsibilities"
      pattern: "Component exists but has no connections"
      severity: medium
      possible_causes:
        - "Extraction error - component incorrectly identified"
        - "Genuine orphan - component in development"
        - "Hidden interface - interface not detected"

    - type: isolated_capability
      description: "Capability that cannot be reached through any interface"
      pattern: "Capability exists but no access path"
      severity: high
      possible_causes:
        - "Extraction error - interface not detected"
        - "Internal capability - intentionally hidden"
        - "Planned capability - not yet implemented"

    - type: unowned_data
      description: "Data entity without any owning component"
      pattern: "Data exists but no owner identified"
      severity: medium
      possible_causes:
        - "Extraction error - ownership not detected"
        - "Shared data - multiple components access"
        - "Legacy data - ownership unclear"

    - type: interface_orphan
      description: "Interface not exposed by any component"
      pattern: "Interface exists but no owner"
      severity: medium
      possible_causes:
        - "Extraction error - component not detected"
        - "External interface - not owned by system"
        - "Deprecated interface - component removed"

  density_anomalies:
    - type: excessive_dependencies
      description: "Component has unusually many dependencies"
      pattern: "Component references many other components"
      severity: low
      possible_causes:
        - "Genuine complexity - system is complex"
        - "God anti-pattern - component does too much"
        - "Extraction error - incorrect dependency detection"

    - type: fan_out_anomaly
      description: "Component or data entity referenced by many others"
      pattern: "Many entities depend on single entity"
      severity: medium
      possible_causes:
        - "Core component - central to architecture"
        - "God object - excessive responsibility"
        - "Shared resource - intentional design"

    - type: circular_dependencies
      description: "Circular dependency chain detected"
      pattern: "A depends on B, B depends on C, C depends on A"
      severity: high
      possible_causes:
        - "Architectural design - intentional circularity"
        - "Design flaw - unintended coupling"
        - "Extraction error - dependency misidentification"

  semantic_anomalies:
    - type: naming_convention_violation
      description: "Entity naming doesn't follow system conventions"
      pattern: "Inconsistent naming across entities"
      severity: low
      possible_cases:
        - "Multiple teams with different conventions"
        - "Refactoring in progress"
        - "Extraction error - name not normalized"

    - type: role_mismatch
      description: "Component type doesn't match its responsibilities"
      pattern: "Worker component with API-only responsibilities"
      severity: medium
      possible_causes:
        - "Extraction error - type misclassified"
        - "Architecture evolution - role changed"
        - "Multi-role component - legitimate complexity"

Anomaly validation rules:

anomaly_validation_rules:
  not_all_anomalies_are_errors:
    principle: "Anomalies don't automatically indicate incorrect extraction"
    handling:

      - severity: medium → "Should investigate before accepting"
      - severity: low → "Can accept with note"
    rationale: "Some patterns are legitimate architectural decisions"

  anomaly_categorization:
    categories:
      - extraction_errors:
        likelihood: "Should be corrected"
        action: "Fix extraction and regenerate"

      - genuine_architectural_features:

20.6 Validation as a Quality Gate

Validation acts as the first quality gate between automated extraction and human review. Schema validation ensures structural correctness. Reference checks ensure graph consistency. Kernel completeness ensures architectural substance. Anomaly detection identifies suspicious interpretations.

20.7 Chapter Summary

  • Validation ensures extracted blueprints are structurally sound and architecturally plausible
  • Two validation categories: schema validation (structural correctness) and semantic validation (architectural consistency)
  • DSL validation verifies blueprint conforms to specification with field presence, type correctness, structural nesting, enum validity, identifier formatting
  • Reference integrity ensures all references between entities resolve and relationships are consistent
  • Kernel completeness checks verify each slot has minimal required content and coverage
  • Capability-interface, component-interface, data-ownership, workflow-task coverage ensure structural connections
  • Anomaly detection identifies suspicious patterns: orphaned components, isolated capabilities, unowned data, excessive dependencies
  • Not all anomalies are errors; some represent legitimate architectural features
  • Anomalies are categorized as extraction errors, genuine features, or architectural debt
  • Validation acts as the first quality gate between extraction and review
  • Fatal validation failures cause rejection; warnings and anomalies cause flagging for review
  • Validation report provides comprehensive assessment of blueprint quality
  • Validated blueprints proceed to review; rejected blueprints require correction
