7  Repo ≠ System: The Missing Layers

A repository is often treated as the canonical representation of a system. In practice, this assumption is fundamentally incorrect. Treating repositories as equivalent to systems leads to extraction errors, architectural misunderstandings, and unreliable blueprints.

A repository contains implementation artifacts—the source code, configuration files, build scripts, and documentation that developers produce during software development. These artifacts represent how the system is implemented.

A system includes architectural elements—the structural organization, runtime behavior, operational constraints, and integration patterns that define what the system is and how it operates.

Many aspects of system architecture exist outside the repository or are only partially represented within it. Blueprint extraction must therefore acknowledge a fundamental limitation:

A repository is evidence about a system, not the system itself.

Understanding this distinction is essential for designing a reliable extraction methodology. It explains why extraction is difficult, why uncertainty is inevitable, and why evidence-based interpretation is necessary.

7.1 Missing Layers: Runtime, Operations, and Constraints

Several critical layers of a system are often absent or only partially represented in repositories. These missing layers create gaps that extraction must acknowledge rather than attempt to fill through speculation.

7.1.1 The Runtime Layer

The runtime environment defines how the system executes in production. This layer includes:

  • Execution topology: How services are deployed, containerized, and orchestrated
  • Service discovery: How components locate and communicate with each other
  • Scaling policies: How the system scales horizontally or vertically
  • Resource limits: CPU, memory, storage, and network constraints
  • Load balancing: How requests are distributed across instances
  • Failover mechanisms: How the system handles component failures

Runtime behavior is often only partially visible in repositories. While deployment scripts and infrastructure definitions may provide hints, the actual runtime state frequently exists outside the codebase in:

  • Container orchestration platforms (Kubernetes, Docker Swarm)
  • Cloud provider configurations (AWS, GCP, Azure)
  • Infrastructure-as-code repositories separate from application code
  • Operational dashboards and monitoring systems

Extraction can detect signals about runtime behavior from deployment manifests and configuration files, but the complete runtime picture often remains inaccessible.

7.1.2 The Operations Layer

The operations layer encompasses how the system is monitored, maintained, and supported in production. Operational concerns include:

  • Monitoring infrastructure: Metrics collection, logging aggregation, distributed tracing
  • Alerting mechanisms: Thresholds, notification channels, escalation policies
  • Failure handling: Retry logic, circuit breakers, fallback mechanisms
  • Rollback procedures: How the system reverts to previous stable states
  • Incident response: Runbooks, on-call procedures, escalation workflows
  • Performance tuning: Optimization strategies, caching mechanisms, query optimization

Some operational signals may appear in repositories through:

  • CI/CD pipeline definitions
  • Observability configuration files
  • Monitoring instrumentation code
  • Infrastructure manifests

However, operational practices are often maintained in external systems, runbooks, or tribal knowledge rather than in source code. The operational layer is therefore a partially observable domain for extraction.

7.1.3 The Constraints Layer

The constraints layer defines limitations, guarantees, and requirements that shape system architecture. Constraints include:

  • Performance requirements: Response time limits, throughput targets, latency budgets
  • Security boundaries: Authentication requirements, authorization policies, data classification
  • Compliance rules: Regulatory requirements, audit trails, data retention policies
  • Resource limits: Budget constraints, infrastructure quotas, capacity planning
  • Service-level objectives (SLOs): Availability targets, error rate limits, performance guarantees
  • Business constraints: Feature flags, rollout strategies, A/B testing requirements

Constraints frequently exist in documentation, infrastructure policies, organizational processes, or external systems rather than in source code. When constraints do appear in repositories, they may be:

  • Documented in policy files or configuration
  • Encoded in infrastructure-as-code templates
  • Referenced in monitoring or alerting configurations
  • Implicit in architectural decisions and patterns

Because these layers are only partially observable, blueprint extraction must treat them as incomplete evidence domains. Missing information must be explicitly marked as unknown rather than silently inferred or guessed.

7.2 Conventions Versus Explicit Evidence

Repositories frequently encode architectural intent indirectly through conventions rather than explicit declarations. This distinction is critical for reliable extraction.

7.2.1 Explicit Evidence

Explicit evidence consists of artifacts that directly encode architectural structure. These artifacts provide strong, unambiguous signals about architecture:

  • API route definitions: HTTP route handlers, OpenAPI specifications, RPC service definitions
  • Workflow DAG specifications: Pipeline definitions, orchestration workflows, task dependencies
  • Database migrations: Schema DDL, migration files, evolution history
  • Interface contracts: Protocol buffers, gRPC definitions, message schemas
  • Deployment manifests: Kubernetes specifications, Docker compose files, infrastructure definitions

Explicit evidence generally yields high confidence because the architectural meaning is directly encoded in the artifact structure.

7.2.2 Convention-Based Inference

Convention-based inference relies on patterns observed in repository organization, naming conventions, or framework-specific behaviors. These conventions suggest architecture but do not explicitly declare it:

  • Directory structures: services/, workers/, plugins/ suggesting component boundaries
  • Naming patterns: *-service.py, *-worker.go, *-scheduler.ts suggesting component types
  • Framework conventions: Spring Boot controllers, Express middleware, Django apps suggesting interfaces
  • Module organization: Package structure suggesting subsystem decomposition

Convention-based inference is inherently weaker than explicit evidence because:

  • Conventions vary across teams, organizations, and projects
  • The same convention may have different meanings in different contexts
  • Conventions may be applied inconsistently or evolve over time
  • Without explicit validation, conventions may be misleading

7.2.3 The Extraction Implications

Distinguishing between explicit evidence and convention-based inference is essential for:

  • Confidence assignment: Explicit evidence receives higher confidence than conventional inference
  • Review prioritization: Convention-based inferences require more careful review
  • Uncertainty representation: Conventions should be marked with appropriate confidence levels
  • Error detection: Inconsistent conventions may signal misinterpretation

Extraction pipelines should therefore classify evidence sources and weight confidence accordingly. Explicit evidence forms the foundation; convention-based inference provides supplementary context that must be validated.

7.3 Mono-Repository and Poly-Repository Systems

Repository topology significantly affects extraction difficulty and methodology. Different organizational patterns present distinct challenges for architectural reconstruction.

7.3.1 Mono-Repository Architectures

In a mono-repository architecture, multiple components or services are stored within a single repository. This pattern is common in organizations that prefer centralized version control and simpler dependency management.

Extraction advantages in mono-repositories:

  • Visible component relationships: Inter-component dependencies are often visible through shared build configurations, internal imports, and cross-module references
  • Unified structural signals: Directory structures may clearly indicate component boundaries and subsystem organization
  • Simplified evidence collection: All evidence exists within a single repository, simplifying access and traceability
  • Consistent conventions: Single repository often enforces consistent patterns across components

Extraction challenges in mono-repositories:

  • Obscured service boundaries: Tightly coupled modules may not correspond to deployable services
  • Implicit dependencies: Shared code and libraries may create hidden coupling between components
  • Mixed responsibilities: Single directories may contain multiple concerns, complicating component identification
  • Build complexity: Complex build configurations may obscure deployment boundaries

Mono-repositories therefore provide rich signals about internal structure but may require careful analysis to identify true architectural boundaries versus module organization.

7.3.2 Poly-Repository Architectures

In poly-repository architectures, system components are distributed across multiple repositories. Each repository typically contains a single service, subsystem, or independent module.

Extraction advantages in poly-repositories:

  • Clear service boundaries: Repository boundaries often align with deployable services
  • Simplified component identification: Each repository likely corresponds to a distinct architectural component
  • Explicit dependencies: External dependencies are declared through repository references or package management
  • Clear ownership: Repository ownership often maps to team responsibility

Extraction challenges in poly-repositories:

  • Inter-service relationships: Relationships between repositories may only be visible through external artifacts such as deployment manifests, API documentation, or infrastructure configuration
  • Scattered evidence: Evidence for system-level architecture is distributed across multiple repositories
  • Dependency inference: Inter-service dependencies must be inferred from deployment configurations, service discovery mechanisms, or communication patterns
  • Integration context: How services integrate and coordinate may be poorly documented

Poly-repositories therefore provide clearer component boundaries but make inter-component relationships and system-level architecture more difficult to reconstruct.

7.3.3 Cross-Repository Aggregation

Extraction pipelines must therefore support cross-repository aggregation to reconstruct complete system architectures. This requires:

  • Repository discovery: Identifying all repositories that constitute a system
  • Relationship inference: Determining how repositories relate to each other
  • Evidence collection: Gathering evidence from multiple repositories
  • Synthesis: Combining repository-specific extractions into system-level blueprints

Cross-repository extraction adds complexity but is essential for systems whose components are distributed across multiple codebases.

7.4 Implications for Blueprint Extraction

Because repositories do not fully represent systems, blueprint extraction must incorporate several safeguards and methodological principles.

7.4.1 Explicit Representation of Missing Information

Extraction must distinguish between:

  • Observed structure: Architectural elements explicitly present in repository artifacts
  • Inferred structure: Architectural elements derived from indirect signals and interpretation
  • Unknown structure: Architectural elements for which insufficient evidence exists

Rather than guessing or silently assuming missing information, extraction must explicitly represent these gaps as unknown fields in the blueprint kernel. This honest representation prevents architectural reconstructions from becoming speculative.

7.4.2 Confidence-Based Interpretation

Because repositories provide incomplete signals, extraction must assign appropriate confidence levels to all architectural claims:

  • High confidence: Elements supported by explicit, strong evidence
  • Moderate confidence: Elements inferred from conventions, patterns, or indirect signals
  • Low confidence: Elements with weak, ambiguous, or conflicting evidence
  • Unknown: Elements for which evidence is insufficient

Confidence annotations guide human review and help identify areas requiring additional investigation or documentation.

7.4.3 Evidence-Centric Methodology

The evidence-centric methodology becomes particularly important given repository limitations:

  • Evidence collection: Systematically gather all available evidence for each architectural element
  • Conflict preservation: Preserve conflicting evidence rather than discarding it
  • Evidence aggregation: Combine multiple weak signals to strengthen interpretations
  • Traceability: Maintain clear links between architectural claims and evidence sources

7.4.4 Human Review as Validation

Given the inherent limitations of repository-based extraction, human review becomes essential for:

  • Validating interpretations: Confirming that extracted architecture reflects system intent
  • Adding missing information: Incorporating knowledge that exists outside repositories
  • Resolving ambiguities: Making judgment calls when evidence is conflicting or insufficient
  • Correcting misinterpretations: Fixing extraction errors that arise from misleading signals

Human review bridges the gap between what repositories reveal and what systems actually are.

7.5 Chapter Summary

  • A repository contains implementation artifacts, but a system includes additional architectural layers
  • The missing layers include runtime environment, operational practices, and system constraints
  • These layers are often absent from repositories or only partially represented
  • Extraction distinguishes between explicit evidence (strong signals) and convention-based inference (weak signals)
  • Mono-repository architectures provide rich internal signals but may obscure true component boundaries
  • Poly-repository architectures provide clear component boundaries but obscure inter-component relationships
  • Cross-repository aggregation is required for systems distributed across multiple codebases
  • Extraction must explicitly represent missing information as unknown rather than speculating
  • Confidence-based interpretation acknowledges the inherent uncertainty of repository-based extraction
  • Evidence-centric methodology ensures traceability and validation of architectural claims
  • Human review is essential for bridging the gap between repositories and actual systems
