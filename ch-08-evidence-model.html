10  Evidence Object Model

Blueprint extraction relies on a fundamental discipline: architectural claims must be grounded in traceable evidence. This discipline is not optional—it is the foundation upon which reliable extraction rests.

Repositories contain large volumes of artifacts, but extraction must isolate the specific fragments that justify architectural interpretations. These fragments are formalized as evidence objects.

An evidence object represents a structured reference to a repository artifact that supports a particular architectural inference. The Evidence Object Model defines how these artifacts are represented, aggregated, and packaged for auditability.

Without a structured evidence model, blueprint extraction would rely on implicit assumptions, opaque decisions, and unverified claims. With it, extraction becomes a verifiable engineering discipline where every architectural element can be traced back to its source.

10.1 Why Evidence Matters

Evidence is the bridge between repository artifacts and architectural understanding. Without evidence, extraction produces speculation. With evidence, extraction produces verifiable architectural knowledge.

The evidence requirement:

Every extracted architectural element must be supported by traceable evidence.

This principle is absolute. An extracted component, interface, workflow, or data entity must reference concrete repository artifacts that justify its existence, structure, and relationships.

What evidence provides:

  1. Traceability: Reviewers can locate and inspect the artifacts supporting each claim
  2. Verifiability: Claims can be validated against original source materials
  3. Debuggability: Errors can be traced to specific evidence or interpretation errors
  4. Reproducibility: The same evidence produces consistent results across runs
  5. Auditability: Extraction decisions can be reviewed and challenged

Without evidence:

Architectural claims become: - Speculative: Based on intuition rather than observation - Unverifiable: Cannot be validated or challenged - Unreliable: Prone to errors and misinterpretations - Irreproducible: Different extractions produce different results - Unmaintainable: Cannot be updated or corrected systematically

With evidence:

Architectural claims become: - Grounded: Based on observable artifacts - Verifiable: Can be inspected and validated - Reliable: Supported by traceable artifacts - Reproducible: Consistent across extractions - Maintainable: Can be corrected and improved systematically

Evidence therefore transforms extraction from heuristic inference into evidence-driven engineering.

10.2 Evidence Object Overview

An evidence object represents a specific fragment of a repository artifact that supports a hypothesis about system structure. Evidence objects are attached to architectural entities such as components, interfaces, workflows, data entities, capabilities, and constraints.

Evidence object characteristics:

  • Specific: Evidence points to precise locations within artifacts
  • Traceable: Evidence can be followed back to original sources
  • Fragmentary: Evidence captures relevant portions, not entire artifacts
  • Typed: Evidence is classified by artifact type and signal category
  • Weighted: Evidence includes strength indicators

Evidence relationships:

  • One-to-many: A single evidence object may support multiple architectural elements
  • Many-to-one: Multiple evidence objects may support a single architectural element
  • Conflicting: Evidence objects may contradict each other
  • Complementary: Evidence objects may support different aspects of the same element

Evidence lifecycle:

Evidence objects progress through the extraction pipeline:

Detection → Collection → Classification → Aggregation → Validation → Review
  1. Detection: Artifacts are identified as potential evidence
  2. Collection: Relevant fragments are extracted from artifacts
  3. Classification: Evidence is typed and categorized
  4. Aggregation: Multiple evidence objects are grouped
  5. Validation: Evidence is checked for conflicts and consistency
  6. Review: Evidence is validated by human reviewers

10.3 Core Evidence Schema

The Evidence Object Model v0.1 defines the canonical structure for representing traceable architectural evidence. This schema ensures consistency across extraction systems and enables automated processing.

Minimal evidence object representation:

evidence:
  id: <unique-evidence-identifier>

  source:
    repo: <repository-identifier>
    file: <file-path>
    revision: <commit-or-tag>
    branch: <branch-name>

  location:
    type: <line-range | json-path | object-path>
    value: <location-reference>

  excerpt: |
    <artifact-fragment>

  artifact_type: <code | config | schema | deploy | doc | test>

  signal_category: <structure | dependency | entrypoint | config | deploy |
                    docs | test | observability | security>

  strength: <strong | moderate | weak>

  confidence_implication: <observed | inferred | unknown>

  created_by:
    extractor: <tool-or-pipeline-id>
    version: <extractor-version>
    timestamp: <iso8601>

  related_entities:
    - <entity-id>
    - <entity-id>

Field explanations:

10.3.1 id

A unique identifier for the evidence object. Evidence identifiers should remain stable within a blueprint pack to support traceability and cross-referencing. IDs typically follow a pattern such as evidence-{hash} or evidence-{counter}.

10.3.2 source

The source field identifies the repository artifact from which the evidence originates:

  • repo: Repository identifier (URL, name, or hash)
  • file: File path within the repository
  • revision: Git commit hash, tag, or other version identifier
  • branch: Branch name (if different from default)

This information ensures that reviewers can locate the exact artifact used during extraction, even as repositories evolve.

10.3.3 location

The location field specifies the exact position within the artifact. Depending on the artifact type, this may include:

  • line-range: For code and text files (e.g., lines 42-58)
  • json-path: For JSON/YAML configuration (e.g., $.services.api.port)
  • object-path: For structured documents (e.g., services.api.host)

Precise location references enable automated tools to retrieve the same evidence fragment during audits and enable reviewers to quickly locate relevant sections.

10.3.4 excerpt

The excerpt field contains the relevant fragment of the artifact. Providing the excerpt improves review efficiency because reviewers can immediately inspect the relevant content without navigating the entire file.

Excerpt guidelines:

  • Minimal: Include only the relevant portion
  • Contextual: Include sufficient surrounding context for understanding
  • Clean: Remove irrelevant whitespace or comments
  • Complete: Ensure the excerpt is syntactically complete

Example excerpts:

# Excerpt from route definition
@app.post("/payments")
async def create_payment(request: PaymentRequest):
    # Process payment
    return {"payment_id": payment.id}
# Excerpt from configuration
services:
  api:
    host: api.example.com
    port: 8000

10.3.5 artifact_type

This field classifies the artifact that produced the evidence:

  • code: Source code files (.py, .js, .go, .java, etc.)
  • config: Configuration files (.yaml, .json, .env, .ini, etc.)
  • schema: Schema definitions (DDL, migrations, OpenAPI, etc.)
  • deploy: Deployment artifacts (Dockerfile, k8s manifests, Terraform, etc.)
  • doc: Documentation files (.md, .rst, .txt, etc.)
  • test: Test files (test_.py, _test.go, etc.)

Artifact classification helps extraction systems evaluate the reliability of the evidence. For example, schema definitions often provide stronger architectural signals than documentation.

10.3.6 signal_category

This field records the signal taxonomy category that produced the evidence:

  • structure: Repository organization signals
  • dependency: Dependency declaration signals
  • entrypoint: Application entrypoint signals
  • config: Configuration signals
  • deploy: Deployment signals
  • docs: Documentation signals
  • test: Test suite signals
  • observability: Monitoring and instrumentation signals
  • security: Security mechanism signals

Signal categories allow the extraction pipeline to trace architectural interpretations back to their originating signal class.

10.3.7 strength

Evidence strength estimates how directly the artifact supports an architectural interpretation:

  • strong: Artifacts that explicitly encode architectural structure
  • moderate: Artifacts that strongly suggest architecture but do not explicitly define it
  • weak: Artifacts that provide indirect hints about system structure

Strength examples:

Strong evidence:

# Schema migration explicitly defining data structure
CREATE TABLE payments (
    id SERIAL PRIMARY KEY,
    amount DECIMAL(10,2) NOT NULL,
    status VARCHAR(50) NOT NULL
);

Moderate evidence:

// Configuration suggesting service dependency
{
  "dependencies": {
    "payment_service": "http://payments:8000"
  }
}

Weak evidence:

# Directory name suggesting component boundary
workers/

Strength classification contributes to confidence evaluation during extraction.

10.3.8 confidence_implication

This field indicates the confidence level that should be assigned to architectural elements based on this evidence:

  • observed: Strong evidence supports high confidence
  • inferred: Moderate evidence suggests moderate confidence
  • unknown: Weak evidence may result in unknown or low confidence

This field provides guidance for confidence calculation rather than mandating specific confidence levels.

10.3.9 created_by

The created_by section records metadata about the extraction process that generated the evidence:

  • extractor: Tool or pipeline identifier (e.g., “blueprint-extractor-v1.2”)
  • version: Extractor version for reproducibility
  • timestamp: ISO8601 timestamp of extraction

This metadata ensures that evidence objects remain traceable to the pipeline that produced them.

10.3.10 related_entities

This field lists architectural entities that this evidence supports. This linkage enables:

  • Evidence lookup: Finding all evidence supporting a specific entity
  • Impact analysis: Understanding which entities would be affected if evidence changes
  • Validation: Ensuring all entities have supporting evidence

10.4 Evidence Aggregation

A single architectural element is rarely supported by a single artifact. Instead, architectural interpretation typically emerges from multiple independent signals.

Aggregation patterns:

Worker component evidence aggregation:

A worker component might be supported by:

# Evidence 1: Directory structure
evidence:
  id: evidence-001
  source:
    file: workers/
  artifact_type: code
  signal_category: structure
  strength: moderate

# Evidence 2: Queue dependency
evidence:
  id: evidence-002
  source:
    file: requirements.txt
  location:
    type: json-path
    value: $.dependencies.celery
  excerpt: "celery==5.3.0"
  artifact_type: config
  signal_category: dependency
  strength: strong

# Evidence 3: Worker implementation
evidence:
  id: evidence-003
  source:
    file: workers/processor.py
  location:
    type: line-range
    value: lines 1-45
  excerpt: |
    from celery import Celery

    app = Celery('tasks')
    @app.task
    def process_payment(payment_id):
        # Process payment
        pass
  artifact_type: code
  signal_category: entrypoint
  strength: strong

# Evidence 4: Deployment configuration
evidence:
  id: evidence-004
  source:
    file: docker-compose.yml
  location:
    type: json-path
    value: $.services.worker
  excerpt: |
    services:
      worker:
        image: myapp/worker:latest
        command: celery -A workers worker
  artifact_type: deploy
  signal_category: deploy
  strength: strong

Aggregation purposes:

  1. Increased confidence: Multiple consistent signals strengthen confidence in the interpretation
  2. Improved auditability: The complete evidence basis for each element is visible to reviewers
  3. Robustness: If one evidence source is incorrect or misleading, other sources may compensate
  4. Validation: Cross-validation across multiple evidence sources improves reliability

Aggregation confidence calculation:

Confidence can be calculated from aggregated evidence:

base_confidence = strength(weakest_evidence)
confidence_boost = log(count(evidence))
conflict_penalty = number_of_conflicts
final_confidence = base_confidence + confidence_boost - conflict_penalty

This formula ensures that: - Weak evidence limits maximum confidence - More evidence increases confidence (diminishing returns) - Conflicts reduce confidence

10.5 Evidence Conflict Resolution

Repositories may contain conflicting signals. Conflicts arise when different artifacts suggest inconsistent architectural interpretations.

Conflict sources:

  1. Outdated documentation: Documentation describes components that no longer exist
  2. Legacy configuration: Configuration files reference deprecated services
  3. Evolutionary drift: Architecture has evolved but artifacts haven’t been updated
  4. Ambiguous naming: Names suggest structures that differ from actual implementation

Conflict examples:

Documentation vs. implementation conflict:

# Documentation (outdated)
evidence:
  id: evidence-001
  source:
    file: README.md
  excerpt: "The payment service is deployed as a standalone microservice"
  artifact_type: doc
  signal_category: docs
  strength: moderate

# Implementation (current reality)
evidence:
  id: evidence-002
  source:
    file: main.py
  excerpt: |
    @app.post("/payments")
    def create_payment(request):
        # Payment processing inline with API
        pass
  artifact_type: code
  signal_category: code
  strength: strong

Configuration vs. code conflict:

# Configuration (legacy)
evidence:
  id: evidence-003
  source:
    file: config.yml
  excerpt: |
    queue_url: redis://legacy-queue:6379
  artifact_type: config
  strength: moderate

# Code (current)
evidence:
  id: evidence-004
  source:
    file: workers.py
  excerpt: "queue = Connection('redis://new-queue:6379')"
  artifact_type: code
  strength: strong

Conflict resolution principles:

Extraction systems must preserve conflicting evidence rather than discard it. Conflict resolution occurs at the mapping or review stage, not during evidence collection.

Preservation strategy:

  1. Retain all evidence: Keep conflicting evidence objects in the evidence bundle
  2. Flag conflicts: Mark evidence relationships as conflicting
  3. Assign confidence: Lower confidence for interpretations based on conflicting evidence
  4. Require review: Flag conflicts for human review and resolution

Resolution approaches:

  • Timestamp-based: Prefer newer evidence over older evidence
  • Strength-based: Prefer stronger evidence over weaker evidence
  • Consensus-based: Prefer interpretations supported by multiple consistent sources
  • Manual resolution: Require human review for unresolvable conflicts

Conflict representation:

evidence_conflict:
  conflict_id: conflict-001
  conflicting_evidence:
    - evidence-001
    - evidence-002
  resolution: pending
  confidence_implication: reduced

This explicit conflict representation ensures that architectural interpretation remains transparent and reviewable.

10.6 Evidence Bundle

To maintain auditability, all evidence objects generated during extraction are grouped into an evidence bundle. The bundle serves as the audit trail for the extraction process.

Evidence bundle contents:

An evidence bundle contains:

  • Evidence index: Complete catalog of all evidence objects
  • Evidence objects: All structured evidence representations
  • Evidence excerpts: Artifact fragments for quick inspection
  • Evidence relationships: Links between evidence and architectural entities
  • Conflict registry: Record of identified evidence conflicts

Bundle directory structure:

blueprint-pack/
├── evidence/
│   ├── index.yaml              # Evidence catalog and relationships
│   ├── evidence-001.yaml       # Individual evidence objects
│   ├── evidence-002.yaml
│   ├── ...
│   └── excerpts/               # Artifact fragments
│       ├── evidence-001.txt
│       ├── evidence-002.txt
│       └── ...

Evidence index example:

evidence_index:
  total_evidence: 42
  by_type:
    code: 15
    config: 12
    schema: 8
    deploy: 4
    doc: 3
  by_strength:
    strong: 18
    moderate: 16
    weak: 8
  conflicts: 3
  evidence_objects:
    - id: evidence-001
      type: code
      strength: strong
      supports:
        - component.worker
        - interface.queue
    - id: evidence-002
      type: config
      strength: moderate
      supports:
        - component.worker

Bundle purposes:

The evidence bundle functions as the audit trail for blueprint extraction:

  1. Traceability: Every architectural element references evidence in the bundle
  2. Verification: Reviewers can verify claims by inspecting the bundle
  3. Debugging: Errors can be traced to specific evidence or interpretation
  4. Reproducibility: The same evidence produces consistent results
  5. Evolution: Blueprints can be updated as evidence changes

Bundle in review:

During human review, the evidence bundle enables reviewers to:

  • Inspect: View all evidence supporting each architectural element
  • Validate: Verify that evidence supports the claimed interpretation
  • Challenge: Question or reject interpretations based on weak evidence
  • Correct: Update interpretations when evidence is misinterpreted
  • Add: Supplement evidence from external sources

10.7 Evidence Quality Assessment

Not all evidence is equally valuable. Evidence quality assessment helps extraction systems prioritize evidence and calculate confidence.

Quality dimensions:

1. Strength

How directly the artifact expresses architectural structure:

  • Strong: Explicit definitions (schemas, routes, workflows)
  • Moderate: Strong suggestions (dependencies, configuration)
  • Weak: Indirect hints (naming, structure)

2. Freshness

How current the evidence is relative to the repository state:

  • Fresh: Recently modified, aligns with current code
  • Stale: Outdated, conflicts with current implementation
  • Unknown: Modification timestamp unavailable

3. Consistency

How well evidence aligns with other evidence sources:

  • Consistent: Aligns with multiple other sources
  • Conflicting: Contradicts other evidence
  • Unique: No corroborating evidence

4. Specificity

How precisely the evidence identifies architectural elements:

  • Specific: Uniquely identifies a single element
  • Ambiguous: Could apply to multiple elements
  • General: Provides broad guidance only

Quality scoring:

Evidence quality can be calculated as:

quality_score = (strength_weight * strength) +
                (freshness_weight * freshness) +
                (consistency_weight * consistency) +
                (specificity_weight * specificity)

Where each dimension is scored 0-1 and weights sum to 1.0.

Quality applications:

  • Confidence calculation: Higher quality evidence increases confidence
  • Review prioritization: Low-quality evidence requires more careful review
  • Evidence ranking: High-quality evidence should be prioritized in extraction
  • Conflict resolution: Higher-quality evidence should win conflicts

10.8 Evidence and Confidence Relationship

Evidence and confidence are closely related. Confidence represents the degree of certainty that an extracted element accurately reflects system architecture, and confidence is derived from evidence.

Confidence from evidence:

Observed confidence (high):

  • Supported by strong evidence (schemas, routes, workflows)
  • Multiple consistent evidence sources
  • No conflicting evidence
  • Specific, unambiguous interpretation

Example:

# Strong, consistent, specific evidence
evidence:
  - type: schema
    strength: strong
    excerpt: "CREATE TABLE payments (...)"
  - type: code
    strength: strong
    excerpt: "class Payment(db.Model): ..."
  - type: test
    strength: moderate
    excerpt: "def test_create_payment(): ..."

confidence: observed

Inferred confidence (medium):

  • Supported by moderate evidence (dependencies, configuration)
  • Some evidence consistency
  • Possible conflicting interpretations
  • Partially specific interpretation

Example:

# Moderate, somewhat consistent, partially specific evidence
evidence:
  - type: dependency
    strength: moderate
    excerpt: "celery==5.3.0"
  - type: structure
    strength: weak
    excerpt: "workers/"

confidence: inferred

Unknown confidence (low):

  • Weak or insufficient evidence
  • Conflicting evidence sources
  • Ambiguous or non-specific interpretation
  • Evidence primarily from documentation

Example:

# Weak, potentially conflicting, ambiguous evidence
evidence:
  - type: doc
    strength: weak
    excerpt: "Payment service processes transactions"

confidence: unknown

Confidence calculation formula:

base_confidence = strength(primary_evidence)
evidence_boost = log(count(consistent_evidence))
conflict_penalty = number_of_conflicts * conflict_weight
quality_multiplier = quality_score / max_quality

final_confidence = (base_confidence + evidence_boost - conflict_penalty) *
                    quality_multiplier

This formula ensures that confidence reflects: - Evidence strength - Evidence quantity (with diminishing returns) - Evidence conflicts - Evidence quality

10.9 Evidence as Extraction Foundation

The Evidence Object Model provides the structural foundation for blueprint extraction. Without it, extraction would lack:

  • Verifiability: Claims could not be validated
  • Auditability: Decisions could not be traced
  • Reproducibility: Results would vary between runs
  • Debuggability: Errors could not be located
  • Reliability: Blueprints would accumulate speculation

With the Evidence Object Model, extraction becomes:

Evidence-driven:

  • Signals initiate hypotheses
  • Evidence validates or refutes hypotheses
  • Mappings translate evidence into ontology structures
  • Blueprints assemble validated structures

Auditable:

  • Every claim references evidence
  • Every evidence references artifacts
  • Every artifact is locatable
  • Every decision is traceable

Improvable:

  • Evidence quality can be measured
  • Evidence collection can be optimized
  • Evidence interpretation can be refined
  • Evidence conflicts can be resolved

Scalable:

  • Evidence is structured for machine processing
  • Evidence bundles can be versioned and diffed
  • Evidence quality can be automated
  • Evidence review can be prioritized

The Evidence Object Model therefore transforms blueprint extraction from heuristic inference into a verifiable architecture reconstruction process.

10.10 Chapter Summary

  • Evidence is the foundation of reliable extraction—every claim must be traceable
  • Evidence objects represent structured references to repository artifacts
  • The evidence schema includes: id, source, location, excerpt, type, category, strength, and metadata
  • Evidence aggregation combines multiple signals to strengthen interpretations
  • Evidence conflicts must be preserved and resolved systematically
  • Evidence bundles provide the audit trail for extraction processes
  • Evidence quality is assessed through strength, freshness, consistency, and specificity
  • Confidence is derived from evidence through calculation
  • Observed confidence comes from strong, consistent, specific evidence
  • Inferred confidence comes from moderate evidence
  • Unknown confidence results from weak or insufficient evidence
  • The Evidence Object Model enables verifiable, auditable, reproducible extraction
  • Evidence-driven extraction is the foundation of reliable architectural reconstruction
