16  Workflow and Control-Plane Extraction

While components define structural units of responsibility and interfaces define interaction surfaces, many systems rely on explicit coordination mechanisms that orchestrate how work flows across the architecture. These coordination mechanisms are represented as workflows.

Workflows describe ordered or conditional execution across components, tasks, or services. They often encode the operational logic that determines how capabilities are realized over time. Blueprint extraction therefore reconstructs both the workflow structures and the control-plane mechanisms that govern execution.

The result is a representation of the system’s control flow architecture, revealing how capabilities are operationalized through orchestrated execution across components and tasks.

16.1 What Is a Workflow?

A workflow is a structured definition of execution order and dependencies between tasks or operations. Workflows encode the orchestration logic that determines when and how tasks execute.

Workflow characteristics:

1. Task coordination:

Workflows coordinate the execution of multiple tasks in a specified order or according to defined conditions.

workflow:
  id: workflow.payment_processing
  description: "Process payment through multiple stages"

  tasks:
    - task: validate_payment
      type: validation
      position: 1

    - task: charge_gateway
      type: external_call
      position: 2
      depends_on: validate_payment

    - task: record_transaction
      type: database_write
      position: 3
      depends_on: charge_gateway

    - task: send_notification
      type: notification
      position: 4
      depends_on: record_transaction

2. Dependency specification:

Workflows explicitly define dependencies between tasks, ensuring that prerequisite tasks complete before dependent tasks begin.

workflow:
  id: workflow.settlement
  description: "Settle completed payments"

  dependencies:
    - task: aggregate_payments
      depends_on: []

    - task: calculate_fees
      depends_on: [aggregate_payments]

    - task: transfer_to_bank
      depends_on: [calculate_fees]

    - task: update_payment_status
      depends_on: [transfer_to_bank]

3. Conditional execution:

Workflows may include conditional logic that determines which tasks execute based on runtime conditions.

workflow:
  id: workflow.payment_handling
  description: "Handle payment with conditional logic"

  conditional_logic:
    - condition: payment.status == 'failed'
      execute_tasks: [retry_payment, notify_support]

    - condition: payment.status == 'completed'
      execute_tasks: [send_confirmation, update_analytics]

    - condition: payment.amount > 10000
      execute_tasks: [manual_review_required]

4. Error handling and retries:

Workflows define how to handle failures, including retry logic and fallback behaviors.

workflow:
  id: workflow.payment_processing
  description: "Process payment with error handling"

  error_handling:
    task: charge_gateway
      on_failure:
        strategy: retry
        max_retries: 3
        backoff: exponential
      fallback_task: queue_for_manual_review

Workflow types:

The workflow ontology defines several common workflow archetypes:

workflow_types:
  Pipeline:
    role: "Sequential execution of tasks"
    characteristics:
      - Linear task sequence
      - Each task depends on previous
      - No branching
    examples: [ETL pipelines, batch processing]

  DAG:
    role: "Directed acyclic graph of tasks"
    characteristics:
      - Task dependencies form graph
      - Multiple dependency paths possible
      - No cycles
    examples: [Data processing workflows, CI/CD pipelines]

  Schedule:
    role: "Time-based execution"
    characteristics:
      - Triggered by time
      - Cron expressions or intervals
      - Recurring execution
    examples: [Periodic jobs, maintenance tasks]

  Event:
    role: "Event-driven execution"
    characteristics:
      - Triggered by events
      - Message queues or webhooks
      - Reactive execution
    examples: [Event handlers, message consumers]

# orchestrate_payment.py
def orchestrate_payment(payment_id):
    """Orchestrate payment processing workflow"""

    try:
        # Stage 1: Validation
        payment = fetch_payment(payment_id)
        validate_payment(payment)

        # Stage 2: Fraud check
        fraud_check_result = check_fraud(payment)
        if fraud_check_result.is_fraudulent:
            raise FraudDetectedError("Payment flagged as fraudulent")

        # Stage 3: Processing
        transaction_result = charge_gateway(payment)
        record_transaction(payment_id, transaction_result)

        # Stage 4: Notification
        send_confirmation_email(payment)

        return {"status": "completed", "payment_id": payment_id}

    except FraudDetectedError as e:
        queue_for_manual_review(payment_id)
        return {"status": "flagged", "payment_id": payment_id}

    except Exception as e:
        log_error(payment_id, str(e))
        return {"status": "failed", "payment_id": payment_id}

3. Checkpoint or retry configuration:

# Workflow with checkpointing
class PaymentWorkflow:
    def __init__(self, state_store):
        self.state_store = state_store

    def process_payment(self, payment_id):
        """Process payment with checkpointing"""

        # Check if already started
        state = self.state_store.get_state(payment_id)
        if state:
            # Resume from checkpoint
            start_stage = state.get('current_stage', 'validation')
        else:
            # Start from beginning
            start_stage = 'validation'

        try:
            if start_stage <= 'validation':
                self.validate_payment(payment_id)
                self.state_store.save_state(payment_id, {
                    'current_stage': 'validation',
                    'status': 'completed'
                })

            if start_stage <= 'processing':
                self.charge_gateway(payment_id)
                self.state_store.save_state(payment_id, {
                    'current_stage': 'processing',
                    'status': 'completed'
                })

            if start_stage <= 'notification':
                self.send_notification(payment_id)
                self.state_store.save_state(payment_id, {
                    'current_stage': 'notification',
                    'status': 'completed'
                })

            self.state_store.clear_state(payment_id)
            return {"status": "completed"}

        except Exception as e:
            # Save error state for retry
            self.state_store.save_state(payment_id, {
                'current_stage': start_stage,
                'status': 'failed',
                'error': str(e)
            })
            raise

16.5 Control-Plane Identification

In many systems, a dedicated component manages workflow execution. This component acts as the control plane, coordinating task scheduling, dependency resolution, and execution monitoring.

Control-plane signal types:

1. Orchestration frameworks:

# orchestration/engine.py
class OrchestrationEngine:
    """Control plane for workflow execution"""

    def __init__(self, workflow_store, task_scheduler):
        self.workflow_store = workflow_store
        self.task_scheduler = task_scheduler
        self.running_workflows = {}

    def submit_workflow(self, workflow_definition, input_data):
        """Submit workflow for execution"""
        workflow_id = generate_id()

        # Store workflow definition
        self.workflow_store.save_workflow(workflow_id, workflow_definition)

        # Create initial state
        self.workflow_store.create_workflow_run(workflow_id, input_data)

        # Schedule initial tasks
        self._schedule_ready_tasks(workflow_id)

        return workflow_id

    def _schedule_ready_tasks(self, workflow_id):
        """Schedule tasks whose dependencies are satisfied"""
        workflow = self.workflow_store.get_workflow(workflow_id)
        state = self.workflow_store.get_workflow_state(workflow_id)

        ready_tasks = self._get_ready_tasks(workflow, state)

        for task in ready_tasks:
            self.task_scheduler.schedule_task(
                workflow_id=workflow_id,
                task_id=task.id,
                task_definition=task
            )

    def handle_task_completion(self, workflow_id, task_id, result):
        """Handle task completion"""
        # Update task status
        self.workflow_store.update_task_status(
            workflow_id, task_id, 'success'
        )

        # Save task result
        self.workflow_store.save_task_result(task_id, result)

        # Schedule next tasks
        self._schedule_ready_tasks(workflow_id)

        # Check if workflow complete
        if self._is_workflow_complete(workflow_id):
            self.workflow_store.mark_workflow_complete(
                workflow_id, 'success'
            )

    def handle_task_failure(self, workflow_id, task_id, error):
        """Handle task failure"""
        # Update task status
        self.workflow_store.update_task_status(
            workflow_id, task_id, 'failed'
        )

        # Check retry policy
        if self._should_retry(workflow_id, task_id):
            self._retry_task(workflow_id, task_id)
        else:
            # Mark workflow as failed
            self.workflow_store.mark_workflow_complete(
                workflow_id, 'failed'
            )

16.7 Chapter Summary

  • Workflows define structured execution order and dependencies between tasks
  • Workflows encode operational logic for how capabilities are realized over time
  • Workflow types include Pipeline (sequential), DAG (graph), Schedule (time-based), and Event (reactive)
  • Workflow signals appear in DAG definitions, pipeline configurations, orchestration scripts, workflow frameworks, and scheduled jobs
  • Triggers initiate workflow execution and include cron schedules, event subscriptions, CI/CD triggers, queue messages, and CLI commands
  • State stores track workflow progress, task status, retries, and failure handling
  • State stores include metadata databases, execution logs, checkpoint mechanisms, and result storage
  • Execution semantics describe how tasks transition between states and how failures are handled
  • Control-plane components manage workflow execution through orchestration frameworks, scheduler services, pipeline controllers, and workflow managers
  • Control-plane identification reveals which components coordinate distributed execution
  • Workflow graphs represent tasks, dependencies, and relationships between workflows, tasks, and components
  • Workflow graphs reveal component participation, control-plane distribution, and workflow complexity
  • Workflows represent the control plane architecture of the system
