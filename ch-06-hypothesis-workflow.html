8  Hypothesis Workflow

Blueprint extraction cannot operate as a purely deterministic transformation from repository artifacts to architectural structures. Repositories expose signals about system structure, but they rarely encode architecture explicitly. The connection between implementation artifacts and architectural meaning must be constructed through interpretation.

Extraction must therefore proceed through hypothesis formation and validation. This workflow transforms raw repository observations into structured architectural knowledge through systematic stages of interpretation, evidence gathering, and validation.

The process follows a structured workflow:

signal → hypothesis → evidence → mapping → blueprint draft → review

Each stage refines the interpretation of repository artifacts and progressively constructs the architectural representation. This workflow ensures that extraction remains evidence-driven, auditable, and improvable.

8.1 Signal Detection

The extraction process begins with signal detection—identifying observable artifacts within the repository that may indicate the presence of architectural structure.

What signals are:

Signals are observable artifacts that indicate the possible existence of architectural elements. They are the first layer of information extracted from repositories and the foundation for all subsequent interpretation.

Signal examples:

  • Directory structures: services/, workers/, pipelines/ suggesting component boundaries
  • Dependency manifests: package.json, requirements.txt, go.mod suggesting capabilities
  • Entrypoint files: main.py, server.ts, cmd/root.go suggesting component roles
  • Configuration files: config.yaml, docker-compose.yml, k8s/*.yaml suggesting operational structure
  • Route definitions: HTTP routes, API endpoints, handlers suggesting interfaces
  • Workflow definitions: DAG specs, pipeline configurations suggesting workflows
  • Migration files: Schema DDL, migration scripts suggesting data entities

Signal characteristics:

  • Observable: Signals can be detected through automated scanning
  • Indirect: Signals do not directly define architecture—they hint at structure
  • Weak: No single signal is sufficient to determine architectural structure
  • Ambiguous: The same signal may support multiple architectural interpretations
  • Diverse: Signals originate from many types of repository artifacts

Signal function in extraction:

Signals function as architectural hints—observations that trigger hypothesis formation. For example:

  • A directory named workers/ signals that background processing components may exist
  • Route definitions signal that API interfaces may exist
  • Migration files signal that data entities may exist
  • DAG specifications signal that workflows may exist

Signal detection produces a catalog of observations about the repository. These observations serve as the input for hypothesis formation.

8.2 Hypothesis Formation

Signals lead to the generation of architectural hypotheses. A hypothesis is a provisional interpretation of repository structure in terms of the system ontology.

What hypotheses are:

A hypothesis is a candidate architectural interpretation that proposes that a specific architectural element (component, interface, workflow, data entity) exists based on observed signals.

Hypothesis examples:

  • Worker component hypothesis: Signals suggesting a directory named workers/, queue dependencies, and background task frameworks may lead to the hypothesis that a worker component exists
  • API interface hypothesis: Route definitions, controller implementations, and OpenAPI specs may lead to the hypothesis of an HTTP API interface
  • Data entity hypothesis: Migration files, ORM models, and schema definitions may lead to the hypothesis of a data entity
  • Workflow hypothesis: DAG specifications, pipeline configurations, and scheduler definitions may lead to the hypothesis of a workflow

Hypothesis characteristics:

  • Provisional: Hypotheses are not yet confirmed as correct
  • Interpretive: Hypotheses represent architectural meaning, not just observations
  • Ontology-aligned: Hypotheses are expressed in terms of system ontology entities (components, interfaces, workflows, data)
  • Testable: Hypotheses can be validated or refuted through evidence collection

Hypothesis formation process:

Hypothesis formation involves:

  1. Pattern recognition: Identifying patterns in signals that suggest architectural structure
  2. Interpretation: Translating patterns into architectural terms using the system ontology
  3. Contextualization: Considering how hypotheses relate to each other and to the overall system
  4. Scoring: Assessing the initial plausibility of each hypothesis based on signal strength

The role of hypotheses:

Hypotheses serve as the bridge between raw observations and validated architectural understanding. They focus the extraction process by identifying specific architectural questions that need to be answered through evidence collection.

Without hypotheses, extraction would lack direction—collecting evidence without knowing what to look for. With hypotheses, extraction becomes a targeted investigation of specific architectural claims.

8.3 Evidence Collection

Once a hypothesis is formed, the extraction process gathers evidence that supports or contradicts the hypothesis. Evidence consists of repository artifacts that directly demonstrate the presence of the architectural structure being proposed.

What evidence is:

Evidence is the traceable basis for architectural claims. Unlike signals, which are weak indicators, evidence provides verifiable justification for blueprint elements.

Evidence examples:

  • Worker component evidence:
    • Code files implementing worker processes
    • Configuration entries defining message queues
    • OpenAPI specifications describing interface contracts
    • Workflow definitions in scheduling systems
  • API interface evidence:
    • Route handler implementations
    • Request/response schemas
    • Integration tests verifying interface behavior
    • Documentation describing API contracts
  • Data entity evidence:
    • Migration files defining table structures
    • ORM models reflecting database schemas
    • Schema validation rules
    • Data access layer implementations

Evidence characteristics:

  • Concrete: Evidence references specific, locatable repository artifacts
  • Traceable: Evidence can be followed back to its source
  • Verifiable: Evidence can be inspected and validated
  • Fragmentary: Evidence captures relevant portions of artifacts, not entire files
  • Multiple: Multiple evidence objects may support a single hypothesis

Evidence collection process:

Evidence collection involves:

  1. Artifact search: Locating repository artifacts that relate to the hypothesis
  2. Fragment extraction: Extracting relevant portions of those artifacts
  3. Classification: Categorizing evidence by type and strength
  4. Association: Linking evidence to the hypotheses it supports or contradicts

Evidence outcomes:

Evidence collection can produce several outcomes:

  • Confirmation: Strong evidence confirms the hypothesis
  • Refutation: Contradictory evidence refutes the hypothesis
  • Ambiguity: Conflicting evidence makes the hypothesis uncertain
  • Insufficiency: Insufficient evidence prevents validation

Evidence objects become the traceable basis for architectural claims and are included in the evidence bundle for review and auditability.

8.4 Ontology Mapping

When sufficient evidence exists, the hypothesis can be translated into system ontology entities. This stage maps repository-derived structures to the canonical architectural vocabulary defined by the system ontology.

What ontology mapping is:

Mapping is the process of translating evidence-supported interpretations into standardized architectural entities and relationships. This translation ensures consistency across repositories and enables automated analysis.

Mapping examples:

  • Worker implementation → Component entity: Code implementing worker processes maps to a Component entity of type “Worker”
  • Route definitions → Interface entity: HTTP route definitions map to an Interface entity of type “API”
  • Migration files → Data entity: Schema migrations map to a Data entity representing a table or collection
  • Pipeline definitions → Workflow entity: Pipeline configurations map to a Workflow entity representing orchestrated execution

Mapping characteristics:

  • Canonical: Mapping produces standardized entity types defined in the system ontology
  • Interpretable: Mapping rules are explainable and reviewable
  • Deterministic: Where possible, the same input produces the same output
  • Evidence-linked: Mapped entities reference the evidence that supports them

Mapping process:

Ontology mapping involves:

  1. Entity identification: Determining what type of entity the evidence supports (component, interface, data, workflow)
  2. Entity construction: Creating the entity with appropriate attributes and metadata
  3. Relationship identification: Determining how the entity relates to other entities
  4. Evidence association: Linking the entity to the evidence objects that support it

Mapping outputs:

Mapping produces two core structures:

  • Entity graph: Nodes representing architectural entities (components, interfaces, data, workflows)
  • Relation graph: Edges representing relationships between entities (exposes, owns, triggers, depends-on)

These graphs form the structural foundation of the blueprint and enable further analysis, visualization, and pattern discovery.

8.5 Blueprint Draft Generation

After mapping, the extracted entities are assembled into a blueprint draft. The draft populates the Blueprint Kernel slots with the identified entities and relationships.

What blueprint drafts are:

A blueprint draft is the initial architectural representation produced by the extraction pipeline. It represents the system’s architecture as interpreted from repository evidence, but has not yet undergone validation or review.

Draft population process:

Blueprint draft generation involves:

  1. Slot population: Populating each kernel slot (Goal, Capabilities, Components, Interfaces, Data, Workflows) with mapped entities
  2. Relationship assembly: Connecting entities through relationships identified during mapping
  3. Confidence assignment: Assigning confidence levels based on evidence strength and consistency
  4. Gap identification: Marking kernel fields as unknown when evidence is insufficient

Draft characteristics:

  • Structured: The draft follows the Blueprint Kernel structure
  • Evidence-backed: Every entity references supporting evidence
  • Confidence-annotated: Every field includes a confidence classification
  • Complete or partial: The draft may be fully populated or have unknown fields

Draft limitations:

At this stage, the blueprint draft is not yet considered authoritative because:

  • Uncertainty may remain in low-confidence fields
  • Additional validation is required to ensure structural correctness
  • Human review is needed to validate interpretations and resolve ambiguities

The blueprint draft represents a structured architectural interpretation of the repository, but requires validation and review before becoming a usable blueprint.

8.6 Modeling Uncertainty

Architectural reconstruction inevitably involves uncertainty. Repositories rarely provide complete, unambiguous information about architecture. The extraction process must therefore explicitly represent uncertainty rather than conceal it.

Two mechanisms for representing uncertainty:

8.6.1 Confidence

Confidence values estimate how strongly evidence supports an extracted architectural element.

Confidence sources:

  • Evidence strength: How directly artifacts express architectural structure
  • Evidence quantity: Number of independent artifacts supporting the interpretation
  • Evidence consistency: Whether different artifacts support the same interpretation
  • Pattern alignment: Whether observed structure matches known architectural patterns

Confidence states:

  • Observed: Directly supported by explicit repository artifacts
  • Inferred: Derived from indirect signals and interpretation
  • Unknown: Insufficient evidence exists to determine presence or structure

8.6.2 Unknown Fields

Unknown fields represent architectural slots for which insufficient evidence exists.

When unknowns are appropriate:

  • Repository artifacts do not provide enough information
  • Signals are too weak or ambiguous to support interpretation
  • Architectural elements exist outside the repository (runtime, operations, constraints)

The importance of explicit unknowns:

Treating unknown values as legitimate outcomes prevents blueprint datasets from accumulating inaccurate assumptions. Unknown fields:

  • Maintain integrity: Blueprints remain honest about what is known versus unknown
  • Guide investigation: Unknown fields identify areas requiring additional research or documentation
  • Enable validation: Unknown fields can be filled later when additional evidence becomes available
  • Prevent overconfidence: Unknown fields prevent over-optimistic interpretations

Uncertainty as a feature, not a bug:

Explicit uncertainty representation is a strength, not a weakness, of the extraction methodology. It enables:

  • Honest representation: Blueprints accurately reflect knowledge gaps
  • Focused review: Review effort can be directed to uncertain, high-impact fields
  • Continuous improvement: Unknown fields identify opportunities to improve extraction
  • Quality assessment: Evidence ratios and unknown counts measure extraction quality

8.7 Human-in-the-Loop

Although extraction pipelines can automate many stages, architectural interpretation cannot be fully automated. Human review is therefore a formal component of the extraction pipeline, not an optional afterthought.

Why human review is necessary:

  • Architectural judgment: Distinguishing correct from incorrect interpretations often requires domain knowledge
  • Ambiguity resolution: When evidence is conflicting or insufficient, human judgment is necessary
  • Contextual understanding: Humans can bring external knowledge about system intent, organizational structure, and operational context
  • Quality assurance: Human validation provides the final quality gate before blueprints enter production use

Human review activities:

Reviewers inspect the blueprint draft together with supporting materials:

  • Blueprint draft: The proposed architectural representation
  • Evidence bundle: All evidence objects supporting the draft
  • Extraction report: Diagnostics, gaps, conflicts, and confidence summaries

Review outcomes:

Reviewers may:

  • Confirm elements that accurately reflect the system
  • Adjust mappings that misinterpret artifacts
  • Add missing information not captured by automated extraction
  • Downgrade confidence for elements with insufficient support
  • Mark fields as unknown when evidence is absent

Critical review scenarios:

Human oversight is particularly important in cases where:

  • Evidence is weak or ambiguous
  • Repository conventions deviate from common patterns
  • Architectural intent is not clearly expressed in artifacts
  • Conflicting evidence requires judgment to resolve
  • Unknown fields require investigation to determine if information exists elsewhere

Hybrid approach:

The extraction pipeline integrates human review as a first-class stage rather than an optional step. Automation performs:

  • Signal detection
  • Evidence aggregation
  • Structural reconstruction

Human reviewers perform:

  • Interpretation validation
  • Ambiguity resolution
  • Architectural refinement

This hybrid approach preserves both scalability and architectural accuracy.

8.8 Extraction as Structured Interpretation

The hypothesis workflow transforms blueprint extraction from ad-hoc reverse engineering into a disciplined engineering process.

The workflow in action:

  1. Signals provide structural hints from repository artifacts
  2. Hypotheses interpret those hints in architectural terms
  3. Evidence validates or refutes interpretations
  4. Mapping translates validated evidence into ontology structures
  5. Drafts assemble structures into blueprints
  6. Uncertainty modeling makes unknowns explicit rather than hidden
  7. Human review validates interpretations and refines accuracy

The discipline payoff:

This structured approach ensures that extraction produces:

  • Evidence-backed architectures: Every claim supported by traceable artifacts
  • Auditable processes: Every step documented and reviewable
  • Reproducible results: Consistent methodology yields consistent outputs
  • Improvable systems: Each component can be enhanced independently
  • Honest representations: Uncertainty explicitly acknowledged rather than concealed

The hypothesis workflow is therefore not merely a technical process but a methodological foundation for reliable architectural understanding at scale.

8.9 Chapter Summary

  • Blueprint extraction requires hypothesis formation and validation rather than deterministic transformation
  • The hypothesis workflow follows: signal → hypothesis → evidence → mapping → draft → review
  • Signal detection identifies observable artifacts that may indicate architectural structure
  • Hypothesis formation interprets signals as provisional architectural claims
  • Evidence collection gathers concrete artifacts that support or refute hypotheses
  • Ontology mapping translates evidence into canonical architectural entities and relationships
  • Blueprint draft generation assembles entities into the kernel structure
  • Uncertainty is modeled through confidence annotations and explicit unknown fields
  • Human review is an essential stage for validation, ambiguity resolution, and quality assurance
  • The structured workflow ensures evidence-backed, auditable, and reproducible extraction
