13  Component Discovery

Components represent the structural units of responsibility within a system. They implement capabilities, expose interfaces, operate on data, and participate in workflows. During blueprint extraction, identifying components is one of the most critical steps because components define the structural decomposition of the system.

However, repositories rarely declare components explicitly. Component discovery therefore relies on identifying structural signals that indicate the presence of distinct execution or responsibility units. The goal is to produce a component inventory: a structured list of architectural components inferred from repository evidence.

Component discovery establishes the structural framework upon which the remainder of the blueprint is constructed. Once components are identified, subsequent extraction stages can determine interfaces, data ownership, and workflows.

13.1 What Is a Component?

Before discovering components, we must define what a component is in the context of blueprint extraction. Components are not arbitrary code units; they represent architecturally meaningful structural units.

Component characteristics:

1. Structural unit of responsibility:

A component represents a cohesive set of responsibilities that are logically grouped together. These responsibilities typically address a specific domain of functionality.

Example:

component:
  id: component.payment_service
  responsibilities:
    - "Process payment transactions"
    - "Validate payment requests"
    - "Interact with payment gateways"
    - "Maintain payment records"

2. Exposes interfaces:

Components expose interfaces through which they interact with external actors or other components.

Example:

component:
  id: component.payment_service
  interfaces:
    - id: interface.http_api
      type: API
      protocol: HTTP
    - id: interface.payment_gateway
      type: RPC
      protocol: gRPC

3. Operates on data:

Components own, read, or modify data entities as part of their responsibilities.

Example:

component:
  id: component.payment_service
  data:
    - id: data.payments
      relationship: owns
    - id: data.payment_logs
      relationship: reads

4. Participates in workflows:

Components execute tasks within orchestrated workflows or respond to events.

Example:

component:
  id: component.payment_service
  workflows:
    - id: workflow.payment_processing
      role: primary
    - id: workflow.settlement
      role: participant

Component vs. module:

A component is not merely a code module. While modules organize code within a repository, components represent architectural boundaries that may span multiple modules or be contained within a single module.

# Module: code organization
module:
  id: payment.models
  type: Python module
  contains: [Payment, Transaction, Refund]

# Component: architectural boundary
component:
  id: component.payment_service
  type: Service
  contains:
    - payment.models
    - payment.routes
    - payment.services
  responsibilities: [payment processing]

Component types:

The component ontology defines several archetypes that appear frequently in software systems:

component_types:
  Service:
    role: Externally reachable application unit
    examples: [API services, web applications, microservices]
    typical_interfaces: [HTTP API, GraphQL API, gRPC]

  Worker:
    role: Asynchronous processing unit
    examples: [Background jobs, message consumers, task processors]
    typical_interfaces: [Queue, Topic, Event stream]

  Scheduler:
    role: Temporal coordination component
    examples: [Cron schedulers, workflow engines, job orchestrators]
    typical_interfaces: [Temporal trigger, DAG definition]

  Plugin:
    role: Extensibility mechanism
    examples: [Dynamic modules, extensions, add-ons]
    typical_interfaces: [Plugin API, registration mechanism]

13.2 Component Discovery Strategy

Component discovery proceeds through a systematic strategy that combines signal detection, pattern matching, and evidence aggregation.

Discovery pipeline:

component_discovery_pipeline:
  step_1_signal_detection:
    action: "Scan repository for component signals"
    outputs:
      - structure_signals
      - dependency_signals
      - entrypoint_signals
      - config_signals
      - deploy_signals

  step_2_pattern_matching:
    action: "Match signals to component archetypes"
    outputs:
      - component_candidates
      - candidate_confidence_scores
      - supporting_evidence

  step_3_evidence_aggregation:
    action: "Aggregate evidence for each candidate"
    outputs:
      - component_evidence_sets
      - evidence_conflict_detection
      - aggregated_confidence_scores

  step_4_boundary_inference:
    action: "Infer component boundaries"
    outputs:
      - component_boundaries
      - responsibility_mapping
      - interface_associations

  step_5_inventory_generation:
    action: "Generate final component inventory"
    outputs:
      - component_inventory
      - confidence_assignments
      - evidence_references

Discovery principles:

1. Multi-signal approach:

No single signal is sufficient for component discovery. Multiple signals must converge on the same component interpretation.

multi_signal_detection:
  component: component.payment_worker

  signals_detected:
    - signal: queue_dependency
      strength: strong
    - signal: worker_directory
      strength: moderate
    - signal: task_definition
      strength: strong
    - signal: deployment_config
      strength: strong

  interpretation:
    type: WorkerComponent
    confidence: 0.92
    rationale: "Strong convergence of worker-related signals"

2. Boundary detection:

Component boundaries are inferred from structural and operational signals rather than explicitly declared.

boundary_detection:
  detected_boundary: api/ directory
  supporting_signals:
    - signal: directory_isolation
    - signal: independent_routes
    - signal: separate_config
    - signal: dedicated_entrypoint
  confidence: 0.88
  rationale: "Structural isolation with independent execution"

3. Responsibility inference:

Component responsibilities are derived from associated artifacts and interfaces.

responsibility_inference:
  component: component.payment_service
  detected_responsibilities:
    - source: HTTP routes
      responsibility: "Process payment requests"
      confidence: 0.95
    - source: Database operations
      responsibility: "Manage payment records"
      confidence: 0.90
    - source: Gateway integration
      responsibility: "Process payments"
      confidence: 0.88

@app.get("/payments/{payment_id}")

@app.post("/payments/{payment_id}/refund")
def refund_payment(payment_id: str, amount: float):
    return payment_service.refund(payment_id, amount)

13.8 Inferring Component Boundaries

Component boundaries are rarely defined explicitly in repositories. Instead, boundaries must be inferred from structural and behavioral signals.

Boundary inference indicators:

1. Directory isolation:

project/
├── api/              # API service boundary
│   ├── routes.py
│   ├── controllers.py
│   └── services.py
├── workers/          # Worker boundary
│   ├── payment_worker.py
│   └── notification_worker.py
└── shared/           # Shared utilities (not a component)
    ├── utils.py
    └── models.py
detected_boundary:
  component: component.api_service
  boundary_type: directory_isolation
  evidence:
    - signal: "api/ directory exists"
    - signal: "contains routes.py, controllers.py, services.py"
    - signal: "separate from workers/ directory"
  confidence: 0.85

2. Independent entrypoints:

13.10 Chapter Summary

  • Components represent structural units of responsibility that implement capabilities and expose interfaces
  • Components differ from code modules; they represent architectural boundaries, not just code organization
  • Component discovery uses a systematic pipeline: signal detection → pattern matching → evidence aggregation → boundary inference → inventory generation
  • Four main component archetypes: Service (externally reachable), Worker (asynchronous processing), Scheduler (time-based coordination), Plugin (extensibility)
  • Service components are identified by HTTP frameworks, server initialization, API controllers, startup scripts, and container port exposure
  • Worker components are identified by queue consumers, background task frameworks, job libraries, queue configuration, and worker startup commands
  • Scheduler components are identified by cron configurations, DAG definitions, scheduling frameworks, and task orchestration
  • Plugin components are identified by plugin directories, dynamic module loading, registration mechanisms, and configuration-driven discovery
  • Component responsibilities are inferred from API controllers, message handlers, database migrations, and workflow definitions
  • Component boundaries are inferred from directory isolation, independent entrypoints, separate deployment, dedicated processes, and distinct dependencies
  • The component inventory is the final output, containing structured component definitions with responsibilities, interfaces, data, workflows, confidence, and evidence references
  • Component inventory forms the foundation of the component layer within the blueprint kernel
