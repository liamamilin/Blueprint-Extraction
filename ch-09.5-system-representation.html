12  System Representation: Making Systems Observable

12.1 The Missing Layer in Extraction

Extraction does not fail because signals are insufficient.

Extraction fails because the target is undefined.

A repository is not a system.
A system is not code.

A system is a representation.

If this representation is not made explicit:

  • component boundaries drift

  • interfaces become ambiguous

  • workflows collapse into implementation detail

Before ontology mapping, the system must be reconstructed in a form that is:

  • observable

  • typed

  • decomposable

12.2 A Minimal System Model

A system is defined as:

System = {
  Subsystems,
  Interfaces,
  Data Models,
  Transformations,
  Flows
}

This is not conceptual.
This is operational.

A system must be representable as:

a directed graph of typed transformations over data

12.3 Subsystems as Transformation Units

A subsystem is not a file or directory.

A subsystem exists only when a transformation exists.

Subsystem = (Input, Output, Transformation)

A valid subsystem must:

  • consume typed input

  • produce typed output

  • perform a stable transformation

12.4 Interfaces as Contracts

Interfaces are not syntax.
They are contracts.

{
  "input_schema": { "text": "string" },
  "output_schema": { "tokens": ["string"] }
}

An interface defines:

  • what enters

  • what exits

  • what is guaranteed

12.5 Data Models as Carriers of Meaning

Data is not raw.

Data is structured state.

text      : str
tokens    : List[str]
counts    : Dict[str, int]

Data models are the only objects that move through the system.

12.6 Transformations as Observable Processes

A transformation is defined as:

output = f(input, state)

For extraction, this is insufficient.

We require:

transformations must be observable

Which means:

  • intermediate states must be visible

  • input/output must be logged

  • execution must be replayable

12.7 Flows as Composition

A system is not a set.
It is a composition.

A → B → C

Or:

output = f3(f2(f1(input)))

Flow defines:

  • execution order

  • dependency structure

  • system topology

12.8 Making a System Observable

We do not understand systems by reading code.

We understand systems by observing data movement.

The goal is:

to see how data evolves across transformations

12.9 Instrumented Example: Data Transformation Trace

This example is intentionally minimal.

Its purpose is to expose:

  • subsystem boundaries

  • interface contracts

  • transformation steps

  • data evolution

from typing import List, Dict, Any


def trace(step: str, input_data: Any, output_data: Any):
    print(f"\n[TRACE] {step}")
    print(f"  input  = {input_data}")
    print(f"  output = {output_data}")


class Tokenizer:
    def run(self, text: str) -> List[str]:
        tokens = text.lower().split()
        trace("Tokenizer", text, tokens)
        return tokens


class Counter:
    def run(self, tokens: List[str]) -> Dict[str, int]:
        result = {}
        for t in tokens:
            result[t] = result.get(t, 0) + 1
        trace("Counter", tokens, result)
        return result


class Formatter:
    def run(self, counts: Dict[str, int]) -> str:
        output = "\n".join(f"{k}: {v}" for k, v in counts.items())
        trace("Formatter", counts, output)
        return output


class TextAnalysisSystem:

    def __init__(self):
        self.tokenizer = Tokenizer()
        self.counter = Counter()
        self.formatter = Formatter()

    def run(self, text: str) -> str:
        tokens = self.tokenizer.run(text)
        counts = self.counter.run(tokens)
        result = self.formatter.run(counts)
        return result


if __name__ == "__main__":
    system = TextAnalysisSystem()

    input_text = "AI is the future and AI is powerful"

    result = system.run(input_text)

    print("\n=== FINAL OUTPUT ===")
    print(result)

12.10 What Becomes Visible

Running this system reveals:

  • each subsystem boundary

  • each input/output transition

  • the exact shape of data at every step

This enables:

  • reconstruction of system graph

  • validation of interfaces

  • detection of hidden transformations

12.11 System Representation (Derived)

From execution, we reconstruct:

Tokenizer:
  input  = text
  output = tokens

Counter:
  input  = tokens
  output = counts

Formatter:
  input  = counts
  output = formatted_text

Flow:

Tokenizer → Counter → Formatter

12.12 LLM-Assisted Extraction Prompts

The following prompts are used to reconstruct system representation from code, logs, or repositories.

They are not generic prompts.
They enforce structure.

12.12.1 Subsystem Extraction Prompt

You are given code or execution traces.

Identify transformation units (subsystems).

A subsystem must:
- consume input
- produce output
- perform a distinct transformation

Return a list of subsystems:

For each subsystem:
- name
- input (type + meaning)
- output (type + meaning)
- responsibility (what transformation it performs)

Do not describe files or classes.
Only identify transformation boundaries.

12.12.2 Interface Extraction Prompt

Given a list of subsystems and their transformations:

Infer the interfaces between them.

For each interface:
- source subsystem
- target subsystem
- data passed
- input_schema
- output_schema

Schemas must be typed and explicit.

Do not use vague descriptions like "data" or "result".

12.12.3 Data Model Extraction Prompt

Analyze the system and list all data models.

For each data model:
- name
- type (primitive / structured)
- fields (if structured)
- where it is created
- where it is consumed

Only include data that flows between subsystems.
Ignore local temporary variables unless they cross boundaries.

12.12.4 Transformation Extraction Prompt

For each subsystem:

Describe the transformation as:

output = f(input, state)

Include:
- input → output mapping
- key logic steps
- whether transformation is deterministic or stateful

Focus on how data changes, not how code is written.

12.12.5 Flow Extraction Prompt

Given subsystems and interfaces:

Construct the system flow.

Return:
- ordered execution chain
- dependencies between subsystems
- whether flow is linear, branching, or cyclic

Output as a directed graph description.

12.12.6 Observability Prompt (关键)

Given execution logs or traces:

Reconstruct the system step-by-step.

For each step:
- subsystem name
- input data
- output data

Then infer:
- hidden transformations
- mismatched interfaces
- implicit assumptions

Your goal is to make the system fully observable.

12.13 Implication

Extraction is not parsing.

Extraction is reconstruction.

A system is not discovered in code.

It is reconstructed from:

  • transformations

  • data movement

  • observable behavior

12.14 Final Definition

A system is:

a directed graph of observable, typed transformations over data,
where each node consumes input, produces output,
and participates in a traceable flow.

12.15 Transition

With system representation made explicit:

  • ambiguity becomes measurable

  • mapping becomes grounded

  • blueprint generation becomes stable

We can now proceed to ontology mapping,
where structure becomes computable.
