19  Blueprint Pack Output Standard

Blueprint extraction does not end when a kernel has been filled or views have been generated. The extracted result must be emitted in a form that is reviewable, versionable, and operationally usable.

For this reason, this volume defines the Blueprint Pack as the standard output unit of extraction. A blueprint pack is not merely a blueprint file; it is a structured output package that contains the blueprint itself together with the supporting materials required for audit, review, and iterative refinement.

The pack standard has three objectives: preserve the extracted architecture in a canonical form, preserve the evidence and review context that justify it, and make changes easy to inspect through normal version-control workflows. A blueprint pack is therefore the portable, reviewable artifact produced by the extraction pipeline.

19.1 Pack as the Output Unit

The extraction pipeline produces more than one kind of result. It produces a kernel-level architectural representation, one or more projected views, evidence references, extraction diagnostics, and review metadata. These outputs must remain connected.

Why a pack structure matters:

pack_rationale:
  connection_requirement:
    problem: "Multiple extraction outputs must remain connected"
    without_pack:
      - blueprint: "Stored without evidence context"
        consequence: "Auditability lost"
      - evidence: "Stored without blueprint"
        consequence: "Cannot be used as architectural knowledge"
      - metadata: "Stored in separate locations"
        consequence: "Difficult to correlate"

    with_pack:
      - blueprint: "Packaged with evidence"
        consequence: "Fully auditable"
      - evidence: "Packaged with blueprint"
        consequence: "Usable architectural knowledge"
      - metadata: "Packaged together"
        consequence: "Easy to correlate"

  version_friendliness:
    problem: "Extraction must work with Git-based workflows"
    without_pack:
      - format: "Opaque binary or complex format"
        consequence: "Diff reviews difficult"
      - structure: "Unstructured or inconsistent"
        consequence: "Meaningful changes hard to identify"

    with_pack:
      - format: "YAML/JSON text files"
        consequence: "Git diff works naturally"
      - structure: "Standardized directory layout"
        consequence: "Changes easily located"
      - determinism: "Stable field ordering"
        consequence: "Minimal noise in diffs"

Pack vs. simple blueprint file:

comparison:
  simple_blueprint_file:
    structure: "Single YAML/JSON file"
    contents:
      - "Kernel structure"
      - "View definitions"
    missing:
      - "Evidence references"
      - "Extraction diagnostics"
      - "Review metadata"
      - "Decision history"
    use_cases:
      - "Simple visualization"
      - "Final documentation"
    limitations:
      - "Not auditable"
      - "Not reviewable"
      - "Not debuggable"

  blueprint_pack:
    structure: "Directory with multiple files"
    contents:
      - "Blueprint (kernel + views)"
      - "Evidence bundle"
      - "Extraction report"
      - "Review metadata"
      - "Decision log"
    includes:
      - "Full evidence context"
      - "Extraction quality metrics"
      - "Review decisions"
      - "Change history"
    use_cases:
      - "Architectural analysis"
      - "Design review"
      - "Pattern discovery"
      - "Documentation"
    advantages:
      - "Fully auditable"
      - "Systematically reviewable"
      - "Easily debuggable"
      - "Version-control friendly"

19.2 Pack Directory Structure

A blueprint pack should use a stable and predictable directory layout. This structure ensures consistency across extraction runs and enables tooling to process packs programmatically.

Canonical pack structure:

blueprint_pack_structure:
  root: "blueprint-pack/"

  directories:
    - path: "/"
      description: "Pack root directory"

    - path: "/blueprint.yaml"
      description: "Canonical kernel-centered blueprint"
      required: true

    - path: "/views/"
      description: "Projected representations derived from kernel"
      required: true

    - path: "/views/architecture.yaml"
      description: "Architecture view (primary output)"
      required: true

    - path: "/views/product.yaml"
      description: "Product view (user-facing)"
      required: false

    - path: "/views/ops.yaml"
      description: "Operations view (operational structure)"
      required: false

    - path: "/views/security.yaml"
      description: "Security/trust view (boundaries and access)"
      required: false

    - path: "/evidence/"
      description: "Evidence indexes and supporting excerpts"
      required: true

    - path: "/evidence/index.yaml"
      description: "Catalog of all evidence objects"
      required: true

    - path: "/evidence/snippets/"
      description: "Artifact fragments for quick inspection"
      required: true

    - path: "/report/"
      description: "Extraction diagnostics and assessment artifacts"
      required: true

    - path: "/report/extraction-report.yaml"
      description: "Quality and uncertainty ledger"
      required: true

    - path: "/review/"
      description: "Human review outcomes and decision history"
      required: true

    - path: "/review/decision-log.yaml"
      description: "Architectural decision records (ADR)"
      required: true

    - path: "/review/status.yaml"
      description: "Current review state and maturity level"
      required: true

    - path: "/metadata/"
      description: "Pack-level metadata"
      required: true

    - path: "/metadata/pack-meta.yaml"
      description: "Source repository, extraction version, timestamps"
      required: true

Directory separation rationale:

separation_rationale:
  blueprint_yaml:
    location: "Root level"
    reason: "Primary output, easily accessible"
    access: "Direct reference in tools"

  views_directory:
    location: "Separate directory"
    reason: "Multiple views, organized together"
    access: "Through index or direct file access"

  evidence_directory:
    location: "Separate directory"
    reason: "Large volume, separate concern"
    access: "Through evidence index"

  report_directory:
    location: "Separate directory"
    reason: "Diagnostic information, separate from blueprint"
    access: "For quality assessment"

  review_directory:
    location: "Separate directory"
    reason: "Review metadata, separate from extraction output"
    access: "For governance and audit"

Example pack file tree:

blueprint-pack/
├── blueprint.yaml
├── views/
│   ├── architecture.yaml
│   ├── product.yaml
│   ├── ops.yaml
│   └── security.yaml
├── evidence/
│   ├── index.yaml
│   └── snippets/
│       ├── evidence-001.txt
│       ├── evidence-002.txt
│       └── ...
├── report/
│   └── extraction-report.yaml
├── review/
│   ├── decision-log.yaml
│   └── status.yaml
└── metadata/
    └── pack-meta.yaml

    deploy: 6
    structure: 5

    docs: 3

      type: code
      category: entrypoint
      strength: strong

19.5 Extraction Report

Every blueprint pack must contain an extraction report. The extraction report records what the extraction pipeline found, what it could not determine, and where uncertainty remains. It serves as the operational summary of the extraction run.

19.6 Review-Oriented Output

A blueprint pack is designed to support human review as a normal part of extraction. The output must make it easy for a reviewer to answer questions about what was extracted, what evidence supports it, and where uncertainty remains.

Review directory structure:

review_directory:
  decision_log:
    file: "review/decision-log.yaml"
    purpose: "Record architectural decisions made during review"
    format: "Architectural Decision Records (ADR)"

  status:
    file: "review/status.yaml"
    purpose: "Track current review state and maturity level"
    format: "Status tracking"

Decision log format:

decision_log:
  version: "1.0"
  last_updated: "2026-03-08T14:30:00Z"

  decisions:
    - id: adr-001
      title: "Architecture pattern selection"
      status: accepted
      date: "2026-03-08"
      context: |
        README describes microservices architecture, but code shows
        monolithic application with background workers.

      decision: |
        Treat system as monolithic with service/worker separation
        rather than true microservices. This aligns with implementation
        evidence which is stronger than documentation.

      consequences:
        - "Architecture view shows service/worker separation"
        - "Deployment shows single application with workers"
        - "No cross-service network boundaries identified"

      alternatives_considered:
        - "True microservices (rejected: no evidence)"
        - "Modular monolith (considered: partial evidence)"

      reviewer: "architect@example.com"
      confidence: 0.85

    - id: adr-002
      title: "Goal statement validation"
      status: accepted
      date: "2026-03-08"
      context: |
        Goal is inferred from capabilities rather than explicitly
        documented in repository.

      decision: |
        Accept inferred goal: "Process payment transactions and manage
        refunds for e-commerce platforms". This aligns with capabilities
        and is reasonable, but confidence remains lower due to lack of
        explicit documentation.

      consequences:
        - "Goal marked as inferred with 0.70 confidence"
        - "Product view may be incomplete"
        - "Recommend explicit documentation in future"

      alternatives_considered:
        - "Mark as unknown (rejected: sufficient supporting evidence)"

      reviewer: "product_manager@example.com"
      confidence: 0.70

    - id: adr-003
      title: "Payment worker component validation"
      status: confirmed
      date: "2026-03-08"
      context: |
        Payment worker identified from Celery task definitions,
        queue dependencies, and deployment configuration.

      decision: |
        Confirm component.payment_worker as legitimate worker
        component. Strong evidence supports this interpretation:
        - Celery task definitions (strong)
        - Queue dependency (strong)
        - Worker deployment config (strong)

      consequences:
        - "Component marked as observed with 0.92 confidence"
        - "No changes needed"

      reviewer: "architect@example.com"
      confidence: 0.92

19.7 Diff-Friendly Representation

Because blueprint packs are expected to evolve with repositories over time, their contents must be diff-friendly. This is a critical requirement for operational usability.

Diff-friendly requirements:

diff_friendly_requirements:
  stable_field_ordering:
    requirement: "Fields must appear in consistent order"
    reason: "Prevents spurious diffs from reordering"
    implementation: "YAML preserves insertion order"

  deterministic_serialization:
    requirement: "Same content produces identical output"
    reason: "Prevents cosmetic changes in diffs"
    implementation: "No random elements or timestamps"

  one_logical_object_per_file:
    requirement: "Each file contains one logical object"
    reason: "Changes isolated to relevant files"
    implementation: "Separate files for different concerns"

  minimal_regenerated_metadata:
    requirement: "Avoid regenerated timestamps in output"
    reason: "Prevents noise in every diff"
    implementation: "Use extraction_run_id, not timestamps in content"

  explicit_identifiers:
    requirement: "Every entity has stable identifier"
    reason: "Enables tracking across versions"
    implementation: "Hash-based or sequential IDs"

Diff-friendly example:

before: "Version 1.0"
blueprint:
  kernel:
    Capabilities:
      - id: cap.create_payment
        name: "Create Payment"
        confidence: 0.95
      - id: cap.process_payment
        name: "Process Payment"
        confidence: 0.90

after: "Version 1.1"
blueprint:
  kernel:
    Capabilities:
      - id: cap.create_payment
        name: "Create Payment"
        confidence: 0.95
      - id: cap.process_payment
        name: "Process Payment"
        confidence: 0.90
      - id: cap.send_notification
        name: "Send Notification"
        confidence: 0.88

git_diff:
  added_lines: 4
  removed_lines: 0
  noise: 0
  clarity: "High - only shows actual change"

Diff-unfriendly example:

before: "Version 1.0"
blueprint:
  kernel:
    Capabilities:
      - id: cap.create_payment

        confidence: 0.95
        generated_at: "2026-03-08T10:30:00Z"
      - id: cap.process_payment
        name: "Process Payment"
        confidence: 0.90
        generated_at: "2026-03-08T10:30:01Z"

after: "Version 1.1"
blueprint:
  kernel:
    Capabilities:
      - id: cap.create_payment
        name: "Create Payment"
        confidence: 0.95
        generated_at: "2026-03-08T11:00:00Z"
      - id: cap.process_payment
        name: "Process Payment"
        confidence: 0.90
        generated_at: "2026-03-08T11:00:01Z"
      - id: cap.send_notification
        name: "Send Notification"
        confidence: 0.88

git_diff:

  noise: 8
  clarity: "Low - includes timestamp noise"

19.8 Git Review Compatibility

The blueprint pack standard is intentionally designed to fit ordinary version-control practices. A reviewer should be able to inspect a blueprint update through a normal Git diff and understand what changed.

Git review workflow:

git_review_workflow:
  scenario: "Repository updated, blueprint regenerated"

  git_diff_interpretation:
    added_capability:
      change: "New capability added"
      file: "blueprint.yaml"
      lines: "4 lines added"
      meaning: "New feature detected in code"

    component_boundary_change:
      change: "Component responsibility modified"
      file: "views/architecture.yaml"
      lines: "2 lines modified"
      meaning: "Architectural change in component"

    evidence_addition:
      change: "New evidence collected"
      file: "evidence/index.yaml"
      lines: "3 lines added"
      meaning: "Better support for existing interpretation"

    confidence_increase:
      change: "Confidence score improved"
      file: "blueprint.yaml"
      lines: "1 line modified"
      meaning: "Additional evidence strengthens interpretation"

  review_questions_answerable:
    - question: "What kernel fields changed?"
      answer: "Check blueprint.yaml diff"
      effort: "Trivial"

    - question: "Which entities were added or removed?"
      answer: "Check entity IDs in diff"
      effort: "Trivial"

    - question: "Did confidence increase or decrease?"
      answer: "Check confidence values in diff"
      effort: "Trivial"

    - question: "Were new gaps or conflicts discovered?"
      answer: "Check report/extraction-report.yaml"
      effort: "Easy"

    - question: "Did review status change?"
      answer: "Check review/status.yaml"
      effort: "Easy"

Pull request template:

# Blueprint Update for payment-system

## Summary
Blueprint regenerated after repository update to abc456

## Changes

### Kernel
- Added capability: `cap.send_notification`
  - Evidence: [e-039, e-040]
  - Confidence: 0.88

### Views
- Architecture view: Updated component responsibilities
- Product view: Added send_notification capability

### Evidence
- Added 3 new evidence objects
- Total evidence: 45 (was 42)

### Report
- No new gaps
- 1 new conflict resolved
- Average confidence: 0.93 (was 0.92)

### Review
- Status: usable (unchanged)
- 1 new ADR required

## Review Focus
- Verify send_notification capability is correct
- Review architectural impact of new capability
- Validate evidence quality

19.10 Chapter Summary

  • Blueprint Pack is the standard output unit of extraction, containing blueprint plus supporting materials
  • Pack connects blueprint, evidence, report, and review metadata in single output container
  • Pack structure includes blueprint.yaml, views/, evidence/, report/, review/, metadata/ directories
  • blueprint.yaml contains canonical kernel-centered blueprint with six slots
  • Evidence bundle includes evidence index and snippets for audit trail
  • Extraction report records gaps, conflicts, confidence listings, and diagnostics
  • Review directory contains decision log (ADR format) and status tracking
  • Pack supports human review with structured decision records
  • Diff-friendly representation enables Git-based version control
  • Diff-friendly requires stable field ordering, deterministic serialization, minimal noise
  • Git review compatibility enables easy inspection of architectural changes
  • Pack standard makes extraction operationally usable, auditable, and maintainable at scale
  • Pack enables large-scale blueprint mining and architectural intelligence
